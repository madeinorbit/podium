Podium loop inventory & split plan

2026-07-17 · POD-829 · measured on ludovico (8 cores, 24GB) at world size 588 sessions (106 live), 794 issues, 52 claude processes, ~65 abduco PTYs. Cost figures are measured where noted, otherwise derived from those measurements.

TL;DR. Both processes are single event loops that mix interactive work (WS frames, tRPC, PTY relay) with O(world) housekeeping on the same thread. The server's dominant recurring cost is the broadcast pipeline (wire-build + JSON diffing of all 588 sessions on nearly every state change, ~90–150ms per run, several runs/sec under load); until today it also had the 60s message sweep at 8s/run (fixed, POD-817). The daemon's dominant cost is PTY output relay + transcript parsing, which is already reasonably tiered. The right split is: (1) kill O(world) byte-diffing with revision counters, (2) move snapshot serialization to a Bun Worker, (3) move janitor timers to a separate process that talks to the server over the existing WS/tRPC surface, (4) keep per-session tailer parsing budgeted (it already yields). Primitives: Bun Worker (already proven in the daemon), Bun.spawn/systemd sibling processes over the existing protocol, WAL SQLite multi-process reads, revision integers instead of byte-compare.

Step 0 — remove O(world) work instead of moving it (biggest win, no new primitives)

  • Replace byte-compare dedup with revision counters. runSessionsBroadcast pays 588 × toMeta + 588 × stringify + a full-list stringify to decide nothing changed. A monotonically bumped rev per session (incremented at each mutation site) makes "anything to send?" an integer compare and turns the broadcast into "serialize only sessions with rev > lastSent". This alone removes most of the estimated 20–40% steady core.
  • getMeta(id)/has(id) for the §4 call sites (H1/H2 first).
  • Event-driven message delivery: the sweep exists because delivery is poll-retried; delivering on session.idle events (already partially wired via onSessionIdle) lets the sweep shrink to expiry-only.
  • Chunk any remaining >50ms synchronous job with await-yields — the daemon's tailer (POD-613) and transcript-indexer already model this.

Step 1 — a publish worker inside the server (Bun Worker, same DB, same process)

Primitive: Bun new Worker() (Web Worker API; the daemon's DiscoveryWorkerClient is the in-repo template, including job timeouts and restart-on-crash). Workers get their own event loop and can open their own bun:sqlite connection (WAL handles concurrent readers).

Move to the worker: wire-building of full snapshots, ledger.reconcile diffing, big JSON.stringify, the changes-table pruning. Main loop sends compact dirty-deltas (ids + revs) via postMessage; the worker posts back ready-to-send strings which the main loop only fans out (WS sends must stay where the sockets live). Rule of thumb: postMessage structured-clone costs about one serialize — so ship small deltas in, final strings out, never the whole world both ways.

Caveat from repo memory: bun --compile misses new Worker(URL) embedding — fine for the systemd server (runs from source), needs care if the desktop binary ever hosts this code path.

Step 2 — a janitor sibling process (systemd unit, existing protocol)

Primitive: a third systemd service (podium-janitor) — or Bun.spawn with its ipc channel if you want the server to own the lifecycle. It opens the same WAL SQLite read-side and talks to the server over the existing WS/tRPC surface for anything that must mutate live state (sendText, spawn).

Move there: messages sweep + expiry, steward polling, automations cron, event-log retention, issue auto-archive, repo-discovery orchestration. These are all "read DB, decide, issue a few commands" jobs — none needs to live on the interactive loop, and a janitor stall then costs nothing user-visible. It also decouples them from the redeploy restart cycle (janitor restarts don't drop client connections).

Watch out: single-writer discipline — keep all SQLite writes on the server (janitor mutates via RPC), or you re-fight the write-funnel/ledger design.

Step 3 — daemon: keep parsing budgeted, consider a tail worker-pool if needed

  • PTY relay must stay near the sockets (it's already tiered/coalesced well).
  • If transcript parse ever dominates again: move the JSONL parse of tier-2/3 (unwatched) sessions to the existing worker (it already owns transcript summarize code) and keep only focused sessions' parse on the main loop.
  • Replace the 65 × 700ms stat-pollers with one shared stat batch per tick (one timer, one pass) — cosmetic now, but removes 65 timers' scheduling jitter.

Step 4 — idle-session cap (human decision 2026-07-17 [spec:SP-c29e]: replaces broader host tuning)

  • Auto-hibernation gains a tunable max-idle-sessions cap per machine: beyond the cap, oldest-idle sessions hibernate even without memory pressure — motivation is the idle fleet's standing CPU (~1.6%/proc ⇒ ~80% of a core at 49 idle) and RSS (15–19GB). The memory-pressure trigger stays as-is. Candidate filter reuses maybeAutoHibernate's safety gates (idle window, PTY-quiet, never needs_user/working, resume ref).
