Mechanism 0 — the concierge (the front door)

The + button in the sidebar opens a persistent per-repo intake thread. It arrives already knowing the board (a zero-LLM digest: ready/blocked/needs-human, live sessions with provenance, recent events) and its first move on any wish is prior art: it searches issues, past conversations and transcripts (omni-search), and existing branches — then asks "continue #34's half-done branch, or fresh?" before filing anything. It structures work as dependency-chained epics whose descriptions double as agent work briefs, answers status questions with cited sources, and cannot start a session without your explicit in-thread word — a prompt rule backed by a code gate (confirmed:true on every spawn-capable tool, including the create --start shortcut) that holds on both backends via server-minted thread identity, and fails closed for identity-less callers.

Taxonomy that emerged in review: the concierge (always-on, gated) · the manual pump — a session you deliberately appoint to orchestrate an epic; your instruction is its confirmation, and the CLI now carries everything it needs (per-issue agent/model/effort, child-start within its subtree, events, cleanup, integrate) · the pump (autonomous — deferred, unbuilt). Only the last acts without you.

Mechanism 1 — the event log (the senses)

Podium already knew everything that happens — but only as transient websocket broadcasts: if nothing was listening at that instant, the fact evaporated. The event log makes state changes durable and replayable: an append-only SQLite table (podium_events) where each row has an auto-incrementing id that doubles as a cursor. Any consumer — the steward, a CLI, a future UI feed — remembers the last id it processed and asks "everything since 741, please". Ids never rewind or get reused, even after pruning, so a cursor is always safe to resume from.

Event kindEmitted whenPayload
issue.createdan issue is filed (any path: UI, CLI, agent, concierge){seq, title}
issue.stage_changedstage moves, except into done{seq, from, to}
issue.closedthe issue flips to closed — by any means: close, board drag-to-Done, update --stage done, supersede, duplicate{seq, reason}
issue.readyimmediately after a close, for each dependent whose last blocker that close removed{seq, unblockedBy}
issue.needs_human / _clearedthe needs-human flag actually flips (deduped){seq, question}
issue.startedworktree+branch created, agent spawned{seq, branch, worktreePath}
session.phasea session's agent-state phase genuinely transitions (working→idle, →needs_user, →errored…). Restart re-seeds are filtered.{phase, verdict?, agentKind, cwd}
The design detail that matters: closed-emission is owned by the one low-level update() that every close path funnels through, keyed on the derived closed-state flipping false→true. That's why a board drag emits the same events as an agent's close --note — and why a double-close emits nothing twice. (The first review round caught that the original version missed the board-drag path entirely — the log would have silently ignored most human closes.)

Emission can never break work: every emit is wrapped so a full disk or corrupt row loses a log entry, never a close. Retention: a 6-hourly job deletes rows older than 14 days and caps the table at 50k rows (measured: ~42ms per pass). Reading it: podium issue events --since <id> (agents get it too, via CLI and MCP).

Mechanism 2 — the steward (the reflexes)

The steward is not an agent, not a session, and not a chat. It is ~300 lines of plain TypeScript inside podium-server that convert events into housekeeping actions. No LLM is involved anywhere in it today — every handler is deterministic code, so a steward pass costs zero tokens. It exists because today you are the message bus: when an agent finishes a blocker, nothing tells the waiting work until you notice.

What one tick does (every 15s while enabled)

  1. Gate: read steward.enabled from settings. Off → return. (This is the entire cost of the dark ship: one settings read.)
  2. Resolve the cursor from steward_state. First time ever (or corrupted): seed it to the newest event id and persist — so enabling the steward never replays history; it starts from "now".
  3. Read the batch: all events after the cursor (capped at 200/tick for backpressure).
  4. Map events → triggers via a pure rule table: issue.closed / issue.ready → unblock:<repo> · issue.needs_human → needshuman:<issue> · everything else consumed silently. Events with the same key coalesce into one batch — an epic close unblocking five issues is one handler run seeing all five, not five runs.
  5. Run handlers, one key at a time (this sequencing is the per-repo serialization). Each handler is wrapped: a throw is logged and dropped — the steward can be wrong, but it can never wedge.
  6. Advance the cursor — only now, after handlers ran. A crash mid-tick means the batch re-runs next tick; handlers are idempotent (see dedup below), so replay is safe.

The two handlers that exist: unblock + parent-nudge

Parent-nudge: when a child issue closes, its parent's live sessions get one counted line — "Child #N closed — 3 of 7 children remain" — plus a deduped comment carrying the child's completion note. This is what lets an orchestrator session (the manual pump) sleep instead of polling. Unblock: for each issue.ready in the batch, two things happen to the newly-ready issue:

  • A durable comment — Unblocked by #58: <the closed issue's completion note> — authored by steward. This is the inter-agent mail: the completion note the finishing agent wrote travels to the work that was waiting on it. Dedup: if that exact marker already exists in the comments, skip (so crash-replays never double-post).
  • A nudge to live sessions in that issue's worktree — a fixed, single-line message ("Blocker #58 closed — you are unblocked… run: podium issue prime") queued through the durable outbox. Strictly filtered: only live/starting, never shells (no text into your bash), never parked sessions (a nudge must never resurrect a 500MB agent process). No agent-authored text is ever interpolated into the nudge — the note stays in the comment.
Failure philosophy — drop, don't wedge: the steward prefers missing one action (which is always visible as a gap in the audit trail) over getting stuck or acting twice. Everything it does is an issue comment or an event row — there is no invisible steward action.
What flipping steward.enabled does NOT do: no sessions started, no LLM calls, no merges, no deletes, no answers to questions. Comments + nudges. That's the entire authority of the current steward.

Mechanism 3 — the pump deferred

The pump is the piece that would start work by itself: when an issue becomes ready and a slot is free and the epic allows it, spawn the agent session. It is a deterministic three-question check (mode? budget? capacity?) — a thermostat, not an AI — but it is the only mechanism in the whole design that creates new token-spending sessions autonomously. It is deferred on your call, and rightly: the cost story has to be airtight first.

What exists todayWhat the pump would add
Nothing starts sessions automatically. Every agent session is started by a human (UI/CLI), by a human-driven concierge command, or explicitly by another agent the human is talking to. The steward only comments and nudges already-running sessions. Auto-dispatch of ready issues under an epic's autonomy mode, bounded by budget + concurrency caps. Unbuilt: the autonomy field, the caps, the budget accounting, the dispatch loop.
Why deferred: "budget" needs a unit we can actually meter and hard-stop on. Sessions-started is trivially enforceable but crude; token attribution per epic doesn't exist yet; wall-clock lies. Until that design is settled (open question Q1 below), the pump stays unbuilt — not dark, unbuilt — so there is no flag anyone can flip into runaway spend.

Mechanism 4 — steering + provenance (the hands)

The superagent (and soon the concierge, which is the same brain with an intake job) can now fully drive sessions instead of just spawning and killing them:

ToolWhat it does / the safety detail
answer_questionanswers a pending AskUserQuestion menu by number/label. Gated: refuses unless the session is verifiably waiting on a menu (needs_user/question) — a stale menu can never be "answered" into a working agent's keyboard.
resume_and_sendwakes a parked session and delivers text via the durable outbox (survives restarts, waits out TUI boot).
wait_for_sessionblocks (≤120s) until the target session's phase changes — reads session.phase events from the log; returns instantly if the session is already settled. This is the first consumer of the event log besides the steward.
snooze / rename / set_work_state / hibernate / continuethe remaining metadata + lifecycle knobs, each a thin wrapper whose description exactly matches its server-side gate.

Provenance: every session now carries spawnedBy (user · issue:<id> · superagent:<thread>), stamped at creation. That's the audit answer to "where did this session come from?", the future basis for UI grouping/dimming of helper sessions, and the precondition for any steward policy like "only nudge sessions I dispatched".

One story, end to end

Issue #71 "API endpoint" blocks #72 "wire the UI". An agent works #71; another session sits in #72's worktree, stuck waiting. Steward enabled.

Agent (#71)
finishes, runs podium issue close 71 --note "endpoint at /api/export, auth via session cookie, see tests"
Event log
server appends [812] issue.closed {seq:71, reason:done} and — because #71 was #72's last open blocker — [813] issue.ready {seq:72, unblockedBy:71}. Both rows durable; nothing else has happened yet.
Steward, ≤15s later
tick reads 812–813 past its cursor → both map to unblock:<repo> → one handler run.
Handler
posts on #72: "Unblocked by #71: endpoint at /api/export, auth via session cookie, see tests" — then finds #72's worktree sessions, filters to the one live agent, queues: "Blocker #71 closed — you are unblocked. See the steward comment on your issue, or run: podium issue prime". Advances cursor to 813.
Agent (#72)
receives the nudge as its next input, runs prime/reads the comment — and has the completion note, i.e. exactly the context it needs to continue. No human was in the loop.
You, later
the whole exchange is on the issues: the note, the steward comment, the events. Audit by reading, not by reconstructing.
If #72 had no live session, the story stops at the comment — visible, durable, waiting. Starting a session for it is the pump's job, which is deferred; today that start stays yours (or the concierge's, on your explicit word).

What the adversarial reviews caught

Every slice went implement → adversarial review → fix → verify, all in subagents. Eight must-fixes never reached live:

Would have happenedCaught in
Silent close-events — board drag-to-Done, supersede, duplicate emitted nothing; orchestrator never dispatches unblocked work#58 round 1
Session resurrection — steward nudge would respawn exited agents (~500MB each, stale context, acting unprompted)#59 round 1
Shell command injection — backticked nudge text (incl. agent-authored note content) typed into live bash = command substitution#59 round 1
Historical replay — first flag-flip would process weeks of dark-period events at once#59 round 1
Dedup collision — closing #5 silently swallowed by a prior "#55" comment; guaranteed past 10 issues#59 round 1
Keystroke injection — answer_question on a stale menu types digits (or a submitting Enter) into a working agent, reports success#62 round 1
Phantom phase events — one spurious row per session on every redeploy (restart re-seed)#58/#59
Vacuous idempotence test — the dedup test passed with the dedup deleted#59 round 1

Enablement runbook (steward)

  1. Pre-flight: done. #63 (boot-seed) and #61 (retention) are live — nothing gates the flip anymore.
  2. Flip: Settings → steward.enabled: true (takes effect ≤15s, no restart).
  3. First tick seeds the cursor to "now" — dark history is skipped by design.
  4. Watch: steward comments on issues, podium issue events, [podium:steward] journal lines.
  5. Rollback: flip off. Cursor persists; re-enable resumes (or re-seeds if pruned past).
