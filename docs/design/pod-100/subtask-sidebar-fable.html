Podium · POD-100 · proposal B · Claude Fable

The sidebar is a status board, not a tree

Proposal A separated the work tree from the agent roster — the right diagnosis. This proposal goes one step further: the sidebar should stop rendering execution as rows at all. An issue row absorbs its agents the way a Linear row absorbs an assignee; depth beyond two levels rolls up into numbers; and anything that needs you surfaces to the top instead of waiting to be found. The full trees — subtasks and agent lineage — live one click away in the peek drawer that already exists.

01Diagnosis

Three relationships, one visual grammar

The screenshot isn't confusing because the styling is off. It's confusing because the sidebar draws three different edges — issue contains issue, issue is executed by session, session spawned subagent — with the same rails, the same indentation, the same chevrons. No styling fixes a category error.

02The reframe

Decide what the sidebar is for, then the layout follows

Podium already has a planning surface (the Tasks board) and now a detail surface (the peek drawer, POD-95). The sidebar is neither. It answers three questions, in priority order: what needs me, what is moving, where do I click to get there. Every pixel that serves those questions earns its place; structural completeness does not.

⚑

Attention is the sort key

Needs-human, unread results, and blocked work must be visible without expanding anything — even when they live four levels deep. Structure never buries a request for you.

▸

Presence is a property

"An agent is working here" is a fact about an issue, like its color or its state — not a child of it. Facts about a row render inside the row.

⌕

Detail is one click away

The complete subtask tree and the complete agent lineage are real and useful — in the peek drawer, at full width, with room to breathe. The sidebar links; it doesn't mirror.

03Interactive proposal

One grammar against five kinds of complexity

Switch scenarios. Chevrons expand subtasks (and nothing else). Click an avatar cluster to jump to a specific agent. Deep trouble surfaces on its own.

Sidebar prototype

    Everything here is honest to the data model: issues have parentId + direct-child counts; sessions bind to issues; native subagents are a count on their session. Nothing new is invented — only re-homed. click rows, chevrons, avatars, folds.

    The five scenarios are the five ways the current sidebar degrades: idle scanning, wide branches, deep branches, many agents on one issue, and long completed tails.

    04Depth

    Two visible levels; exceptions teleport up

    Proposal A solved depth by moving the camera — branch focus with a breadcrumb. That works, but it makes the user do the walking, and a monitoring surface should not require navigation to deliver bad news. This proposal inverts it:

    Roll up, don't drill down. The sidebar renders a branch at most two levels deep. Everything deeper is summarized into its deepest visible ancestor (3 deeper · 1/6 done). But any deep descendant that needs a human, holds an unread result, or is blocked escapes the roll-up and surfaces as a flagged exception row directly under the branch root — with a path whisper showing where it came from. Resolve it, and it sinks back into the numbers.

    DepthRenderingAttention behavior
    RootFull row: identity, roll-up, presence, timeraggregates beacons from the whole branch
    ChildCompact row on one railown state, own presence
    DeeperRolled into N deeper on its ancestorexception rows teleport to the branch root
    Full treePeek drawer (click the roll-up line)complete, scrollable, no viewport budget

    The tree stays arbitrarily deep in the data model. The sidebar just refuses to spend title width on rails — a third indent level costs ~15% of an already narrow column and buys nothing the roll-up line doesn't say better.

    05Execution detail

    Lineage gets a real home, not a smaller font

    06Grammar

    Seven laws

    01

    One row per unit of work

    Issues get rows. Sessions bound to an issue never do — their presence renders inside the issue row. (Unbound shells and worktree sessions keep their current rows; they are the unit there.)

    02

    The chevron keeps one promise

    Expanding an issue reveals subtasks. Always. There is no second kind of child.

    03

    Presence is a property

    Avatar cluster + liveness dot + timer on the row. Multiple agents stack; overflow becomes +2; native subagents are a ×N badge on their session's avatar.

    04

    Two levels, then numbers

    Depth beyond the second level rolls into N deeper · k/m done on its ancestor. The full tree lives in the peek, one click away.

    05

    Attention outranks hierarchy

    Needs-human, unread, and blocked descendants surface as exception rows at the branch root, with a path whisper. Depth may hide work; it may never hide a request for you.

    06

    Done decays

    Fresh completion stays visible with its badge; settled completion folds into Done · N. Parent progress counts direct children only, so the arithmetic stays explainable.

    07

    Spin-offs stay siblings

    Discovered work is a top-level row with a ⤷ origin whisper (POD-85, kept verbatim). Provenance is a hyperlink, not a rail.

    The honest trade-off: proposal B hides quiet deep structure harder — you cannot browse a depth-4 subtree in the sidebar at all. That is deliberate. Browsing structure is a planning activity, and both planning surfaces (board, peek) do it better. What the sidebar loses in completeness it gains in a promise it can actually keep: if nothing is flagged, nothing needs you.
