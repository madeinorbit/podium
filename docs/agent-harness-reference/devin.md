# Devin — orchestrator integration reference

Authoritative integration reference for hosting Devin inside a remote/web/mobile multi-agent orchestrator. Binary name: `devin` (the "Devin for Terminal" CLI — a Rust TUI in the Claude-Code mold: modes, slash commands, hooks, MCP, resume, `--print`, ACP server). Install: `curl -fsSL https://cli.devin.ai/install.sh | bash` (default path `~/.local/bin/devin`) or `brew install --cask devin-cli`.

> **Two distinct surfaces — don't conflate.**
> 1. **Devin CLI ("Devin for Terminal")** — the *local* PTY-hosted coding agent. **This is the integratable harness** and the primary focus of this document.
> 2. **Devin Cloud (External API)** — the original cloud autonomous agent over REST at `https://api.devin.ai/v1/...` (v3 for org/enterprise), no PTY. Relevant only because the CLI can **hand off** a local session to a cloud Devin via `/handoff` (one-way push to a *different* id namespace, e.g. `devin-<hex>`). Covered where it matters.
>
> **Verification status.** CLI flags, slash commands, keyboard shortcuts, permission modes, hook event names, and config-file keys below are sourced from the official docs (`docs.devin.ai/cli/...`, `cli.devin.ai`, `cognition.ai`) and are treated as verifiable. **On-disk shapes** — exact transcript glob/dir, session-file schema, ATIF export schema, token-file path, hook-payload `session_id`/`cwd` presence — are **doc/observation-derived and UNVERIFIED against a live install**; each is flagged inline. The documented surface is unusually complete (`cli/reference/commands`, `keyboard-shortcuts`, `permissions`, `hooks/{overview,lifecycle-hooks}`, `models`, `configuration/config-file`, `handoff`, `acp/zed`, `troubleshooting`). Gate behavior on `devin --version` where on-disk/feature assumptions matter.
>
> **Headless first-class.** Unlike a pure-TUI agent, Devin exposes `devin --print` (one-shot) and `devin acp` (JSON-RPC over stdio) as non-PTY surfaces — prefer these for clean programmatic control and the orchestrator's own one-shot LLM work.

## 1. Discovery & identity

- **Binary detection / "installed?":** Single binary `devin` on PATH; default install `~/.local/bin/devin` (macOS/Linux/WSL). **"Installed?"** = `command -v devin` resolves, or stat `~/.local/bin/devin`. No documented alias/rename machinery. **Do not** detect the unofficial third-party PyPI packages `devin-cli` / `devin-mcp-cli` (API wrappers, not this harness), and distinguish from the *cloud* product (no binary). "Signed in?" is a separate gate — `devin auth status` (§4). *(established method)*
- **Version detection / gating:** `devin --version` (also `devin version`) prints the version and exits. No documented per-feature min-version contract; modes/flags evolve via stable/next changelogs (`docs.devin.ai/cli/changelog/{stable,next}`). Treat resume/hooks/`--print`/`acp` as present on any recent build; **exact min-version per feature UNVERIFIED**. Gate on `--version` before relying on newer flags. *(established method = `--version`; gating = best-effort)*
- **Existing-session discovery (paths/globs):** **Authoritative enumerator = `devin list` (alias `devin ls`)** — interactive picker by default; **`devin list --format json`** (also `--format csv`) for machine reading. *Prefer shelling out to `devin list --format json` over globbing files.* **On-disk (UNVERIFIED):** per-session transcripts observed as `{session_id}.jsonl` with a sibling subagent-transcript directory, under the config dir `~/.config/devin/` or data dir `~/.local/share/devin/`; run-logs are confirmed at `~/.local/share/devin/cli/logs/`. Env override for the config location: `--config <PATH>` (per-spawn) / config-dir redirection. *(established method = `devin list --format json`; file glob = fallback, exact glob unverified)*
  - **Conflicts / open questions:** Transcript layout is reported two ways — per-session **`{session_id}.jsonl`** under `~/.config/devin/` or `~/.local/share/devin/`, vs. a `transcripts/` subtree of single-object JSON files under `~/.local/share/devin/cli/transcripts/`. The verifiable, version-stable read is **`devin list --format json`** plus the documented **ATIF `--export`** (§10); treat the raw on-disk glob as unverified and probe a live install before depending on it.
- **Stable session identity:** A **human-readable slug** (animal-name style, doc example `brisk-otter`), **not a UUID**. Obtained at launch from `devin list --format json` or the per-directory "most recent session" state; for a pre-existing session, read the slug from `devin list --format json` or pass it to `-r`. (The *cloud* API uses a separate `devin-<id>` namespace over REST — never conflate after `/handoff`.) *(established method)*
  - **Conflicts / open questions:** One source reports a `session_id`/`sessionId` field inside the transcript JSON as the identity; the other reports the slug as the canonical id. Both can be true (the slug *is* the session id); resume accepts the slug. Use the value returned by `devin list --format json` as the source of truth.
- **Session ↔ repo/worktree/cwd mapping:** Sessions are **directory-scoped** — `devin -c`/`--continue` resumes "the most recent session **in the current directory**." The launch cwd is the binding key: spawn with the worktree as cwd, then back-map via per-dir most-recent + the slug. Transcript also carries the working directory (a `working_directory`-style field, **UNVERIFIED** exact key) for reverse mapping. *(established method = cwd-binding)*

## 2. Launch & process model

- **Spawn mechanism / PTY host:** Interactive Rust **TUI** hosted in a **PTY** (local / SSH-remote / daemon-subprocess). Either direct-exec `devin` in the worktree cwd, or (orchestrator-typical) shell-PTY-with-typed-command. Two **non-TUI** modes also exist: `devin --print` (one-shot, no REPL) and `devin acp` (JSON-RPC server over stdio). For a terminal-mirroring UI use the PTY+TUI path; for clean headless control use `--print`/`acp` (§6/§12/§15). *(established method)*
- **Full launch command / args / env:** Base `devin` (interactive) or `devin -- <prompt>` (REPL seeded with the prompt). Documented global flags: `--model <MODEL>`, `--permission-mode <normal|dangerous|bypass>`, `-c/--continue`, `-r/--resume <SESSION_ID>`, `-p/--print [PROMPT]`, `--prompt-file <FILE>`, `--config <PATH>`, `--export [PATH]` (ATIF after each turn), `--respect-workspace-trust`, `--sandbox` (pairs with Autonomous mode), `--version`, `--help`. **`--config <PATH>`** redirects the config/profile dir (the multi-account / isolation knob). Proxy via config `proxy` key. **Env:** the External API uses `DEVIN_API_KEY` (Bearer); **whether the *local CLI* reads `DEVIN_API_KEY` is UNVERIFIED** (CLI auth is the OAuth token store, §4). No documented "strip inherited keys" requirement for the CLI, but the orchestrator should defensively scrub stray provider keys. **Whether any instrumentation env (hook callback port/token) is injected is UNVERIFIED — Devin hooks are stdin/stdout, not HTTP** (§8), so no injected port is expected. Generic pane-identity env (pane/tab/worktree ids) is injected by the orchestrator's spawn path. *(established method = flags; env injection details unverified)*
- **Initial-prompt injection mode:** Multiple, all documented — **(a) argv** `devin -- "<prompt>"` (the `--` separator is **required** if the prompt starts with a flag) → REPL seeded; **(b) `--prompt-file <FILE>`** load prompt from file (best for large/multiline; avoids quoting + bracketed-paste hazards); **(c) `-p/--print "<prompt>"`** one-shot non-interactive; **(d) type-after-ready** into the live composer (generic PTY path: bracketed paste + a separate CR to submit). **Preferred for orchestrator seeding: `--prompt-file` or argv.** Note: `devin -- <prompt>` **auto-submits immediately** — if you want review-before-send, do NOT seed via argv; launch bare `devin` and use the type-after-ready / paste-into-composer path instead. *(established method)*
  - **Conflicts / open questions:** One source treats injection as `stdin-after-start` (launch bare `devin`, wait for the agent foreground, then write `<prompt>\r` as plain typed input + a single CR — *not* bracketed paste, *not* argv). The other documents argv / `--prompt-file` / `--print` as first-class. Both work; choose by intent: **argv/`--prompt-file`** for fire-and-forget auto-submit, **type-after-ready (plain input + `\r`, or bracketed paste for multiline)** for review-before-send.
- **First-run / trust handling:** First run requires `devin auth login` (OAuth / "Log in with Devin for Enterprise"). `--respect-workspace-trust` opts into workspace-trust gating. To avoid a trust/onboarding menu eating the first prompt: ensure auth is done first (`devin auth status` green; for SSH/headless use `devin auth login --force-manual-token-flow`) and pre-seed config (`~/.config/devin/config.json` or `--config`). **Exact first-run trust-menu byte sequence UNVERIFIED.** No persisted trust-hash allowlist step is documented (cf. some agents). For fully unattended runs, rely on `--permission-mode bypass`. *(established method; exact menu sequence unverified)*
- **Permission / YOLO / sandbox modes:** `--permission-mode <normal|dangerous|bypass>` — **`bypass` = YOLO** ("all tool calls auto-approved"). Richer runtime set via **modes** (§6): **Normal**, **Accept-Edits**, **Plan**, **Bypass**, **Autonomous** (`--sandbox`; OS-level sandbox so exec/fetch auto-approve but edits still prompt). Config: `sandbox` key + `permissions{allow,deny,ask}` arrays. *(established method)*
- **Model / reasoning-effort / fast-mode at launch:** **Model:** `--model <MODEL>` — short names `opus | sonnet | swe | codex | gemini` resolve to "latest in family"; also `swe-1-6-fast`, `gpt`, plus open-source `kimi`, `glm`, and the `adaptive` router. Default from config `agent.model` (e.g. `"swe-1-6-fast"`); recommended default = **Adaptive** router. **Fast mode IS a concept = SWE-1.6 Fast** (`--model swe-1-6-fast` at launch, or `/fast` at runtime). **Reasoning effort = "thinking level"** — toggled at runtime with `Alt+T` / `Opt+T`; **no documented launch flag** for it. *(established method = `--model`; fast-mode = SWE-1.6 Fast; reasoning-effort flag = not supported / runtime only)*
  - **Conflicts / open questions:** One source records **no agent-specific model/reasoning/fast-mode handling at all** (model only via free-form args). The other documents the full `--model`/`/fast`/`Alt+T` surface. Prefer the documented surface (verifiable flags); reasoning-effort as a *launch flag* remains unverified.

## 3. Resume & reattach

- **Resume support / flags / id source:** Yes, first-class. **`-c/--continue`** (most-recent session in cwd), **`-r/--resume <SESSION_ID>`** (specific slug, or interactive picker if no id given). In-session equivalents: **`/continue [session-id]`**, **`/fork [step]`** (branch a new session from a step), **`/revert <step>`** (rewind), **`/steps`** (list steps). Id source = `devin list --format json` or the per-dir most-recent state. *(established method)*
  - **Conflicts / open questions:** One source models resume as a `--resume <id>` flag with a key of `session_id` (i.e. `devin --resume <session_id>`). The documented flag is **`-r/--resume <SESSION_ID>`** taking the slug. Same mechanism; pass the slug from `devin list --format json`.
- **State restored vs lost on resume:** Conversation history is restored; config `agent.show_history_on_continue` controls whether prior history is **re-displayed**. **Model:** comes from config/launch flag at resume — **re-specify `--model` to be safe** (whether the prior session's model is persisted+auto-reapplied is **UNVERIFIED**). **Permissions:** re-evaluated from current config + `--permission-mode`. **cwd:** binding means resume from the same directory. **Lost:** the live PTY *screen* (the TUI re-renders from history). *(established method; model-persistence unverified)*
- **Reattach to a still-running PTY after restart:** Devin provides **no native PTY reattach** — this is the orchestrator's job: wrap the PTY in a multiplexer (tmux/abduco) or daemon-held PTY, reattach, and issue a **redraw nudge**. **`Ctrl+L`** is the explicit repaint/refresh key — use it on SIGWINCH/reattach to force a clean repaint. If the live PTY is gone but the session persisted, **cold-resume** with `devin -r <slug>` (not a reattach). *(established method = generic mux + `Ctrl+L`; cold-resume fallback)*
- **Idempotency / race hazards:** (a) **`--continue` resolves "most-recent-in-dir"** → two concurrent spawns in the same cwd can target/clobber the same session; once the slug is known, **pin `-r <slug>`**. (b) **ATIF `--export` writes after every turn** → guard against reading partial files mid-turn. (c) For the *cloud* API create-session path, `idempotent: true` dedups (REST only). (d) Only one PTY client should hold the multiplexer master (double-attach → echo-doubling). *(guard rules)*

## 4. Auth & subscription

- **Credential locations:** OAuth via `devin auth login` (subcommands `login` / `logout` / `status`). Tokens stored in the CLI data/config dir — **exact token file path UNVERIFIED** (config dir `~/.config/devin/`, data dir `~/.local/share/devin/`; trace-level run-logs can leak `Authorization` headers, confirming a bearer token is held). For the **External API**, the key is from the settings page, sent as `Authorization: Bearer $DEVIN_API_KEY` (service-user keys prefixed `cog_`). `DEVIN_API_KEY` is **confirmed for the API**; CLI use is **UNVERIFIED**. No OS-keychain use is documented. *(established method = OAuth token store; exact path unverified)*
- **Reuse model:** **Spawn-and-let-CLI-auth** is the natural path — ensure `devin auth status` is green, else drive `devin auth login` (or `--force-manual-token-flow` headlessly). No documented orchestrator-managed credential capture/materialize. **Multi-account isolation = profiles**: each profile is a separate config dir (`--config <PATH>`), "fully isolated including session caches and active session IDs" — select per spawn. *(established method = spawn-and-let-CLI-auth; multi-account = per-config-dir profiles)*
  - **Conflicts / open questions:** One source found **no agent-specific auth handling** (no managed-account manager entry, no token capture) — consistent with spawn-and-let-CLI-auth. The profiles-as-config-dirs mechanism is documented and is the recommended multi-account path.
- **Token refresh:** Handled internally by the CLI's OAuth client against Cognition's auth/identity provider (Enterprise IdP for "Log in with Devin for Enterprise"). The orchestrator does **not** refresh. Endpoint / client-id / write-back path **UNVERIFIED** (not documented). *(owned by the CLI)*
- **Env hygiene:** No documented requirement to strip inherited keys for the CLI. For headless API calls, `DEVIN_API_KEY` is the only credential needed. Strip unrelated provider keys defensively. *(no agent-specific stripping)*
- **Multi-account isolation / selection:** Via **profiles** = distinct config dirs (`--config <PATH>`), each with its own auth + session cache; selected per spawn. *(established method)*

## 5. Models, usage & accounting

- **Model listing (settings UI):** **No documented `devin models` / `--list-models` subcommand** emitting a machine-readable list. Authoritative *live* list = the in-session **`/model`** selector (a TUI picker). For a settings UI, enumerate the documented families/short-names: `opus`, `sonnet`, `swe` (`swe-1-6-fast`), `gpt`, `codex`, `gemini`, plus `kimi`, `glm`, and the `adaptive` router. Short names resolve to "latest in family" → can't pin exact dated versions from the CLI; **freshness via the changelog, not an API.** *(not supported as a machine list; fallback = hardcoded short-name enum + `/model` picker)*
- **Runtime switching mid-session:** Yes — **`/model [name]`** (no-arg opens selector), **`/fast`** (→ SWE-1.6 Fast), **`Alt+T` / `Opt+T`** cycles thinking/reasoning level for the current model. Mode (gates *tools*, not model) via **`Shift+Tab`** / **`/mode`**. *(established method, in-TUI affordances)*
- **Usage windows / quota / rate-limits:** Billed in **ACUs (Agent Compute Units)**; consumption visible in the **Enterprise Analytics dashboard** (web), and possibly an enterprise v3 analytics API (fields **UNVERIFIED**). **No documented local-CLI command or file exposing remaining quota / reset window.** *(not supported locally; web dashboard only)*
- **Token accounting for analytics:** **No documented live per-turn CLI token field.** `/context` shows context-window usage (a live gauge, not a billing export). The **ATIF export** (`--export [PATH]`, written after each turn) is the structured per-turn record and the most promising place for token/usage fields. Hook payloads (§8) do **not** include token counts. *(best-effort via ATIF; otherwise not available)*
  - **Conflicts / open questions:** One source reports token totals **are** recoverable post-hoc from the on-disk transcript metadata — per-step `total_input_tokens`/`input_tokens` + `output_tokens` + `cache_read_tokens`/`cache_read_input_tokens` + `cache_creation_tokens`/`cache_creation_input_tokens`, read from `step.metadata` or `step.metadata.metrics`, summed across steps. The other reports the **exact ATIF/transcript token schema as UNVERIFIED**. Net: token accounting is **best-effort post-hoc** — parse `step.metadata{,.metrics}` token fields from the transcript / ATIF export; **validate field names against a live file before trusting them.** No live per-turn token feed.
- **Pricing / cost:** Cost expressed in **ACUs**, surfaced only in the Enterprise Analytics dashboard — **no local pricing data file, no per-model $ rates from the CLI.** The orchestrator must maintain its own price table if needed. *(not supported — orchestrator-owned)*

## 6. Driving & input — two-way control

- **Send a new user message mid-session:** **TUI** = type into the PTY composer and submit with **`Enter`** (`0x0d`). Multi-line: **`Shift+Enter`** or **`Alt+Enter`** inserts a newline; submit is a *separate* `Enter`. For large/multiline content prefer paste (`Ctrl+V` / `Shift+Insert`) or `Ctrl+G` (external editor). Safe orchestrator transport = bracketed paste + a separate CR (single-line can be plain text + `\r`). **Non-PTY** = `devin acp` JSON-RPC `prompt` method (fully programmatic). *(established method)*
- **Read the agent's current input-box contents (draft sync out):** **Not supported natively** — no documented API/OSC/transcript channel for the composer buffer. Only route = **screen-scrape** the composer region from the xterm buffer (best-effort; **dim-placeholder styling UNVERIFIED**). For `acp` clients the draft lives in the orchestrator, so this is moot. *(not supported; screen-scrape only)*
- **Pre-fill / set the input box (draft sync in):** **No documented flag/env to pre-fill the *live* composer.** Closest = `--prompt-file` / argv at launch (seeds the *first* message), or typed paste into the composer. No persistent draft-prefill mechanism. *(launch-only seed; no live prefill)*
- **Answer interactive prompts / permission menus:** Two routes. **(1) Programmatic via hook (cleanest):** the **`PermissionRequest`** hook fires (stdin `tool_name`, `tool_input`) and a command hook can return **`approve` / `block` / `deny`** on stdout — auto-answer without scraping (§8). **(2) TUI menus:** permission requests appear as menus; selection via arrow keys + `Enter`; the **modes** system governs whether prompts appear at all. Reading the *option set* from the TUI = screen-scrape. The `ask_user_question` tool (Plan/Ask profiles) → waiting state, with its option set delivered through the transcript/TUI. **Exact menu byte sequences UNVERIFIED.** *(established method = `PermissionRequest` hook decision; TUI scrape = fallback)*
- **Interrupt / cancel / escape:** **`Ctrl+C`** (`0x03`) cancels current input or the running agent; **`Esc`** (`0x1b`) cancels the running agent; **`Ctrl+D`** (`0x04`) exits when input is empty. No documented double-tap requirement. *(established method)*
- **Slash commands:** Rich set — `/mode`, `/normal`, `/accept-edits`, `/plan`, `/bypass`, `/ask <q>`, `/model [name]`, `/fast`, `/clear` | `/new`, `/continue [id]`, `/fork [step]`, `/steps`, `/revert <step>`, `/context`, `/handoff [task]` (→ cloud), `/help`, `/exit` | `/quit`. Trigger by typing `/` at the composer. *(established method)*
- **Attachments / large paste:** **`@`** opens file/directory autocomplete to add files to context; **`Ctrl+V` / `Shift+Insert`** paste; **`Ctrl+G`** external editor for big messages. Image attachment handling depends on the terminal — **UNVERIFIED** (no doc detail). *(established method for files/paste; images unverified)*
- **Important keyboard shortcuts & modifiers (mobile soft-keyboard surfacing):** submit `Enter`; newline `Shift+Enter` / `Alt+Enter`; interrupt `Ctrl+C`; cancel-agent `Esc`; exit `Ctrl+D`; history fuzzy-search `Ctrl+R`; redraw `Ctrl+L`; paste `Ctrl+V` / `Shift+Insert`; external editor `Ctrl+G`; mode-cycle `Shift+Tab`; thinking-level `Alt+T` (`Opt+T`); bash mode `!`; file autocomplete `@`; thinking-trace viewer `Ctrl+O`. **Alt/Option-fidelity REQUIRED** for `Alt+Enter` and `Alt+T` — docs explicitly say macOS should configure **Option-as-Meta**. The orchestrator's terminal must pass Alt/Meta bytes correctly. *(established method)*
  - **Conflicts / open questions:** One source enumerated **only** submit = `\r` as the Devin-specific key (everything else generic). The other provides the full documented shortcut set above. Prefer the documented set — it is from `cli/reference/keyboard-shortcuts`.

## 7. Agent-state classification

- **State vocabulary:** Map into the orchestrator's shared vocab: **`working | waiting | done`** (+ an `interrupted` flag). The *cloud* API status enum (`working | blocked | finished | suspend_requested | resume_requested | resumed`) is a useful target for the cloud path but is **not** the local-CLI vocab. `blocked` ≈ `waiting` for the local mapping. *(established method)*
- **Signal sources & authority rank:**
  1. **Hooks (authoritative)** — `SessionStart`, `UserPromptSubmit`, `PreToolUse`/`PostToolUse`, **`PermissionRequest`**, `Stop`, `SessionEnd`, `PostCompaction` (§8). Claude-Code-compatible.
  2. **Process foreground/child presence + output activity** (busy heuristic; generic).
  3. **Transcript events** (tail; schema **UNVERIFIED**).
  4. **Process exit** (done/crash; §14).
  5. **OSC title** — **UNVERIFIED** whether Devin emits OSC title sequences; treat as absent until confirmed.
- **Event→state mapping table:**
  | Hook event | State |
  |---|---|
  | `SessionStart` | **null / ready** — do NOT map to `working` (mapping it shows a phantom "Devin - Running" spinner before the user types); clears the per-turn cache |
  | `UserPromptSubmit` | `working` (also the new-turn boundary) |
  | `PreToolUse` | `working` |
  | `PostToolUse` | `working` (continuing) |
  | `PostCompaction` | `working` |
  | `PermissionRequest` | **`waiting`** (the key "stopped but needs user" signal) |
  | `Stop` | `done` (+ `interrupted:true` if `is_interrupt === true`) |
  | `SessionEnd` | `done` / exited |
  | anything else | null (ignored) |
- **"Stopped but needs the user" detection:** Cleanly distinguished by **`PermissionRequest` → `waiting`** (vs `Stop`/`SessionEnd` → `done`). Also the `ask_user_question` tool (Plan/Ask) → waiting. Far better than scraping. The pending tool name/input is read from the `PermissionRequest` payload (`tool_name`/`tool_input`) for surfacing what's being requested. *(established method)*
- **Reconciliation:** Hooks win over output-activity heuristics; quiet-window/debounce on output activity; **stickiness** so a `PermissionRequest → waiting` state isn't flipped back to `working` by stray output until resolved. New-turn reset on `UserPromptSubmit`; full per-turn cache clear on `SessionStart`. No agent-specific reducer rules beyond these. *(generic reducer + new-turn resets)*
- **Sub-agent / nested-task state:** Subagents exist (`docs.devin.ai/cli/subagents`) with their **own transcript dir** (sibling to `{session_id}.jsonl`). Hook payloads carry `tool_name` but **no documented nested-session id field** → nested-state identity inheritance **UNVERIFIED**; treat subagent activity as parent-session `working`. *(not separately classified; unverified)*
- **Latency:** Hooks fire synchronously around tool boundaries (sub-second). Output-activity heuristic = immediate but noisy. Process-exit = immediate. Debounce = generic reducer. *(established)*

## 8. Hooks & instrumentation install

- **Supported?** **Yes — full hooks system, explicitly Claude-Code-compatible** ("uses a hook format compatible with Claude Code hooks; existing Claude Code hook configs work automatically"). This is a major integration win: an orchestrator that already speaks Claude-Code hooks reuses most of its pipeline. Mechanism = **command hooks** (a shell command; **JSON over stdin → JSON decision on stdout**) plus a `"type":"prompt"` variant. *(established method)*
- **Events emitted/installed:** `PreToolUse`, `PostToolUse`, **`PermissionRequest`**, `UserPromptSubmit`, `Stop`, `SessionStart`, `SessionEnd`, `PostCompaction` (8 events). *(established method)*
- **Install location & trust:** Config locations (in precedence): project **`.devin/hooks.v1.json`** (standalone — the whole file *is* the hooks object, no `"hooks"` wrapper); or a `"hooks"` key inside `.devin/config.json` / `.devin/config.local.json` / `~/.config/devin/config.json`; **and Claude-Code files** `.claude/settings.json` / `~/.claude.json` are read when `read_config_from.claude` is enabled. Config is **JSONC** — parse with a JSONC parser, not strict `JSON.parse`. **Matchers are regexes** and "omitted = all" — so **omit the matcher entirely** rather than using Claude's `"*"` (a literal `"*"` is a bad regex here). **No documented trust/hash-allowlist step** (Claude-Code compat *suggests* similar behavior; **UNVERIFIED**). *(established method)*
- **Payload fields (stdin JSON):** Common: `hook_event_name`. Per event — `PreToolUse`: `tool_name`, `tool_input`; `PostToolUse`: + `tool_response`{`success`, `output`, `error`}; `PermissionRequest`: `tool_name`, `tool_input`; `UserPromptSubmit`: `prompt`; `Stop`: `stop_hook_active` (+ observed `is_interrupt` for user-interrupt); `SessionStart`: `source`; `SessionEnd`: `reason`; `PostCompaction`: `summary`. Also extractable when present: `session_id`, `transcript_path`. Hook entry shape: `{"type":"command","command":"./script.sh","matcher":"<regex on tool_name>","timeout":<sec>}`. *(established method; payload extraction = Claude-style)*
  - **Conflicts / open questions:** The lifecycle doc's enumerated payloads did **not** explicitly list `session_id` / `cwd`; Claude-Code compatibility *implies* they're present (CC sends `session_id`, `cwd`, `transcript_path`) and one source's extractor reads `session_id` from Devin payloads — but **presence of `session_id`/`cwd` in Devin's payload is UNVERIFIED. Confirm against a live install before depending on `session_id` in the payload** (the session slug from `devin list` is the safe fallback id).
- **Transport back to the orchestrator:** **Decision protocol = stdout JSON** (`approve` / `block` / `deny` + reason); exit codes `0` ok, `2` block, other = non-blocking error (**fail-open** → agent continues). **Transport is stdin/stdout only — there is NO documented HTTP-callback-with-injected-port pattern.** Therefore the orchestrator must wire a hook **script** that itself forwards to the orchestrator (HTTP/file/socket of the script's own making). A practical pattern: a managed POSIX `.sh` (Windows `.cmd`) script that POSTs the payload to a local orchestrator endpoint with a token header, and re-sources a live endpoint file each invocation so a surviving PTY keeps reporting to a restarted orchestrator; missing endpoint/port/token → `exit 0` (fail-open), curl errors swallowed. *(established method = stdin/stdout decision; HTTP-forwarding = orchestrator-implemented inside the script)*
  - **Conflicts / open questions:** One source describes Devin's hook transport as a managed-shell-script that POSTs to `http://127.0.0.1:$PORT/hook/devin` (form-urlencoded, `X-...-Token` header) — i.e. the *orchestrator's* HTTP-callback convention implemented **inside** the script. The other confirms Devin itself **only** does stdin/stdout. These are consistent: Devin gives you stdin/stdout; any HTTP callback is something the orchestrator's own hook script does. **There is no Devin-injected callback port.**
- **Lifecycle:** Install = drop `.devin/hooks.v1.json` (or merge into config). Idempotent file write (strip prior managed entries before re-adding; match managed entries by the script filename). Versioned format (`hooks.v1.json`) handles format-skew. Update/uninstall = rewrite/remove the file/entries. Status = how many of the 8 events carry the managed command (`installed`/`partial`/`not_installed`/`error`). **Quirk:** when `read_config_from.claude` is enabled, imported Claude hooks fire **alongside** Devin's — surface a warning to avoid double-firing. Remote install (SSH): write script then config (script-before-config, POSIX-only) to avoid `command not found`. *(established method)*

- **Parsing / schema (raw, UNVERIFIED):** Role mapping in the single-JSON-object shape — `metadata.is_user_input === true` → user; else `step.role === 'assistant'` or presence of `step.tool_calls` → assistant. ## 12. Background / headless invocation

- **Non-interactive?** **Yes — `devin -p "<prompt>"` / `devin --print [PROMPT]`**: single-turn, no REPL, prints the response to stdout and exits. Prompt via argv after `-p`, or via `--prompt-file <FILE>`. Combine with **`--model swe-1-6-fast`** (cheap/fast) and **`--permission-mode bypass`** (unattended). **Structured-output format of `--print` UNVERIFIED** (likely plain text; use ATIF `--export` for structure). *(established method)*
  - **Conflicts / open questions:** One source states the orchestrator **deliberately does NOT** use the one-shot path (it keeps an interactive hosted REPL alive instead, since `devin -- <prompt>` auto-submits). The other documents `--print` as a first-class headless surface. Both are valid design choices: use `--print` for the *orchestrator's own* one-shot LLM work; keep the interactive REPL for user-facing hosted sessions.
- **Which features could use it:** Commit messages, PR titles/bodies, branch names, summaries, classification — all viable via `-p`. **Model discovery is NOT well-served** (no machine model list, §5). *(established)*
- **Auth path for headless:** Same OAuth/token store (`devin auth status` must be green; `--force-manual-token-flow` for SSH/headless first-auth), or `DEVIN_API_KEY` for the External-API route. Profiles (`--config`) for account selection. *(established)*
- **Cost/latency/timeout/caps:** Billed in ACUs; pick `swe-1-6-fast` for low latency/cost. **The orchestrator must impose its own timeout** (kill the process). Local execution; the cloud External API is the remote alternative for heavier one-shots. *(established)*

## 14. Failure, exit & recovery

- **Crash/exit detection:** Process exit code (`0` clean / non-zero error); `Ctrl+D` and `/exit`|`/quit` are clean exits. The **`SessionEnd`** hook fires with a **`reason`** field (clean vs error); a clean turn-complete is the **`Stop`** hook → `done`. **Distinguish clean-done (`Stop`/`SessionEnd`) from crash (PTY exit without `Stop`/`SessionEnd`).** `interrupted` is a user-interrupted `Stop` (`is_interrupt === true`). The cloud API status enum surfaces `blocked` (needs intervention) vs `finished`. *(established method)*
- **Reattach-after-restart healing:** Generic — exited-but-alive multiplexer rows heal via **cold-resume `devin -r <slug>`**; orphan/zombie reaping by **tracked PID** (not `pkill -f`, which can self-kill). `Ctrl+L` to repaint a recovered PTY. No Devin-specific healing protocol; a hook script that re-sources its live endpoint file keeps surviving PTYs reporting to a restarted orchestrator. *(generic + cold-resume)*
- **Error surfacing:** Signals to surface = `SessionEnd.reason`, non-zero exit, and (cloud) `blocked` status. The run-log `~/.local/share/devin/cli/logs/devin_<ts>_<pid>.log` is always written (handle secrets carefully). **`PostToolUse.tool_response` carries `{success, error}`** — capture `error` text for tool-failure surfacing (there is no separate `PostToolUseFailure`/`ErrorOccurred` event, so a failed tool stays `working` until `Stop` unless you read `tool_response.error`). `--export` ATIF may include error entries (**UNVERIFIED**). *(established method)*
  - **Conflicts / open questions:** One source reports tool-failure error text is **not** specially captured (no failure event in the installed set) — a failure stays `working` until `Stop`. The other documents `PostToolUse.tool_response.error`. Resolution: there is **no dedicated failure event**, but the **`PostToolUse` payload does carry `tool_response.error`** — read it to surface tool failures.

## 15. Remote / transport

- **Transports supported:** **(a) Local PTY** (primary). **(b) SSH / remote worktree** — supported; headless first-auth via **`devin auth login --force-manual-token-flow`** (manual token paste). **(c) Daemon-subprocess** — natural fit. **(d) `devin acp`** — JSON-RPC-over-stdio server: the cleanest *non-PTY* programmatic transport (drive prompts/tool-events/permissions over a structured channel instead of scraping a PTY); intended for editors but reusable by an orchestrator. **(e) Cloud handoff** — `/handoff` pushes repo + branch + conversation + uncommitted diffs to a cloud Devin VM, tracked in the web app / External API (`api.devin.ai/v1`), fully decoupled from any local PTY. *(established method)*
- **Forwarding for remote sessions:** PTY/git/fs over the orchestrator's relay (generic). **Hook callbacks are stdin/stdout to a local script** — for a remote session the **hook script must itself forward** to the orchestrator (HTTP/socket), since there is no injected callback port. A managed remote install writes the POSIX `.sh` script + JSONC config (script-before-config ordering) to the remote `~/.config/devin/config.json` + a shared managed-script dir over SFTP; **POSIX-only on remote** (Windows-remote deferred). The **ACP** path and the **cloud External API** are the two structured (non-PTY) remote options and avoid the relay's PTY-fidelity concerns entirely. *(established method)*
- **Limitations:** ACP rendering differs from the native TUI (surfaced through the client's ACP rendering). Cloud handoff is one-way push to a different id namespace. Enterprise gating: CLI use may require a "Use Devin CLI" RBAC role. *(established)*
