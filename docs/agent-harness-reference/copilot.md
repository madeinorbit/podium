# GitHub Copilot CLI — orchestrator integration reference

Authoritative integration reference for hosting the **GitHub Copilot CLI** inside a remote/web/mobile multi-agent orchestrator. Binary name: `copilot`. Distributed as the npm package **`@github/copilot`** (`npm install -g @github/copilot`); repo `github.com/github/copilot-cli`; docs at `https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli` and `https://docs.github.com/en/copilot/how-tos/use-copilot-agents/use-copilot-cli`.

> **Disambiguation — this is NOT `gh copilot`.** The deprecated `gh copilot` extension (`gh-copilot`, deprecated 2025-09-25) only did `gh copilot suggest`/`explain` and is irrelevant for an agent harness. The modern `gh copilot` subcommand now *downloads and shells out to* the new `copilot` binary (into `~/.local/share/gh/copilot`). Detect/launch the real binary, and disambiguate from the deprecated extension by stderr fingerprint when needed (see §1).

> **Version caveat.** Copilot CLI is young and moves fast. The structured JSONL event stream (`--output-format json`) is **version-gated** — require **≥ v1.0.28** for reliable JSONL. On-disk paths assume the default config dir `~/.copilot` (override with `COPILOT_HOME`). Several on-disk field spellings (e.g. the exact `config.json` trust-array key) and the `hooks/` lifecycle-script contract are not fully byte-verified; gate behavior on `copilot --version` and treat the structured stream (JSONL / ACP) as the authoritative integration surface rather than on-disk file shapes.

> **Three integration surfaces.** Copilot CLI exposes three modes, and the right answer to most taxonomy questions depends on which you use:
> 1. **Headless one-shot** — `copilot -p "<prompt>" --output-format json`, read JSONL events off stdout (pipe, no PTY). **Recommended for orchestrator-owned work and the richest structured signal.**
> 2. **ACP server** — `copilot --acp --stdio`, JSON-RPC (Agent Client Protocol) over stdin/stdout. Best for a long-lived programmatic session with clean permission/fs RPCs.
> 3. **Interactive TUI** — `copilot` in a real PTY. For a terminal-mirroring harness; structured-event answers do not apply (use screen-scrape / OSC fallbacks).

## 1. Discovery & identity

- **Binary detection / "installed?":** Binary is **`copilot`** on PATH (npm `@github/copilot`). Resolve by an explicit config/env override first, then PATH lookup: probe order `COPILOT_CLI_PATH` (or a harness-specific override env) → bare `copilot` → `which/LookPath copilot`. **"Installed?"** = `copilot` resolves on PATH AND `copilot --version` returns a version string. **Critical disambiguation from the deprecated `gh copilot` extension:** fingerprint stderr — only treat it as the deprecated tool when stderr contains **both** the literal `gh-copilot` **and** a deprecation marker (`has been deprecated` / `no commands will be executed`), so the new CLI's own "copilot-cli" banners are not misflagged. *(established method)*
- **Version detection / gating:** `copilot --version` / `copilot -v`, or the subcommand `copilot version` (which also checks for updates). **Gate the JSONL event stream on ≥ v1.0.28** — older builds do not emit `--output-format json` JSONL reliably. *(established method; exact `--version` string format unverified)*
- **Existing-session discovery (paths/globs):** Config dir defaults to **`~/.copilot/`** (override via `COPILOT_HOME`). Sessions live **one directory per session** under **`~/.copilot/session-state/<session-id>/`** (dir named by the session UUID). Each session dir contains **`events.jsonl`** (append-only event log = full transcript), **`workspace.yaml`** (metadata incl. repo/branch/cwd), and **`checkpoints/`** (snapshot history). An aggregate SQLite DB **`~/.copilot/session-store.db`** holds searchable metadata (powers `/chronicle` and `/session`). **Globs:** `~/.copilot/session-state/*/` for sessions; `~/.copilot/session-state/*/events.jsonl` for transcripts. *(established method)*
  - **Conflicts / open questions:** One source describes the transcript as a flat file `~/.copilot/session-state/*.jsonl` (one `.jsonl` per session, no subdir). The other (matching official config-dir docs) describes the per-session **directory** layout `session-state/<id>/events.jsonl` + `workspace.yaml` + `checkpoints/` + the `session-store.db`. **Prefer the per-directory layout** (it matches the documented config-dir reference and SQLite metadata store), but scan defensively for both flat `*.jsonl` and `*/events.jsonl` to be robust across versions. Exact per-file field names on disk are unverified.
- **Stable session identity:** A **session UUID**, assigned at session start. In the **headless JSON stream** it is emitted as `data.sessionId` on the `session.start` event **and** as the top-level `sessionId` on the final synthetic `result` event — **capture from `session.start` first** (defensively), then let `result` override (the `result` event can be absent on crash/cancel/timeout — see §3 hazards). In **ACP mode** it is the `sessionId` returned by the `session/new` JSON-RPC result. On disk it is the **directory name** under `session-state/` and the `sessions.id` column in `session-store.db`. **No OSC-emitted id.** For a pre-existing session, take the directory name (or the `session.start` line of `events.jsonl`). *(established method)*
- **Session ↔ repo/worktree/cwd mapping:** The CLI records the working directory and (if a GitHub-hosted git repo) repo + branch in `session-state/<id>/workspace.yaml` and in the `sessions` table (`repository`, `branch` columns). At launch the orchestrator controls cwd: child process working dir (headless) or the `cwd` param in the ACP `session/new` request. Reverse-map repo→sessions by reading `workspace.yaml` / the `sessions` table. Note: the native **`--remote`** streaming feature *requires* cwd to be a git repo hosted on github.com. *(established method)*

## 2. Launch & process model

- **Spawn mechanism / PTY host:** Three models (see header). **(a) Headless one-shot** — direct `exec` of `copilot -p … --output-format json`; read JSONL from a stdout pipe (no PTY needed). Use a generous line-buffer cap (verified production code uses a **10 MB max line**) since a single JSONL event can be large. **(b) ACP** — `copilot --acp --stdio` over JSON-RPC on stdin/stdout (no PTY). **(c) Interactive TUI** — `copilot` hosted in a PTY (local / remote / daemon-subprocess), for a terminal-mirroring UI. For headless/ACP, deliver work over the pipe/RPC; for the PTY launch, deliver the command after shell-ready so any rc-file env (`COPILOT_HOME`) is exported before Copilot starts. *(established method)*
- **Full launch command / args / env:** Verified headless argv: `copilot -p "<prompt>" --output-format json --allow-all --no-ask-user [--model <m>] [--resume <id>] [extra args]`. The prompt is passed as a **single CLI argument** (`-p`), not on stdin. **Env Copilot reads:** `COPILOT_HOME` (config dir), `COPILOT_MODEL`, `COPILOT_ALLOW_ALL`, and auth tokens with precedence `COPILOT_GITHUB_TOKEN` > `GH_TOKEN` > `GITHUB_TOKEN`. **Env injected by the orchestrator:** inherit the process env; for ACP force a stable `HOME` for the child. **Env to strip:** when relying on the CLI's *own* `~/.copilot` login (recommended), avoid injecting conflicting `*_TOKEN` vars; if you shell `gh auth token` as a fallback, strip `GITHUB_TOKEN`/`GH_TOKEN` so `gh` reads its own store instead of echoing the env back. **Block user-supplied overrides** of the harness-controlled flags (`-p`, `--output-format`, `--resume`, `--acp`, `--allow-all`/`--yolo`, `--allow-all-tools`/`--allow-all-paths`/`--allow-all-urls`, `--no-ask-user`) in any pass-through "extra args" channel. *(established method)*
- **Durable backing for survival:** **No agent-specific survival mechanism.** Headless one-shot is fire-and-forget (no survival needed — persist the captured JSONL and `--resume` later, §3). For the interactive PTY case, fall back to a **generic tmux/abduco-style multiplexer**. Note: GitHub's own `--remote` feature streams a *live* session to github.com / GitHub Mobile but **requires the machine to stay online** — it is **not** a detach/survival mechanism. *(falls back to generic mux for PTY; not supported natively)*
- **Initial-prompt injection mode:** **Headless = argv flag** `-p "<prompt>"` (single argument, not stdin). **ACP** = prompt delivered in the `session/prompt` JSON-RPC params as `prompt: [{type:"text", text:…}]`. **Interactive** = type-after-ready into the PTY then submit with **Enter** (bracketed paste for multi-line; see §6). *(established method per transport)*
- **First-run / trust handling:** On first interactive run the CLI shows a **directory-trust menu** (≈3 options: proceed this session / remember this folder / exit), which can eat the first message. To pre-seed and avoid it: add the cwd to the **trusted-folders array in `~/.copilot/config.json`** ahead of launch (use the realpath; preserve unrelated keys such as `loggedInUsers`/auth data and refuse to overwrite a corrupted config). For headless, the trust TUI does not block when allow flags are set — pass `--allow-all`/`--allow-all-paths`/`--add-dir`/`--no-ask-user`. There is **no documented `--trust` flag**; file pre-seeding is the only declarative bypass for the interactive path. *(established method = config pre-seed for TUI; allow-flags for headless)*
  - **Conflicts / open questions:** The exact JSON key for the trust array in `config.json` is reported as both `trustedFolders` and `trusted_folders` and is **unverified** (no live binary inspected). Read the existing array under whichever key is present and append to it rather than assuming the spelling; preserve all sibling keys.
- **Permission / YOLO / sandbox modes:** **`--allow-all`** (alias **`--yolo`**) grants all permissions; finer-grained: `--allow-all-tools`, `--allow-all-paths`, `--allow-all-urls`, `--allow-tool=TOOL`, `--deny-tool=TOOL`, `--allow-url=URL`, `--add-dir=DIR`; **`--no-ask-user`** stops the agent pausing for input. Env equivalent: `COPILOT_ALLOW_ALL=true`. Sandbox is an interactive slash command **`/sandbox enable`**. For non-interactive orchestrator work, the standard combo is `--allow-all --no-ask-user`. *(established method)*
- **Model / reasoning-effort / fast-mode at launch:** **Model:** `--model=<m>` (e.g. `gpt-5.2`, `claude-sonnet-4.6`, `claude-sonnet-4.5`, `haiku-4.5`), env `COPILOT_MODEL`, or `--model auto` to let Copilot pick. **Reasoning-effort:** **no documented dedicated launch flag** (`Ctrl+T` toggles *display* of reasoning, not effort). **Fast mode:** **no dedicated flag** — the closest is picking a small/fast model (`haiku-4.5`) or `--model auto`. *(established method = `--model`; reasoning-effort = not supported / no known flag; fast mode = no flag, pick a small model)*
  - **Conflicts / open questions:** One source documents headless flags `--silent`/`--stream off`/`--no-custom-instructions` and a `--effort <low|medium|high>` reasoning knob (plus a `*-fast` model id). The other (verified production code + official cheatsheet) finds **no `--effort` flag** and treats reasoning-effort as not exposed. Treat `--effort` and `*-fast` as **unverified / version-dependent**; rely on `--model` for model/speed selection and the documented permission/output flags. Use `-s` (or `--silent`) to strip stats/decoration for clean one-shot text output (verified for the plain-text path).

## 3. Resume & reattach

- **Resume support / flags / id source:** **Yes.** `--resume[=VALUE]` where VALUE is an optional **session id, id-prefix, or session name** (no value → interactive picker); **`--continue`** resumes the most recent session in the current cwd (falling back to the globally most recent); slash command **`/resume`** inside an interactive session. Headless: pass `--resume <session-id>` where the id came from `session.start`/`result.sessionId` (or the `session-state/<id>` dir name). *(established method)*
  - **Conflicts / open questions:** One source states resume is **not supported** for this CLI. The other documents `--resume`/`--continue`/`/resume` from official docs and verified production code that passes `--resume <id>`. **Prefer "resume IS supported"** — it is in the official CLI docs and exercised by working integrations. (The "not supported" claim describes a specific harness that simply never wired resume, not a CLI limitation.)
- **State restored vs lost on resume:** Restored from `events.jsonl` + `workspace.yaml` + `checkpoints/`: full conversation history, context, tracked files, plans, checkpoints. **Re-supply `--model` on resume** to be safe (model is a launch setting). **`--remote` must be explicitly re-enabled on resume** (docs explicit: it is *not* remembered). The live PTY screen (interactive mode) is not persisted — the TUI re-renders from history. *(established method)*
  - **Conflicts / open questions:** Exactly which of {model, cwd, permission grants} persist across resume vs. need re-passing is **unverified**; `--remote` is confirmed to need re-enabling.
- **Reattach to a running PTY after restart:** **Headless/ACP** — there is no PTY to reattach to; the one-shot process does not survive a restart. Recovery = persist the captured JSONL + the on-disk session, then `--resume <id>` to continue in a *new* process (cleaner than terminal-buffer reattach — replay structured events, §11). **Interactive TUI** — no native reattach; fall back to a **generic multiplexer reattach + redraw nudge (SIGWINCH)** as for other TUI agents. *(established method headless = `--resume`; TUI = generic mux + SIGWINCH)*
- **Idempotency / race hazards:** (a) **Capture `sessionId` from `session.start`, not just `result`** — the synthetic `result` event may never arrive (timeout, cancel, crash, or a `session.error` before result), so a resume pointer taken only from `result` can drift to a stale turn. (b) For interactive reattach, generic hazards apply: only one PTY client should hold the multiplexer master (double-attach → echo-doubling); check master-alive before reporting exit; don't resume a session already live in another client. *(guard rules)*

## 4. Auth & subscription

- **Credential locations:** Three sources, in resolution order. **(1) Env vars**, precedence `COPILOT_GITHUB_TOKEN` > `GH_TOKEN` > `GITHUB_TOKEN`. **(2) `gh auth token`** (GitHub CLI credential store, `hosts.yml`) as a fallback. **(3) The CLI's own login store** written by `copilot login`, persisted in **`~/.copilot/config.json`** (holds authentication data incl. `loggedInUsers`; treat as keys to *preserve*, not to read/mutate). **Token types:** supported = `gho_` (OAuth, default from `copilot login`), `github_pat_` (fine-grained PAT with Copilot Requests permission), `ghu_` (GitHub App); **NOT supported** = `ghp_` (classic PAT). *(established method)*
- **Reuse model:** Two viable paths. **(1) spawn-and-let-CLI-auth (simplest, recommended):** rely on an existing `copilot login` / `~/.copilot` session; do not inject tokens. **(2) orchestrator-managed token injection (multi-account):** set `COPILOT_GITHUB_TOKEN`/`GH_TOKEN` per spawn, and/or point each account at a separate config dir via **`COPILOT_HOME`**. For *direct Copilot API* reuse, the raw GitHub token can be exchanged for a short-lived Copilot API token (see refresh). *(both established)*
- **Token refresh / write-back:** The CLI owns refresh of its own `gho_` OAuth token. For orchestrator-driven auth: OAuth **device-code flow** (client id `Ov23li8tweQw6odWQebz`, the shared Copilot-CLI client id) against `https://github.com/login/device/code` + `https://github.com/login/oauth/access_token`. For *direct Copilot API* use, exchange a raw GitHub token at **`GET https://api.github.com/copilot_internal/v2/token`** (headers `Authorization: token <raw>`, `Editor-Version: vscode/<v>`, `User-Agent: GitHubCopilotChat/<v>`) → a semicolon-separated bearer token + `expires_at`; cache in-process and refresh ~120 s before expiry. *(established method; CLI refreshes its own token)*
- **Env hygiene:** When letting the CLI use its own `~/.copilot` login, avoid injecting conflicting `*_TOKEN` vars (they take precedence over the stored login). When shelling `gh auth token`, strip `GITHUB_TOKEN`/`GH_TOKEN` from that subprocess so `gh` reads its own store. *(established method)*
- **Multi-account isolation / selection:** Two knobs — **per-spawn token env** (`COPILOT_GITHUB_TOKEN`) and **per-account config dir via `COPILOT_HOME`** (point each account at a separate `~/.copilot`-like dir; `config.json` carries `loggedInUsers`). There is no in-CLI account switcher. *(established method)*
  - **Conflicts / open questions:** One source states there is **no orchestrator-side account capture/store/switch** for this CLI; the other documents the `COPILOT_HOME` + token-env isolation pattern from working auth code. The CLI *supports* both isolation knobs; whether a given harness wires them is a harness choice, not a CLI limitation.

## 5. Models, usage & accounting

- **Model listing (settings UI):** **No documented machine-readable `copilot models --json` subcommand.** Practical enumeration sources: the `--model` help text (example slugs `gpt-5.2`, `claude-sonnet-4.6`, `claude-sonnet-4.5`, `haiku-4.5`), the interactive **`/model`** picker (lists currently-available, policy-filtered models), `--model auto` (defers selection), or the **Copilot models API** (`GET https://api.githubcopilot.com/models` with an exchanged Copilot token). Because the picker is **policy-filtered per account/org**, a settings UI should keep a **curated/static catalog** locally and treat it as the source of truth, refreshed manually or via the models API. *(established method = static/curated catalog; fallback = models API or `/model` picker; no dedicated CLI list subcommand)*
  - **Conflicts / open questions:** One source carries a large hard-coded model catalog (auto + several Claude `claude-*-4.x` + GPT `gpt-5.x`) with a `gpt-5.4` default and OpenAI thinking levels per GPT model. The other finds only a short verified example set and no per-model thinking-level knob. **Use a curated static list** (it is the only reliable enumeration), but verify slugs against the live `/model` picker / models API per account — the exact catalog drifts and is account-policy-dependent.
- **Runtime switching mid-session:** **Yes — `/model`** slash command switches model mid-conversation (interactive). **Reasoning-effort:** no switch (`Ctrl+T` only toggles *display* of reasoning). **Fast mode:** switch to a small model via `/model`. Headless/ACP are single-shot per process — "switch" = start a new `--resume` turn with a different `--model`. *(established method = `/model`; reasoning-effort switch = not supported)*
- **Usage windows / quota / rate-limits:** **Session-level reads exist; account-level quota does not (via the CLI).** Interactive: **`/usage`** (session stats + token usage) and **`/context`** (visual token-usage overview). **Headless:** the final **`result` event carries `usage`** with `premiumRequests` (Copilot's billing unit), `totalApiDurationMs`, `sessionDurationMs`, and `codeChanges{linesAdded, linesRemoved, filesModified[]}` (verified struct). **Account-level remaining quota / reset windows are NOT exposed via the CLI** — they require GitHub's Copilot billing/usage API separately. *(established method = `/usage`/`result.usage`; account quota = not supported via CLI, external API)*
- **Token accounting:** **Partial.** Per-turn output tokens: **`assistant.message.outputTokens`** (accumulate per model). Per-session billing: **`result.usage.premiumRequests`**. On disk, the `session.shutdown` event records model metrics + token totals. **Input / cache / reasoning token counts are NOT reliably present** in the verified event struct — only output tokens + premium-requests are dependable; richer per-turn input/cache/reasoning breakdown is **unverified**. *(established method for output tokens + premium requests; input/cache/reasoning = unverified)*
- **Pricing / cost:** **Not exposed as currency.** Copilot bills in **premium requests**, not per-token dollars; `result.usage.premiumRequests` (float) is the cost signal. No per-token USD rates are emitted; mapping premium-requests → $ depends on the user's Copilot plan (external pricing). The orchestrator must supply its own plan→price mapping. *(cost = premium-request count; USD pricing not exposed — orchestrator-owned)*

## 6. Driving & input — two-way control

> Headless `-p` and ACP modes are **single-shot per process** — you don't "type mid-session"; you start a new `--resume`/`session/prompt` turn. The live-steering answers below apply to the **interactive PTY** unless noted.

- **Send a new user message mid-session:** **Interactive PTY** = type into the composer and submit with **Enter**; large/multi-line content via **bracketed paste** (`\x1b[200~` … `\x1b[201~`) followed by Enter to submit (a short post-paste submit delay, ~50 ms, avoids races). **Headless** = a new `copilot -p … --resume <id>` process (a fresh turn on the same session). **ACP** = a new `session/prompt` RPC. GitHub's `--remote` lets messages come from web/mobile to a live session. *(established method per transport)*
- **Read the agent's input-box contents (draft sync out):** **Not supported natively** — no API/OSC/transcript exposure of the live composer. The only route for the interactive PTY is **screen-scraping** the composer line (generic). For headless/ACP the draft lives in the orchestrator, so this is moot. *(not supported; screen-scrape only)*
- **Pre-fill / set the input box (draft sync in):** **No native composer prefill** — no flag or env to pre-fill the interactive composer (there is no `--draft`/prefill flag and no draft env var). `-p` only sets the *one-shot* prompt; `Ctrl+G` opens the prompt in `$EDITOR` (scriptable but fragile). Fallback = **bracketed-paste-after-ready** into the PTY, gated on a readiness signal (DECSET 2004 `\x1b[?2004h` followed by a stream-quiet window, ~1500 ms, since there is no dedicated composer-ready glyph signal). For ACP the client owns the draft entirely. *(not supported natively; fallback = typed/bracketed paste gated on ready)*
- **Answer interactive prompts / permission menus:** **Interactive PTY** — approval prompts answered with **single keys**: `y` (allow once), `n` (deny once), `!` (allow all similar this session), `#` (deny all similar this session), `?` (show details); for AskUser "more info" requests, type a reply + Enter. The *set of options* is read by **scraping the prompt line** (not parsed structurally). **Headless** — avoid prompts entirely with `--allow-all`/`--no-ask-user`. **ACP (cleanest programmatic path)** — permission requests arrive as `session/request_permission` JSON-RPC requests with an outcome object; reply with the chosen option (or `{outcome:{outcome:"cancelled"}}` to deny). State-wise, the `askUser` tool and blocking notifications map to a **waiting/blocked** state (§7). *(established method per transport; ACP authoritative for programmatic prompt handling)*
- **Interrupt / cancel / escape:** **Interactive PTY** — **`Esc`** (`0x1b`) cancels the current operation; **`Ctrl+C`** (`0x03`) cancels/clears input, **press twice to exit**; **`Ctrl+D`** (`0x04`) graceful shutdown. **Headless** — SIGTERM/SIGKILL or context cancel (then close stdout; use a wait-delay, ~10 s, to avoid hangs). *(established method)*
  - **Conflicts / open questions:** One source models interrupt as a **double-Escape** semantics (first Esc = menu/editor cancel, second Esc on the same turn = interrupt the running turn), mirroring `opencode`. The official cheatsheet documents single **`Esc` = cancel current operation** and **`Ctrl+C` ×2 = exit**. **Prefer the documented `Esc` = cancel / `Ctrl+C` to interrupt** mapping; if double-Esc behavior is observed at runtime on a given build, the harness may need the double-Esc inference as a fallback. Verify against the live TUI.
- **Slash commands the orchestrator may issue:** `/resume`, `/model`, `/usage`, `/context`, `/diff`, `/session`, `/chronicle`, `/mcp` (`/mcp add`), `/sandbox enable`, `/compact`, `/clear`, `/login`, `/cwd` (`/cd`), `/add-dir`, `/agent`, `/plan`, `/review`, `/delegate` (push + AI PR), `/fleet` (spawn parallel subagents), `/skills`, `/every`, `/after`, `/settings`, `/feedback`, `?` (help). *(established set per docs; exact availability drifts by version)*
- **Attachments / large paste:** `@FILENAME` injects file contents into the prompt; `!COMMAND` runs a shell command inline (bypassing Copilot). The Copilot API supports vision (`Copilot-Vision-Request: true` header); **CLI image/attachment UX is unverified**. Large paste = generic bracketed-paste into the PTY (handled atomically). *(established for `@`/`!`; image attachment = unverified)*
- **Important keyboard shortcuts (mobile soft-keyboard surface):** Verified set — **submit = Enter**; **interrupt = `Esc`** and **`Ctrl+C`** (×2 to exit); **`Ctrl+D`** shutdown; **`Ctrl+L`** clear screen; **↑/↓** history nav; **`Ctrl+G`** external-editor; **`Ctrl+O`** expand latest timeline item; **`Ctrl+E`** expand all timeline items (also toggles the `--remote` QR code in remote mode); **`Ctrl+T`** toggle reasoning display; **`Shift+Tab`** cycle modes (standard → plan → autopilot); **`Ctrl+X` then `/`** insert a slash command mid-prompt; **`@`** file inject; **`!`** shell escape. Mobile surface to expose: Enter, Esc, Ctrl+C, Tab/Shift+Tab, ↑/↓, `/`, `@`, `!`. Only the Ctrl-chords + Shift+Tab need modifier fidelity; **no Alt/Option/Meta-specific bindings** found. *(established method)*
  - **Conflicts / open questions:** The composer newline key (Shift+Enter vs. Alt+Enter) is **unverified**.

## 7. Agent-state classification

- **State vocabulary:** Maps to `working | waiting (on permission/AskUser) | done | error/blocked` + `interrupted/aborted/timeout`. *(established method)*
- **Signal sources & authority rank:**
  1. **Headless JSONL event stream (highest authority, richest):** `--output-format json` events drive state directly (table below).
  2. **ACP `session/update` notifications** (`agent_message_chunk`, `agent_thought_chunk` → working; `session/request_permission` → waiting-on-user; prompt RPC returning → done). Cleanest discriminator for "waiting-on-user."
  3. **Process exit code** (secondary): exit cause distinguishes timeout (deadline) vs aborted (cancel) vs failed (nonzero).
  4. **On-disk `events.jsonl` tail / `session-store.db`** for a non-attached observer.
  5. **Lifecycle hooks** (`~/.copilot/hooks/`) — an alternative event source if a hook script is installed (§8); event names are less verified than the JSONL stream.
  6. **OSC title / output activity** (interactive PTY only) — generic fallback. The PTY window title is recognizable as **`GitHub Copilot`**.
  There is **no OSC-9999 JSON status channel** emitted by Copilot. *(established method; JSONL stream authoritative)*
- **Event → state mapping (verified JSONL/ACP):**

  | Event | State |
  |---|---|
  | `session.start` | working (session begun) |
  | `assistant.turn_start` | working (running) |
  | `tool.execution_start` / `toolRequests` in `assistant.message` | working |
  | `assistant.message` / `assistant.message_delta` / `assistant.reasoning(_delta)` | working (streaming) |
  | `tool.execution_complete` (success or fail) | working (turn continues) |
  | ACP `session/request_permission`; `askUser` tool; blocking notification (permission prompt / elicitation dialog) | **waiting-on-user / blocked** |
  | `session.warning` | working + warning surfaced |
  | `session.error` | error/failed |
  | final `result` exitCode 0 / `session.shutdown` | done |
  | final `result` exitCode ≠ 0 / nonzero process exit | failed |

- **"Stopped but needs the user" vs "done":** **Headless** with `--no-ask-user` never stops to ask, so "stop" ⇒ done/failed only. **ACP** is the clean discriminator: a `session/request_permission` (or `fs/*` request) arriving = waiting-on-user, distinct from the prompt RPC resolving = done. **Interactive** = detect the permission prompt by screen-scrape (the `y/n/!/#/?` line). Reserve `blocked`/`waiting` for `askUser` tool calls and blocking notifications; plain `Stop`/`SessionEnd` → `done`. *(established method)*
- **Reconciliation:** Resolve conflicts by precedence — `result.exitCode` and `session.error` override a default `completed`; process-exit cause (timeout/cancel) overrides; once `failed`, it sticks. Fold streaming deltas into the authoritative full `assistant.message` (reset the delta buffer on the full message to avoid double-counting). Events are discrete, so no quiet-window is needed for the structured stream; debounce/quiet-window applies only to the screen-scrape (interactive) path. Treat `session.start`/`UserPromptSubmit` as new-turn markers that reset per-turn state. *(established method)*
- **Sub-agent / nested-task state:** Copilot has real sub-agents — the `task` tool (`agent_type: general-purpose|explore`), `/fleet` (parallel subagents), `list_agents`/`read_agent`. Their progress surfaces as `tool.execution_*` events on the parent stream, and envelope events carry a **`parentId`** for linkage. (If using the lifecycle-hooks path, sub-agent start/stop hook events exist but may not map to a distinct state.) *(established method via `parentId`)*
- **Latency / debounce:** JSONL events are near-real-time (streaming deltas); no debounce needed for the event stream. Debounce only the screen-scrape (interactive) path. *(established method)*

## 8. Hooks & instrumentation install

- **Mechanism:** **Two viable surfaces.** **(1) The structured event stream is the preferred instrumentation — no install needed:** `--output-format json` JSONL (headless) or ACP (live) already deliver structured lifecycle events without writing any hook file. **(2) Native lifecycle hooks:** the config dir has a dedicated **`~/.copilot/hooks/`** directory for lifecycle hook scripts wired through Copilot's hooks JSON (plus `skills/`, `agents/` for `*.agent.md`, `installed-plugins/` + `plugin-data/`, `extensions/`, and MCP via `mcp-config.json` / `mcp-oauth-config/` / `mcp-secrets/`). For an orchestrator that owns the pipe, **prefer (1)**; use (2) only when instrumenting an interactive PTY session you don't otherwise control. *(established method = structured stream; hooks = fallback)*
- **Install location & trust (hooks path):** Hooks config under `~/.copilot/` (COPILOT_HOME-aware); a managed shell script (`.sh` posix / `.ps1` windows) registered per event. MCP servers → `~/.copilot/mcp-config.json` (or `copilot mcp add` / `/mcp add`). Plugins → `installed-plugins/`, tracked in `config.json` (`installedPlugins`). Trust = directory trust via the trust-folders array in `config.json`; MCP OAuth in `mcp-oauth-config/`. No config-hash trust step is required for the hook script (unlike some agents). *(established method; exact hook-dir script contract / event names unverified)*
- **Hook event taxonomy + payload:** The richest **verified** taxonomy is the **JSONL stream** (§7/§10): `session.start{sessionId,selectedModel}`, `assistant.turn_start`, `assistant.message{messageId,content,toolRequests[],outputTokens,reasoningText}`, `assistant.message_delta{deltaContent}`, `assistant.reasoning(_delta)`, `tool.execution_start`, `tool.execution_complete{toolCallId,model,success,result{content,detailedContent},error{message}}`, `session.warning`, `session.error{errorType,message}`, synthetic `result{sessionId,exitCode,usage}`. Envelope: `{type, data, id, timestamp, parentId, ephemeral}`. The **lifecycle-hooks** path uses a separate, VS Code-compatible PascalCase event set: `SessionStart`, `SessionEnd`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `subagentStart` (documented in camelCase), `SubagentStop`, `PreCompact`, `Stop`, `ErrorOccurred`, `PermissionRequest`, `Notification`. Hook payload fields worth extracting: tool name/args (`tool_name`/`toolName`/`name`, `tool_input`/`toolInput`/`arguments`), `notification_type` (e.g. `permission_prompt`, `elicitation_dialog`), `error_context`/`recoverable`, and `transcript_path` (last-assistant recovery). *(JSONL taxonomy verified; lifecycle-hook event names less verified)*
- **Transport back to orchestrator:** **Native (preferred):** stdout JSONL pipe (headless) or stdin/stdout JSON-RPC (ACP) — **no port/token callback needed**; the orchestrator owns the pipe. **Fail-open:** keep captured text/deltas even if the final `result` never arrives. **Hooks path (if used):** a managed hook script POSTs to a local HTTP endpoint with an injected port + token, printing `{}` to stdout first (Copilot consumes hook stdout) and exiting 0 on any missing-env / empty-payload / network failure (fail-open); the script can re-source its endpoint env so surviving PTYs find the current server after a restart. *(established method)*
- **Lifecycle:** Headless/ACP need no install/uninstall (just spawn). For hooks/MCP/plugins, edits to the JSON files / `copilot mcp` subcommand are the integrator's responsibility; make hook installs idempotent (remove managed entries before re-adding) and carry a protocol-version marker to detect skew. *(established method)*

## 9. Session titles

- **Source of truth:** The CLI **auto-generates a session summary/title**, stored as `sessions.summary` in `session-store.db`, shown in the `--resume` picker and `/session`/`/chronicle`. The structured stream emits **no explicit title event**. *(established method)*
- **How/when read & updated:** Read from `session-store.db` (`sessions.summary`), or derive your own tab title from the **first user prompt** (the `-p` text you sent) or the first `assistant.message`. There is no title-update event in the JSONL stream — re-read the DB or use the first-prompt. *(established method)*
- **Fallback / synthetic:** First-prompt truncation/regex (the orchestrator already has the `-p` prompt at launch), or an LLM-generated title via a headless one-shot (§12). *(fallback = generic)*

## 10. Transcript & history

- **Storage:** Per-session directory **`~/.copilot/session-state/<session-id>/`** with **`events.jsonl`** (append-only JSON-Lines event log = full transcript), **`workspace.yaml`** (repo/branch/cwd), and **`checkpoints/`** (snapshots). Aggregate **`session-store.db`** (SQLite; tables include `sessions`, `turns`, `checkpoints`, `session_files`, `session_refs`, and an FTS5 `search_index`). One `events.jsonl` per session. *(established method; see §1 conflict on flat-vs-directory layout — exact on-disk field names unverified)*
- **Parsing / schema mapping (verified via the JSONL stream; same envelope on disk):**
  - Envelope: `{type, data, id, timestamp, parentId, ephemeral}`.
  - **Roles/messages:** `assistant.message.content` (assistant text), `assistant.message_delta.deltaContent` (streaming), user messages = the `-p`/prompt text (and the `turns` table on disk). On disk a user message may carry a `transformedContent` variant alongside raw `content` (prefer `transformedContent` for title/preview when present).
  - **Tool calls:** `assistant.message.toolRequests[]` = `{toolCallId, name, arguments(json), type, intentionSummary}`; results in `tool.execution_complete` = `{toolCallId, success, result{content, detailedContent}, error{message}}`.
  - **Thinking/reasoning:** `assistant.message.reasoningText`, and `assistant.reasoning`/`assistant.reasoning_delta` (`content`/`deltaContent`).
  - **Diffs/code changes:** `result.usage.codeChanges{linesAdded, linesRemoved, filesModified[]}`; per-file detail on disk in `session_files`.
  - **Errors:** `session.error{errorType,message}`, `session.warning{warningType,message}`.
  - **cwd:** recoverable from `workspace.yaml` (or, on some builds, a `session.info` message string in the transcript).
  - **Tokens:** `assistant.message.outputTokens` per turn; `session.shutdown` records cumulative token totals + model metrics on disk.
  *(established method)*
- **Live tail vs historical:** **Live** = read the JSONL stdout stream in real time (line scanner) or ACP `session/update` notifications. **Historical** = tail/read `events.jsonl` on disk, or query `session-store.db` (FTS5 search). Detect new entries by file append / new JSONL lines; order by stream order + `timestamp`; dedup deltas-vs-full-message by resetting the delta buffer when the full `assistant.message` arrives. *(established method)*
- **Consumers:** Chat view ← `assistant.message`(+deltas), reasoning, tool requests/results. Last-message/preview ← last `assistant.message.content`. Clickable items ← `toolRequests[].arguments` (file paths), `codeChanges.filesModified`, `session_refs` (commits/PRs/issues). State classification ← §7 events. Session-list metadata (title, model, token total, message count, cwd) ← `session-store.db` / `workspace.yaml`. *(established method)*
- **Transcript ↔ terminal display:** Headless/ACP have **no terminal** — the JSONL/JSON-RPC *is* the transcript; render it directly (no PTY reconciliation). Interactive mode needs generic transcript↔PTY reconciliation. *(established method)*
- **Special content types / quirks:** Reasoning may arrive as both a delta stream **and** a full `reasoningText` on `assistant.message` — **dedup required**. `tool.execution_complete` has `result.content` vs `result.detailedContent` (short vs long). `ephemeral:true` marks transient envelopes (drop from history). Built-in tool names worth special-casing for clickability: `view`, `create`, `edit`, `bash`, `grep`, `glob`, `web_fetch`, `task`, `sql`, `store_memory`, `report_intent`, async shell (`write_bash`/`read_bash`/`stop_bash`/`list_bash`), and GitHub MCP (`github-mcp-server-*`). Images: vision-capable via API header, but CLI attachment storage is **unverified**. *(established method)*

## 12. Background / headless invocation

- **Non-interactive one-shot:** **First-class and the recommended orchestrator path.** Argv: `copilot -p "<prompt>" --output-format json --allow-all --no-ask-user [--model <m>] [--resume <id>]`. Output: JSONL events on stdout (final `result` line has `sessionId`/`exitCode`/`usage`); or plain text with `-p` alone (`-s`/`--silent` strips stats/decoration for clean text). Prompt delivery = **CLI argument** (`-p`), not stdin. ACP (`--acp --stdio`) is the alternative for a long-lived programmatic session. *(established method)*
- **Uses:** Commit messages, PR titles/bodies (there is even a `/delegate` for AI PRs), branch names, summaries, title generation (§9), one-shot code edits, model-backed analytics. With `--allow-all` it also *does* file/tool work headlessly. Model discovery is not available as a subcommand (§5). *(established method)*
- **Auth path for headless:** Inherited account — env tokens (`COPILOT_GITHUB_TOKEN`/`GH_TOKEN`/`GITHUB_TOKEN`), `gh auth token`, or the CLI's own `~/.copilot` login; per-account via `COPILOT_HOME`/token env. *(established method)*
- **Cost/latency/timeout/output caps:** Use a generous run timeout (verified production code uses **20 min**), a wait-delay (~10 s) on cancel, and a **10 MB max JSONL line** buffer; on timeout → status `timeout`. Cost = `premiumRequests` from `result.usage`. `--remote` streaming has a **60 MB output limit**. Local subprocess execution; model calls go to GitHub's Copilot backend. *(established method)*

## 14. Failure, exit & recovery

- **Crash/exit detection:** **Headless** — process **exit code** plus the synthetic `result.exitCode`. Distinguish: clean (`completed`, exit 0, no `session.error`), `failed` (nonzero exit / `session.error` / wait error), `timeout` (deadline exceeded), `aborted` (cancel). `session.error{errorType,message}` is the in-band error event; `session.warning` is non-fatal. Tail stderr and append it to the error message. `PostToolUseFailure`/`ErrorOccurred` (hooks path) surface error text; an `ErrorOccurred` with `recoverable === true` → still working, else done. **ACP** — errors arrive as JSON-RPC `error` objects. *(established method)*
- **Reattach-after-restart healing:** Headless one-shot does not survive a restart — recover by `--resume <persisted session id>` (a fresh process continuing the on-disk session). Interactive PTY — generic exited-but-alive healing + multiplexer reattach. Use a wait-delay (~10 s) and close stdout on cancel to avoid orphan/zombie hangs. *(established method)*
- **Error surfacing:** `session.error` → error state; nonzero exit → failed with `"copilot exited with code N"` + stderr tail; timeout/abort messages distinct. In the hooks path, the captured error text becomes the last-assistant message. *(established method)*

## 15. Remote / transport

- **Supported transports:** **Local subprocess** (headless `-p` or `--acp --stdio`) — the primary, fully-supported path. **SSH / remote worktree / daemon-subprocess** — supported generically (it is a plain CLI over stdin/stdout; spawn on the remote host in the worktree, forward the JSONL/JSON-RPC over the relay). No agent-specific transport limitation for the structured modes. *(established method)*
- **Native remote (GitHub's own):** `copilot --remote` / `--continue --remote` streams a *live* session to **github.com and the GitHub Mobile app** with a QR code (`Ctrl+E` toggles it). Constraints: cwd must be a **github.com-hosted git repo**, **60 MB output limit**, **machine must stay online**, `--remote` must be **re-enabled on resume**, and some slash commands are unavailable from the remote interface. This overlaps a custom orchestrator's own mobile UI — for a custom orchestrator the JSONL/ACP path is preferable. *(established method; overlaps own UI)*
- **Forwarding PTY/git/fs/instrumentation over a relay:** **Headless/ACP** — forward the stdout JSONL / stdin-stdout JSON-RPC frames; in ACP also service the `fs/read_text_file` / `fs/write_text_file` / `session/request_permission` RPCs client-side (apply path-confinement to cwd and deny credential-file writes). **Interactive PTY** — generic PTY-over-relay. Git/fs ops run on the host where `copilot` is spawned (the worktree cwd). For the hooks path, a managed hook script can be installed remotely over SFTP (always the POSIX script) with its HTTP callback pointed at the host port/token. *(established method)*
