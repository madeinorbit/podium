# OpenAI Codex CLI — orchestrator integration reference

Authoritative integration reference for hosting the OpenAI Codex CLI inside a remote/web/mobile multi-agent orchestrator. Binary name: `codex`. Distributed as the npm/Bun package `@openai/codex` — a thin JS launcher (`.../@openai/codex/bin/codex.js`) that shells to a platform-native Rust binary in a sub-package (e.g. `@openai/codex-linux-x64/vendor/x86_64-unknown-linux-musl/bin/codex`). Verified against `codex-cli 0.141.0`.

> **Version caveat.** Codex moves fast. Many capabilities below (`hooks`, `plugins`, `app-server`, `remote-control`, `fork`, `archive`/`delete`, `multi_agent` subagents, `--remote` TUI, fast mode) are recent and may be absent or feature-flagged on older builds. Several feature names are even individually toggleable per install (`codex features list`). **Gate all behavior on `codex --version`** and, where relevant, on `codex features list`. On-disk paths assume the default `CODEX_HOME=~/.codex`.

---

## 1. Discovery & identity

- **Binary detection / "installed?":** Binary is `codex` on PATH (often a version-manager symlink, e.g. `~/.bun/bin/codex -> .../@openai/codex/bin/codex.js`). Resolve to an absolute path by probing PATH then version-manager dirs (nvm/volta/asdf/fnm/mise/bun/pnpm/yarn; `.cmd/.exe/.bat` on Windows), falling back to bare `codex`. The launcher sets `CODEX_MANAGED_BY_BUN` / `CODEX_MANAGED_BY_NPM` / `CODEX_MANAGED_PACKAGE_ROOT`. The packaged platform binary reports as `codex-<arch>` (e.g. `codex-aarch64-…`, `codex-x86_64-…`) — recognize by the `codex-` prefix in process tables, and recognize the install path by `node_modules/@openai/codex/`. **"Installed?"** = `which codex` resolves AND `codex --version` returns `codex-cli <semver>`. No known aliases/renames; in-CLI shorthands `e`=`exec`, `a`=`apply`. **"Signed in?"** is a *separate* gate: `<CODEX_HOME>/auth.json` exists (§4). *(established method)*
- **Version detection / gating:** `codex --version` → `codex-cli 0.141.0`. Richer probes: `codex doctor --json` (redacted machine-readable health: config/auth/runtime), `codex app-server daemon version` (CLI + running app-server versions as JSON), and `codex features list` (rows `<feature> <stage> <effective-bool>`; stages stable/experimental/under-development/deprecated/removed). Behavior gating is required because resume/fork/hooks/app-server flags differ across builds; e.g. modern Codex (≈0.129+) silently drops untrusted hooks unless a per-hook trust hash is present, and the deprecated `codex_hooks` feature flag was renamed to `hooks` (≈0.133). *(established method: `--version`; doctor/features = richer fallbacks)*
- **Existing-session discovery (paths/globs):** Session "rollout" transcripts: `<CODEX_HOME>/sessions/YYYY/MM/DD/rollout-<UTC-timestamp>-<uuid>.jsonl` (e.g. `~/.codex/sessions/2026/06/19/rollout-2026-06-19T10-20-23-019edef7-832d-7701-9a36-892f46e2a3fa.jsonl`). **Glob:** `~/.codex/sessions/**/rollout-*.jsonl` (walk `*.jsonl` recursively for robustness across layouts). Supporting files in `<CODEX_HOME>`: `session_index.jsonl` (lightweight rows `{id, thread_name, updated_at}`), `history.jsonl` (cross-session *input* history rows `{session_id, ts, text}`), and a SQLite thread store (`state_5.sqlite`, plus `goals_1.sqlite`, `memories_1.sqlite`, and the large telemetry `logs_2.sqlite`). The `sqlite` feature is `removed/true`, i.e. SQLite persistence is now the default backing store (the rollout JSONL remains the readable transcript). When the orchestrator manages its own `CODEX_HOME`, scan **both** that managed `sessions/` tree and the system `~/.codex/sessions`. *(established method)*
- **Stable session identity:** A **UUIDv7** thread/conversation id (e.g. `019edef7-3e34-7513-92b9-35f3a0dac891`), embedded both in the rollout filename and in the first JSONL line `session_meta.payload.id`. This id is accepted by `codex resume <id>`, `codex fork <id>`, and `codex exec resume <id>` (a thread *name* is also accepted; UUID parses take precedence). The binary exports it to child processes/hooks as env `CODEX_THREAD_ID`. **No OSC-emitted id.** At launch the new id is obtained by: (a) reading the meta line of the newest `rollout-*.jsonl` after start (disambiguate first — see §3 race hazards); (b) parsing a hook payload's `session_id`; or (c, cleanest) driving `app-server`/`mcp-server` JSON-RPC, whose response carries the thread id. For a pre-existing session, take `session_meta.payload.id` (fall back to filename basename). *(established method)*
- **Session ↔ repo/worktree/cwd mapping:** `session_meta.payload.cwd` + `session_meta.payload.git` carry the working dir and git context; each turn also records `turn_context.payload.cwd` and `turn_context.payload.workspace_roots`. Reverse-map repo→sessions by scanning rollouts and matching `cwd` via canonical-path containment. `codex resume`/`fork` default to **cwd-filtering** their picker (pass `--all` to show all sessions with a CWD column). `session_meta.payload.source` distinguishes a user thread (`"cli"`) from a nested agent (`{"subagent":{…}}`), and `thread_source` is `"user"` vs `"subagent"`. Mapping is cwd-derived, not an explicit id link. *(established method)*

## 2. Launch & process model

- **Spawn mechanism / PTY host:** Two distinct models. **(a) Interactive TUI** — `codex [PROMPT]` is a full-screen alt-screen ratatui app; host it in a PTY (local / remote / daemon-subprocess). **(b) Programmatic** — `codex app-server` / `codex mcp-server` speak JSON-RPC over stdio (no PTY) and are the *intended* headless orchestration surface (§6/§15). For a terminal-mirroring UI use the PTY+TUI path; for clean headless control use app-server. For the PTY launch, deliver the command after the shell signals readiness (rather than blind-pasting into xterm) so any rc-file wrapper that re-exports `CODEX_HOME` runs *before* Codex starts. *(established method)*
- **Full launch command / args / env:** `codex [-C <dir>] [-m <model>] [-c key=val …] [-s <sandbox>] [-a <approval>] [--search] [-i <img> …] [-p <profile>] [--enable/--disable <feature>] [PROMPT]`. **`CODEX_HOME`** (default `~/.codex`) relocates **all** config/auth/sessions — the canonical multi-account / isolation knob. Env Codex *reads*: `CODEX_HOME`; auth `OPENAI_API_KEY` / `CODEX_API_KEY` / `CODEX_ACCESS_TOKEN`; `OPENAI_ORGANIZATION`; base-URL overrides `CODEX_API_BASE` / `CODEX_URL` / `openai_base_url`; `NO_COLOR`, `TERM`/`COLUMNS`/`LINES`; and internal overrides incl. `CODEX_INTERNAL_ORIGINATOR_OVERRIDE`, `CODEX_ROLLOUT_TRACE_ROOT`, `CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT`, `CODEX_TUI_RECORD_SESSION` / `CODEX_TUI_SESSION_LOG_PATH`, `CODEX_NON_INTERACTIVE`, `CODEX_CA_CERTIFICATE`. Env Codex *injects* into spawned shell/hooks: `CODEX_THREAD_ID`, sandbox markers (`CODEX_SANDBOX*`, `CODEX_SANDBOX_NETWORK_DISABLED`), escalation-socket vars. `shell_environment_policy` config (`inherit` / `include_only` / `exclude` / `set`) governs which env vars reach model-executed commands. **Orchestrator instrumentation env** (hook port/token, pane/tab/worktree ids) is injected here too, consumed by the hook script (§8). *(established method)*
- **Initial-prompt injection mode:** Multiple. **(1) argv positional** — `codex "<prompt>"` (TUI) or `codex exec "<prompt>"`; POSIX-single-quote-escape (PowerShell/cmd quoting on Windows). **(2) stdin (exec)** — `codex exec -` or piped (`echo prompt | codex exec`); piped stdin + a positional prompt appends stdin as a `<stdin>` block. **(3) type-after-ready (live TUI)** — paste bytes into the PTY (bracketed paste for multi-line, then a separate CR to submit; see §6). **(4) app-server** — a `turn/start` / `sendUserTurn` JSON-RPC with message items. The dominant TUI-launch failure mode is the first prompt being swallowed by the trust gate — pre-seed it (next bullet). *(established method; mode chosen per transport)*
- **First-run / trust handling:** Trust is **per-project**, persisted in `config.toml` as `[projects."<abs path>"] trust_level = "trusted"`. Pre-seed by writing that table (use the *realpath*) **before** launch so the "Do you trust this folder?" onboarding never fires and can't eat the first prompt. When the orchestrator manages its own `CODEX_HOME`, write the trust entry into **both** `~/.codex/config.toml` and the managed `<CODEX_HOME>/config.toml`. Auth onboarding is skipped when `auth.json` already exists (§4). For exec outside a git repo, add `--skip-git-repo-check`. *(established method)*
- **Permission / YOLO / sandbox modes:** `-s, --sandbox <read-only|workspace-write|danger-full-access>`; `-a, --ask-for-approval <untrusted|on-request|never>` (also `on-failure`, deprecated); `--dangerously-bypass-approvals-and-sandbox` (aka `--yolo`) skips all confirmation + sandbox; `--add-dir <DIR>` widens writable roots; `--dangerously-bypass-hook-trust` runs hooks without persisted trust. Config equivalents: `sandbox_mode`, `approval_policy`, and the `sandbox_workspace_write` table (`network_access`, `writable_roots`, `exclude_tmpdir_env_var`, `exclude_slash_tmp`). Linux sandbox uses bubblewrap (`bwrap`); Windows uses a `[windows] sandbox=…` mode. For an interactive launch, prefer **not** to pass sandbox/approval flags (let Codex use its defaults + the per-project trust). Note `--dangerously-bypass-approvals-and-sandbox` changes *approval/sandbox policy*, not project *trust* — it is not a substitute for the trust preset above. *(established method)*
- **Model / reasoning-effort / fast-mode at launch:** **Model:** `-m/--model <MODEL>`, or `-c model="<slug>"`, or config `model=`. **Reasoning effort:** config `model_reasoning_effort = "low|medium|high|xhigh"` (also `plan_mode_reasoning_effort`), or override `-c model_reasoning_effort=high`; **no dedicated flag.** **Fast mode IS a concept:** the `fast_mode` feature is stable/true; the model catalog exposes `additional_speed_tiers:["fast"]` and `service_tiers:[{id:"priority", "1.5x speed"}]`; config key `service_tier`. Toggle via `-c service_tier=<value>` (exact value unverified — see Open questions) or the in-TUI model/speed picker. OSS/local models: `--oss --local-provider <lmstudio|ollama>`. *(established method)*
  - **Conflicts / open questions:** Whether reasoning effort can be passed as a CLI flag vs. only as `-c model_reasoning_effort=` — confirmed as `-c` override (no dedicated flag). Fast-mode toggling is confirmed to exist as a concept (`service_tier`); the exact accepted `service_tier` value (`"priority"` vs `"fast"`) is unverified.

## 3. Resume & reattach

- **Resume support / flags / id source:** Yes, first-class. **Interactive:** `codex resume [SESSION_ID|name] [PROMPT]`, `codex resume --last`, picker by default, `--all` to drop the cwd filter, `--include-non-interactive` to include exec sessions. **Headless:** `codex exec resume [SESSION_ID] [PROMPT]` / `--last`. **Fork:** `codex fork [SESSION_ID]` → a *new* thread with copy-on-write history. The id is the UUIDv7 from §1 (rollout filename / `session_meta.id`, or the live hook `session_id`). Resume is a **subcommand**, not a `--resume` flag. The app-server has explicit thread resume/fork RPCs. *(established method)*
  - Note: resume requires the rollout JSONL to be a real, readable file. If the orchestrator presents the same physical log under two roots (e.g. a managed `CODEX_HOME` and `~/.codex`), use a **hardlink, not a symlink** — Codex resume ignores symlinked JSONL sessions.
- **State restored vs lost on resume:** Restored by Codex from the rollout JSONL: full conversation history (incl. reasoning items via `encrypted_content`), `cwd`/`workspace_roots`, and recorded `turn_context` (`approval_policy`, `sandbox_policy`) — though model/approval/sandbox can be *overridden* on the resume command line (`-m`/`-s`/`-a`). **Lost:** the live PTY *screen* state (only logical history is in the rollout; the TUI re-renders from history on resume). The orchestrator re-establishes its own per-pane state and does not re-inject model/permissions unless it chooses to. *(established method)*
- **Reattach to a running PTY after restart:** **TUI path** — Codex provides *no* native PTY reattach; this is the orchestrator's job: wrap the PTY in a multiplexer (tmux/abduco) or a daemon-held PTY, reattach, and issue a **redraw nudge**. Codex is an alt-screen TUI, so a single SIGWINCH triggers a full repaint (it does `\x1b[H\x1b[2J`-style full repaints) — skip any delayed double-resize for alt-screen sessions, as one resize already forces a clean repaint without content loss. **Headless path** — Codex DOES survive natively: `codex app-server daemon` keeps threads alive, and `codex app-server proxy --sock <path>` (or `codex --remote ws://… / unix://PATH` with `--remote-auth-token-env <ENV>`) reconnects a fresh client to the running daemon's live threads after an orchestrator restart. *(established method TUI = generic mux + SIGWINCH; established method headless = daemon/proxy)*
- **Idempotency / race hazards:** (a) The newest `rollout-*.jsonl` at launch may be a **subagent/guardian** thread, not the user thread — disambiguate via `thread_source`/`source` before binding identity (§7). (b) Resume may create a *new* rollout file for the continued session in some builds — don't assume the resumed file's id equals the original (unverified for 0.141). (c) Only **one** PTY client should hold the multiplexer master; a double-attach causes echo-doubling. (d) Don't resume a session already live in another client. (e) When bridging one physical log under two roots, migrate symlink→hardlink only when the link still points at the expected source, and clean temp replacement files on failure. *(guard rules)*

## 4. Auth & subscription

- **Credential locations:** `<CODEX_HOME>/auth.json` (mode `0600`). Keys: `auth_mode` (`"chatgpt"` | `"apikey"`), `OPENAI_API_KEY` (null under ChatGPT), `tokens:{id_token, access_token, refresh_token, account_id}`, `last_refresh`. The `id_token` is a JWT whose claims include `email`, `exp`, and `https://api.openai.com/auth` → `{chatgpt_plan_type, chatgpt_account_id, organizations:[…]}` — read this (read-only) for account identity/plan. MCP-server OAuth creds are stored separately in `<CODEX_HOME>/.credentials.json` (per-server `{server_url, client_id, access_token, refresh_token, expires_at, scopes}`). A `secret_auth_storage` feature exists (currently off) for future OS-keychain storage; today there is **no OS keychain** use. *(established method)*
- **Reuse model:** Two supported paths. **spawn-and-let-CLI-auth:** `codex login` (browser OAuth, ChatGPT plan), `codex login --with-api-key` (key on stdin), `codex login --with-access-token` (stdin), `codex login --device-auth`, `codex login status` (prints e.g. `Logged in using ChatGPT`), `codex logout`. **orchestrator-managed multi-account:** point `CODEX_HOME` at a per-account directory, each holding its own `auth.json`/`config.toml`/`sessions/`; run `codex login` into that home, read identity from the `auth.json` JWT, and switch active account by repointing `CODEX_HOME` per spawn. Alternatively inject `OPENAI_API_KEY`/`CODEX_API_KEY`/`CODEX_ACCESS_TOKEN` directly. *(both established)*
- **Token refresh / write-back:** Codex auto-refreshes the ChatGPT OAuth access token against `https://auth.openai.com/oauth/token` (`grant_type=refresh_token`, embedded public `client_id`) using the stored `refresh_token`, writing back `tokens` + `last_refresh` into the same `CODEX_HOME`'s `auth.json`. The orchestrator does **not** refresh tokens itself; if it manages homes, it should read Codex's in-place refreshed `auth.json` back so the next launch reuses it. Test/override endpoints exist: `CODEX_REFRESH_TOKEN_URL_OVERRIDE`, `CODEX_REVOKE_TOKEN_URL_OVERRIDE`, `CODEX_AUTHAPI_BASE_URL`, `CODEX_APP_SERVER_LOGIN_CLIENT_ID`. *(established method; refresh owned by the CLI)*
- **Env hygiene:** To force the CLI's *own* stored creds, **strip inherited `OPENAI_API_KEY` / `CODEX_API_KEY` / `CODEX_ACCESS_TOKEN` / `OPENAI_ORGANIZATION`** from the spawn env — otherwise Codex prefers the API-key path (precedence `OPENAI_API_KEY` → `CODEX_API_KEY` → `CODEX_ACCESS_TOKEN`) over the stored ChatGPT login. Isolation between managed accounts is by `CODEX_HOME` redirection per child; never mutate the global process env. `shell_environment_policy` further governs what reaches model-executed commands. *(established method)*
- **Multi-account isolation / selection:** Per-`CODEX_HOME` isolation: one dir per account (its own `auth.json`/`config.toml`/`sessions/`); selection = set `CODEX_HOME` per spawn. There is **no in-CLI account switcher**. Recommended: mark each managed home with an ownership marker and confine it by realpath containment under an accounts root; mirror a canonical `config.toml` into each so switching swaps only credentials/quota identity, not config. Switching a *live* session requires a restart for the new account to take effect (surface a restart prompt over live panes). *(established method)*

## 5. Models, usage & accounting

- **Model listing (settings UI):** Authoritative source = `<CODEX_HOME>/models_cache.json`, fetched from OpenAI and ETag-cached `{fetched_at, etag, client_version, models:[…]}`. Each entry: `slug`, `display_name`, `description`, `default_reasoning_level`, `supported_reasoning_levels:[{effort,description}]` (low/medium/high/xhigh), `shell_type`, `visibility`, `supported_in_api`, `priority`, `additional_speed_tiers`, `service_tiers`, `availability_nux`. Freshness = ETag-revalidated on startup. Config key `model_catalog_json` allows a custom catalog. A secondary dynamic probe is `codex debug models` (JSON `{ models:[{ slug, display_name, supported_reasoning_levels:[{effort}], default_reasoning_level }] }`; map `xhigh`→"Extra High"). Keep a hard-coded fallback list for offline/old builds. *(established method = `models_cache.json`; fallback = `codex debug models` then static list)*
  - **Conflicts / open questions:** There is no top-level `codex models` subcommand in 0.141; `codex debug models` is the dynamic command and `models_cache.json` is the on-disk authoritative read. Prefer the cache file; use `debug models` if you need a forced refresh.
- **Runtime switching mid-session:** **TUI** — model + reasoning effort are switchable via the model picker and the keymap actions `increase_reasoning_effort` / `decrease_reasoning_effort`; slash `/model` exists. **app-server** — model/effort are set per turn in `turn/start` params. The orchestrator cannot force a mid-session model change on the *interactive TUI* other than via these in-TUI affordances or a resume relaunch. *(established for TUI keymap + app-server; not orchestrator-driven for the raw PTY)*
- **Usage windows / quota / rate-limits:** Several reads, all gated on `auth.json` existing. **(1) app-server RPC (preferred):** spawn `codex -s read-only -a untrusted app-server`, do the JSON-RPC `initialize`→`initialized` handshake, then `account/rateLimits/read`; response `{ rateLimits:{ primary, secondary } }` maps to a 5h "session" window (≈300 min) and a weekly window (≈10080 min), each `{ usedPercent, resetsAt (unix s), resetDescription }`; plus reset-credit availability. **(2) Backend HTTP:** `https://chatgpt.com/backend-api/wham/usage` and a reset-credits consume endpoint, called with `Authorization: Bearer <access_token>` + `ChatGPT-Account-Id: <account_id>` (token/id from `auth.json`). **(3) PTY fallback:** spawn `codex`, wait for the `›`/`>` prompt, send `/status\r`, regex-parse `5h limit: N%` / `weekly limit: N%` / `resets …`. Account-scoped via per-child `CODEX_HOME`. Per-turn rate-limit data is also delivered by the backend and surfaced in the TUI status line; `codex doctor --json` reports auth/runtime health. *(established method = app-server RPC; fallbacks = backend HTTP, `/status` PTY scrape)*
  - **Conflicts / open questions:** The exact remaining-quota field names from the per-turn/streaming path are unverified; the app-server `account/rateLimits/read` shape and the `/status`-scrape are the verified reads.
- **Token accounting:** Yes, rich. From the rollout JSONL `event_msg` records with `payload.type=="token_count"`: `info.total_token_usage` (cumulative) and `info.last_token_usage` (this turn), each `{input_tokens, cached_input_tokens (or cache_read_input_tokens), output_tokens, reasoning_output_tokens, total_tokens}`. `task_started` records `model_context_window`. Identical events stream over `exec --json` / app-server. **Quirks for correct accounting:** totals are *mutable snapshots* (compaction/resume rewrites them) — use `last_token_usage` as the billable delta with a monotonic/stale-regression guard; `cached_input_tokens` is aliased from `cache_read_input_tokens` and clamped ≤ input; `reasoning_output_tokens` is already inside `output_tokens`, so don't double-count it into the total. *(established method)*
- **Pricing / cost:** **Not exposed.** No per-model price field in `models_cache.json`; Codex computes no cost. The orchestrator must supply its own price table keyed by `slug`. Mark cost as inferred when the model slug can't be resolved. *(not supported — orchestrator-owned)*

## 6. Driving & input — two-way control

- **Send a new user message mid-session:** **TUI** = write bytes into the PTY then submit; multi-line content should use **bracketed paste** (`ESC[200~ … ESC[201~`) followed by a *separate* CR (`0x0d`) to submit; single-line can be text+CR. The composer keymap (`TuiGlobalKeymap`) names `submit`, `queue` (queue a message while the agent is busy), `insert_newline`. **Headless** = app-server `turn/start` / `sendUserTurn` JSON-RPC with message items (clean programmatic transport). *(established method)*
- **Read the agent's current input-box contents (draft sync out):** **Not supported natively** — no API/OSC/transcript exposure of the live composer. The only route is screen-scraping the TUI composer line, which is a **dim-placeholder `›` prompt** (read with dim-text handling). For app-server clients the draft lives in the orchestrator, so this is moot. *(not supported; screen-scrape only)*
- **Pre-fill / set the input box (draft sync in):** No flag/env to prefill the *interactive* composer beyond the initial positional `PROMPT`. For a live TUI, paste-into-the-PTY is the only path — gate the paste on bracketed-paste being enabled (DECSET 2004) **and** the `›` composer glyph being present (paste-ready signal), rather than a generic quiet window. For app-server the client owns the draft entirely. There is no native `--prefill` flag. *(fallback = typed/bracketed paste gated on composer-ready; not supported natively)*
- **Answer interactive prompts / permission menus:** **TUI** — approval dialogs are driven by `TuiApprovalKeymap` actions `approve`, `approve_for_session`, `approve_for_prefix`, `decline`, `open_fullscreen`, `open_thread`; selection is via the bound keys (typically arrow + Enter, or number keys). Read the option set by scraping the dialog (bindings are user-rebindable via `[tui] keymap`, so don't hardcode). **app-server** — approval surfaces as an Item of kind "approval request"; the client replies with an approval-decision RPC. **Hooks** — the `PermissionRequest` event fires with the proposed-action payload (§8); a design that lets the hook exit *without* a decision keeps Codex's normal approval UI visible while letting the orchestrator flip the pane to a waiting state. *(established method per transport)*
- **Interrupt / cancel / escape:** **TUI** keymap actions `interrupt_turn` (global), `interrupt_message`, `cancel`. Bytes: **Ctrl-C = `0x03`**, **Esc = `0x1b`** (Esc backs out of menus; Ctrl-C / double-Esc interrupts — exact per keymap). **app-server** has an interrupt/cancel RPC for the running turn. No Codex-specific interrupt byte beyond the standard terminal path. *(established method)*
- **Slash commands:** TUI supports slash commands (`/model`, `/approvals`, `/init`, `/review`, `/compact`, `/mention`, `/status`, etc.; `mentions_v2` feature stable). The orchestrator may issue `/status\r` for the rate-limit fallback read (§5). `codex review` is also a top-level subcommand. *(established for `/status`; broader set exists, exact list unverified)*
- **Attachments (files/images):** `-i, --image <FILE>…` (repeatable / comma-separated) attaches images to the initial prompt (exec + interactive + resume + fork). In-session, `@`-mentions reference files (`mentions_v2`). Rollout `user_message` payload carries `images`, `local_images`, `text_elements` arrays. Large paste: `disable_paste_burst` config controls burst-paste handling; for very large inline drafts on Windows, fall back to post-ready paste past an argv length ceiling. *(established method)*
- **Important keyboard shortcuts / modifiers (mobile soft-keys):** The full keymap is config-driven and rebindable (`[tui] keymap`) — read it, don't hardcode. Notable actions: `TuiGlobalKeymap`{ open_transcript, open_external_editor, clear_terminal, **submit**, **queue**, toggle_vim_mode, toggle_raw_output, **interrupt_turn** }; `TuiChatKeymap`{ **interrupt_turn**, **decrease_reasoning_effort**, **increase_reasoning_effort**, edit_queued_message }; `TuiComposerKeymap`{ history_search_previous/next }; `TuiEditorKeymap`{ **insert_newline**, move_word_left/right, kill_line }; `TuiListKeymap`{ move_up/down, page_up/down, jump_bottom, **accept**, **cancel** }; `TuiApprovalKeymap`{ approve, approve_for_session, approve_for_prefix, decline }; plus full Vim keymaps. **Surface on mobile:** Enter (submit, `0x0d`), Shift/Alt+Enter (newline), Esc (`0x1b`), Ctrl-C (`0x03`), Up/Down (history + menu nav, `ESC[A`/`ESC[B`), Tab, `/` (slash), `@` (mention). **Alt/Option fidelity matters** for word-motion and some menu nav. `CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT` env disables the Kitty keyboard-enhancement protocol if a terminal mangles modifiers. On Windows PowerShell, CSI-u Shift+Enter is treated as inert. *(established method)*

## 7. Agent-state classification

- **State vocabulary:** Map to `working | waiting (approval/input) | done/idle | error` (+ an `interrupted` flag on done). Codex's own turn lifecycle is `task_started → … → task_complete/turn-end` with an explicit interrupt action.
- **Signal sources (authority rank):**
  1. **Hooks (highest, structured)** — `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop` fire as commands with a JSON payload on stdin (§8). Cleanest authoritative source for tool-start/stop/permission/done.
  2. **app-server / `exec --json` events** (equally authoritative for the headless path) — `item/started`, `item/completed`, deltas, turn start/complete, approval-request items.
  3. **Rollout JSONL tail** — `event_msg.payload.type` ∈ {`task_started`, `user_message`, `token_count`, …} and `response_item.payload.type` ∈ {`message`(role developer/user/assistant), `reasoning`, `function_call`, `function_call_output`}. A `function_call` present with no matching `function_call_output` yet ⇒ tool-running/working.
  4. **OSC title** — **none observed** (no JSON status channel; the terminal title is config `[tui] terminal_title` / `status_line`). A generic OSC-9999 JSON status processor can be wired but Codex does not emit it, so it is effectively unused for Codex.
  5. **Process / output activity (lowest)** — PTY byte activity ⇒ working; quiescence ⇒ idle; process exit ⇒ done/crash. Also useful for account-switch restart detection via foreground-process presence, and OSC 133 C/D command-lifecycle marks (emit from a shell wrapper to clear a stuck spinner when the foreground command exits without a Stop hook).
- **Event→state mapping:** `SessionStart`→ready/idle; `UserPromptSubmit` / `task_started`→working; `PreToolUse` / `function_call`→working (tool); `PermissionRequest` (or approval-request item)→**waiting-on-user**; `PostToolUse` / `function_call_output`→working; `Stop` / turn-complete→done/idle; non-zero exit→error. Anything else → ignore.
- **"Stopped but needs the user" detection:** The dedicated signal is the **`PermissionRequest`** hook / app-server **approval-request item** — explicitly distinct from `Stop`→done. (`exec_permission_approvals`, `guardian_approval`, `tool_call_mcp_elicitation` features show approvals/elicitations can also originate from MCP/guardian flows.) *(established method)*
- **New-turn reset:** Treat `SessionStart` or `UserPromptSubmit` as a new-turn boundary; reset per-pane prompt + tool-snapshot caches there.
- **Tool preview extraction:** Read `tool_name`/`name` and input from `tool_input`/`input`/`arguments` on `PreToolUse|PermissionRequest|PostToolUse`, and `last_assistant_message` on `Stop`. Map Codex-ish tools (`apply_patch`, `view_image`, `exec_command`, `shell_command`) to human previews / path previews.
- **Reconciliation / dedup / stickiness:** Prefer hooks/app-server events over scrape; dedupe on `turn_id`/`call_id` (both present in rollout). Quiet-window debounce on output-activity to avoid flapping; keep `waiting` sticky until an approval reply is sent. Install the status hook *first* in each event group so a slow user `Stop` hook can't leave the sidebar stuck. Warn-once on env/version mismatch.
- **Sub-agent / nested-task state:** Subagents (e.g. `guardian`) get their **own** rollout file with `thread_source:"subagent"`, `source:{subagent:{other:"guardian"}}`, and `parent_thread_id` linking to the parent. `multi_agent` feature is stable; `max_threads` / `max_depth` / `job_max_runtime_seconds` config bound fan-out. Inherit identity via `parent_thread_id`; classify subagent activity separately or roll up into the parent's "working."
- **Latency:** Hooks fire synchronously at each lifecycle point with a `timeout` (≈5s default in the installed config) — sub-second in practice; keep the callback fast (e.g. curl `--connect-timeout 0.5 --max-time 1.5`). `--json`/app-server events are real-time. Rollout-tail latency = fsync cadence (near-real-time). Output-activity = immediate but noisy.

## 8. Hooks & instrumentation install

- **Mechanism:** Yes — the `hooks` feature is **stable/true**: managed **shell-command hooks** (like Claude Code's), invoked as commands that receive a JSON payload on stdin. There is also a plugin system (`codex plugin add/list/marketplace/remove`, `<CODEX_HOME>/plugins`) and skills (`<CODEX_HOME>/skills`), but the hook mechanism is the instrumentation surface. *(established method)*
- **Install location + trust:** `<CODEX_HOME>/hooks.json`, structure `{"hooks":{"<Event>":[{"hooks":[{"type":"command","command":"…","timeout":N}]}]}}`. Hooks may also live in `[hooks]` tables in `config.toml`; scripts can live under `<CODEX_HOME>/hooks/`. **Trust is enforced via content hashing:** `config.toml` carries `[hooks.state."<hooks.json path>:<event_snake>:<i>:<j>"]` entries with `trusted_hash = "sha256:…"` and `enabled = true`; a hook only runs once its source hash is trusted (first run otherwise prompts). Example key: `hooks.json:pre_tool_use:0:0`. Install = write `hooks.json` + matching `[hooks.state]` trust entries (compute the sha256 over the hook source, keyed on the snake_case event label + group/handler indices + command). `--dangerously-bypass-hook-trust` runs untrusted hooks for automation; `requirements.toml` `allow_managed_hooks_only = true` is an admin lockdown. When managing your own home, mirror system user hooks into the managed home (strip plugin-placeholder commands) preserving trust state, and normalize the deprecated `codex_hooks` flag to `hooks`. *(established method)*
- **Event taxonomy + payload:** Events (JSON/PascalCase) `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop` (config-state mirrors snake_case `session_start`, `user_prompt_submit`, `pre_tool_use`, `post_tool_use`, `permission_request`, `stop`; extra known labels `pre_compact`/`post_compact`). Payload (JSON on stdin) fields worth extracting: `session_id` (also via `CODEX_THREAD_ID` env), `prompt`/`user_prompt`, `tool_name`/`name`, `tool_input`/`input`/`arguments`, tool result/exit, `cwd`, and `last_assistant_message` (on `Stop`). *(established method)*
- **Transport back + fail-open:** The hook command can do anything: an **HTTP callback** (inject port/token via env, e.g. POST `application/x-www-form-urlencoded` to `http://127.0.0.1:${PORT}/hook/codex` with an auth header carrying `paneKey,tabId,worktreeId,env,version,payload`), or a **file/NDJSON** transport (write `{event, ts, payload}` keyed by a per-session env id, e.g. into a status dir). **Fail-open** always: missing port/token/pane key → `exit 0`; curl errors `|| true`; give every hook a `timeout` so a slow/failed hook can't wedge the turn. Re-source the endpoint each invocation so a PTY that outlives an orchestrator restart reaches the new server. *(established method)*
- **Lifecycle / idempotency / version-skew:** Idempotent if you write a fixed `hooks.json` + matching `trusted_hash`. Update/uninstall = rewrite the file (the hash changes ⇒ re-trust required — this *is* the version-skew guard). Sweep stale managed entries by a broad matcher (script filename, not exact command) to clean prior builds/paths; remove events no longer subscribed; clean legacy representations (old `~/.codex/hooks.json` entries, legacy profile blocks). For remote installs, write `~/.codex/hooks.json` + the POSIX hook script + the trust TOML over SFTP (§15). *(established method)*

## 9. Session titles

- **Source of truth:** A **thread name** exists — `session_index.jsonl` row `thread_name` (e.g. `"user scope models"`), keyed by id; agent/first-prompt-derived. The TUI also sets the terminal window title via `[tui] terminal_title` config. There is **no OSC-title JSON channel**, and Codex does not reliably emit a live OSC title — so foreground-process/synthetic detection is the fallback identity source. *(established method = `thread_name`)*
- **How/when read & updated:** Read `thread_name` from `session_index.jsonl` by id, or derive from the first `user_message` in the rollout. `codex archive`/`unarchive`/`delete` accept a session **name** as well as id, confirming names are stable handles. Event by re-reading / fs-watching the index. *(established method)*
- **Fallback / synthetic:** When no `thread_name`, synthesize from the first user prompt (truncate `history.jsonl`/rollout `user_message.text`), or LLM-generate a title via headless `codex exec` (§12). A useful synthetic profile is `{ workingLabel:"Codex", permissionLabel:"Codex - action required", idleLabel:"Codex ready" }`, synthesizing only the *terminal* (idle/permission) states and leaving any native spinner title intact. *(fallback)*

## 10. Transcript & history

- **Storage:** One **JSONL "rollout" file per session** at `<CODEX_HOME>/sessions/YYYY/MM/DD/rollout-<ts>-<uuid>.jsonl`, plus a SQLite thread store (`state_5.sqlite`; `local_thread_store_compression` is under-development). Also global `history.jsonl` (input history) and `session_index.jsonl` (index). `--ephemeral` (exec) suppresses persistence entirely. *(established method)*
- **Schema mapping:** Each line `{timestamp, type, payload}`. `type` ∈:
  - `session_meta` — `payload.{id, parent_thread_id, cwd, originator, cli_version, source, thread_source, model_provider, base_instructions, git, multi_agent_version}`.
  - `turn_context` — `payload.{turn_id, cwd, workspace_roots, current_date, timezone, approval_policy, sandbox_policy, permission_profile, model}`.
  - `event_msg` — `payload.type` ∈ {`task_started` (carries `model_context_window`), `user_message` (`message`), `agent_message` (`message`), `token_count` (`info.total_token_usage` / `info.last_token_usage`), …}.
  - `response_item` — `payload.type` ∈ {`message`{role developer/user/assistant, content[{type:input_text/output_text, text}]}, `reasoning`{summary, encrypted_content}, `function_call`{name, arguments, call_id}, `function_call_output`{call_id, output}}.
  - Roles seen: `developer` (injected system/permission instructions), `user`, `assistant`.
- **Live tail vs historical read:** Append-only JSONL ⇒ tail by polling file size/inode (or fs-watch the day-dir / specific file) and parse new lines; order by line order; dedupe on `turn_id`/`call_id`. Historical = full-file streaming read (e.g. `readline`), reusing prior processed results when mtime/size are unchanged. The clean live alternative is consuming `exec --json` / app-server streaming events. Live *state* during a session is best from hooks (§7), not transcript tailing. *(established method)*
- **Consumers:** chat-view → `response_item.message` (output_text) + `event_msg.user_message`; last-message/preview → last assistant `message` (or `last_assistant_message` from the `Stop` hook); clickable items → `function_call.arguments` (commands/paths) + diffs from apply-patch tool outputs; usage analytics → `token_count` aggregates per session/day/model/cwd; resume id → `session_meta.id`; state classification → §7.
- **Transcript ↔ terminal reconciliation:** Two parallel views — the live PTY/xterm and the structured JSONL. The TUI renders the same logical items; reconcile by `turn_id`. The TUI `open_transcript` action dumps the transcript. When presenting one physical log under two roots, use the hardlink bridge (§3) so the orchestrator scan and Codex's own resume see the same file.
- **Special content types / quirks:** **Reasoning** is stored as `reasoning` items with `summary` (often empty) + **`encrypted_content`** (opaque base64 — NOT human-readable; do not render; `hide_agent_reasoning`/`show_raw_agent_reasoning` config gate display). **Images** → `user_message.images` / `local_images` (paths) + `-i` attachments + an `input_image` content type for inline. **Tool output** → `function_call_output.output` (plain text incl. sandbox errors, e.g. bwrap loopback failures — these are *tool* failures, not agent crashes). **Diffs** → apply-patch tool calls; `codex apply <TASK_ID>` re-applies the latest diff as `git apply`. **Token totals** are mutable snapshots (compaction/resume rewrites them) — see §5 accounting quirks.

## 11. Terminal rendering & display fidelity

- **PTY → xterm:** Codex TUI is a full-screen **ratatui alt-screen** app. Capture PTY bytes → stream → xterm.js. Handle alt-screen chrome as needed; intercept any orchestrator-injected status OSC and OSC 133 lifecycle marks. `--no-alt-screen` (general TUI mode; also on resume) runs **inline**, preserving scrollback — useful for a chat-style mirror. `CODEX_TUI_ROUNDED`, `[tui] theme`, `status_line`/`status_line_use_colors`, `animations` config affect rendering. *(established method)*
- **Resize/reflow / redraw nudge:** Treat Codex as an alt-screen TUI: a single SIGWINCH on cols/rows change triggers a full repaint (`\x1b[H\x1b[2J`-style), so **skip the delayed double-resize** on attach — one resize already forces a clean repaint without content loss. Config `terminal_resize_reflow_max_rows` exists, but `terminal_resize_reflow` is `removed/false` (reflow handling changed across versions — verify per build). On a mobile↔desktop breakpoint remount, re-send size + nudge. *(established method)*
- **Scrollback model:** Native TUI **alt-screen** — Codex owns its history rendering and re-renders on resize; the orchestrator does *not* own xterm scrollback for Codex (the pager keymap `TuiPagerKeymap`: scroll_up/down, half_page_up/down drives Codex's own history). Use `--no-alt-screen` if you'd rather own scrollback in xterm. *(established method)*
- **Mouse forwarding:** Generic — when Codex enables mouse tracking (alt-screen), forward SGR mouse sequences (scroll/click) into the TUI; enable while attached, disable when showing orchestrator chrome. No Codex-specific toggle. *(falls back to generic)*
- **Snapshot/serialize for reattach:** **No native screen snapshot** in the rollout (only logical history). The orchestrator must headless-xterm + serialize the PTY buffer itself, then replay on reattach. Validate the serializer against Codex's alt-screen + cursor-relative (Ink) + ratatui inline-viewport (`╭─ Codex ──╮`) repaint shapes and de-duplicate repeated repaints. `CODEX_TUI_RECORD_SESSION` / `CODEX_TUI_SESSION_LOG_PATH` can record the TUI session for debugging. *(established orchestrator-side method)*
- **Local cache for instant render then live reconnect:** Not provided by Codex. Pattern: persist the last serialized xterm buffer, paint it instantly on return, then reattach the live PTY (mux) and issue a redraw nudge to reconcile. For headless, `app-server proxy`/`--remote` reconnect gives live thread state with no cached buffer needed. *(orchestrator-owned)*
- **Composer/cursor specifics:** Composer is a **dim-placeholder `›` prompt** line. The Codex-specific paste hook: wait for DECSET 2004 (bracketed-paste enabled) **then** for the `›` glyph as the **paste-ready** signal before pasting a draft (instead of a generic quiet window). Read the composer with dim-text handling when scraping. *(established method)*
- **Color/theme/OSC:** `NO_COLOR`, `--color always|never|auto` (exec), `[tui] theme`, `status_line_use_colors`. No custom OSC status protocol observed; standard terminal sequences only. If managing a `CODEX_HOME`, symlink `themes` from `~/.codex` so the user keeps their theme. *(no special handling)*

## 12. Background / headless invocation

- **Non-interactive invocation:** Yes — primary headless surface is `codex exec [OPTIONS] [PROMPT]` (alias `codex e`). Prompt via argv, or `-`/piped stdin (piped + argv ⇒ stdin appended as `<stdin>`). Output: human text by default; **`--json`** for JSONL events; **`-o/--output-last-message <FILE>`** to capture just the final answer; **`--output-schema <FILE>`** to constrain the final response to a JSON Schema (structured extraction). `--ephemeral` avoids writing session files; `--skip-git-repo-check` for non-repo dirs; `-s read-only` and `-a untrusted`/`never` for safe one-shots; `-c model_reasoning_effort=<level>` to tune. `codex exec resume`/`--last` continues a thread headlessly. Also `codex mcp-server` (Codex *as* an MCP server) and `codex app-server` (JSON-RPC) for richer control, and `codex debug models` for model discovery. *(established method)*
- **Features that use it:** commit messages, PR title/body, branch names, summaries, **session titles (§9)**, structured extraction (`--output-schema`), model discovery (read `models_cache.json` or `codex debug models`), and `codex login` for managed accounts. *(established method)*
- **Auth path for headless:** Inherits the same `<CODEX_HOME>/auth.json` (ChatGPT plan or API key); set `CODEX_HOME`/`OPENAI_API_KEY`/`CODEX_ACCESS_TOKEN` per child, never mutating global env. Gate on `auth.json` existing so you never spawn for an un-signed-in user. **ToS note:** reusing a ChatGPT-plan login for orchestrator background work is the gray area; an API key avoids it. *(established method)*
- **Cost/latency/timeout/output caps:** Config bounds: `tool_output_token_limit`, `model_auto_compact_token_limit`, `background_terminal_max_timeout`, `job_max_runtime_seconds`. Reasoning effort dominates latency — pick `low`/fast tier for cheap one-shots. Practical orchestrator timeouts seen in deployment: app-server RPC ≈10s (cold/remote ≈25s); `/status` PTY scrape ≈15s with a ~500ms post-parse settle; `codex login` ≈120s; cap diagnostic output (e.g. 100k chars). Runs locally, via `--oss` local model, or via `--remote`/`app-server` daemon. *(established method)*

**Known special-cases / workarounds:**
1. Packaged binary reports as `codex-<arch>` — recognize by the `codex-` prefix in process tables.
2. Reasoning items are encrypted (`encrypted_content`) — opaque, never render.
3. The newest rollout at launch may be a **subagent** thread — filter by `thread_source`/`source` before binding identity.
4. `logs_2.sqlite` telemetry DB can bloat to multiple GB and **block startup** (Codex checkpoints it before drawing the TUI; observed empty/hung panes are *this*, not orchestrator code). Fix: delete `logs_2.sqlite*`; Codex rebuilds a small one and starts in ~200ms. Recurs as the DB regrows.
5. Composer is a dim placeholder `›` — read with dim-text handling; use it as the paste-ready signal.
6. Hooks need sha256 trust entries (or `--dangerously-bypass-hook-trust`); modern builds silently drop untrusted hooks.
7. `-c key=value` parses the value as TOML, falling back to a literal string — quote carefully.
8. `--strict-config` errors on unknown keys (a version-skew guard).
9. Config has ~96 top-level fields and fully rebindable keymaps — read `[tui] keymap`, don't hardcode key bytes.
10. `CODEX_HOME` re-export in any shell rc wrapper so a user profile can't override the orchestrator's managed home.
11. Bridge a shared physical rollout with a **hardlink, not a symlink** (resume ignores symlinked JSONL).
12. Avoid `--profile-v2` for an interactive launch (loads a second hook representation + warns).
13. On Windows, route a `.cmd` Codex through `cmd.exe /c` (avoid DEP0190 / batch-spawn issues).
14. Windows PowerShell treats CSI-u Shift+Enter as inert.

## 14. Failure, exit & recovery

- **Crash/exit detection:** `codex exec` returns standard exit codes (0 = success). Distinguish clean turn-end (`Stop` hook / turn-complete event, exit 0) from a crash (non-zero exit, or PTY EOF without a `Stop`). Sandbox/tool failures surface *inside* `function_call_output.output` (e.g. `Process exited with code 1`, bwrap errors) — these are *tool* failures, not agent crashes. Classify launch errors: ENOENT on a bare `codex` → "CLI not found"; ENOENT on a resolved path → "found but couldn't run — Node.js may not be in PATH"; auth errors → status `unavailable` vs `error`; not-signed-in → `unavailable`. `codex doctor --json` diagnoses install/auth/runtime faults. *(established method)*
- **Reattach-after-restart healing:** **TUI path** — heal exited-but-alive multiplexer rows: a redeploy that SIGKILLs the *attach client* must not be recorded as an agent exit while the master lives; on boot, reattach surviving masters. The hook script's endpoint re-source keeps a surviving PTY reporting to the new server. **Headless path** — `app-server daemon` survives the orchestrator; reconnect via `proxy`/`--remote`. Reap orphan subagent threads via `parent_thread_id`. Rollout housekeeping config: `max_rollout_age_days`, `max_rollouts_per_startup`, `min_rollout_idle_hours`. *(established method)*
- **Error surfacing:** Non-zero exit, PTY EOF, `function_call_output` error text, a missing `Stop`/turn-complete, or `codex doctor` failures. Surface a rate-limit/auth panel with `error` vs `unavailable` messages. On account switch, surface a restart prompt over live Codex panes (the new account only takes effect after restart). `interrupt_message` config customizes the interrupt notice. *(established method)*

## 15. Remote / transport

- **Supported transports:** (a) **Local** — PTY+TUI, or local `exec`/`app-server`. (b) **SSH / remote worktree** — `codex app-server daemon bootstrap` explicitly installs durable local app-server management for SSH-driven use; for the TUI path, forward the PTY byte stream + SIGWINCH and run hooks on the remote host. (c) **Daemon-subprocess** — `codex app-server daemon` + `codex app-server proxy --sock <path>` (stdio↔Unix socket). (d) **Network remote** — `codex --remote ws://|wss://|unix://[PATH]` with `--remote-auth-token-env <ENV>` connects a TUI/fork/resume client to a remote app-server over WebSocket/Unix, bearer-token authed; `codex app-server --listen` accepts `stdio://` / `unix://` / `ws://IP:PORT` / `off`; `codex remote-control start/stop` manages a remote-control-enabled daemon. WSL is a first-class remote-ish runtime (managed homes, `codex login`, rate-limit RPC all work via `wsl.exe -d <distro> -- bash -lc 'export CODEX_HOME=…; exec codex …'`). *(established method)*
- **Forwarding PTY/git/fs + instrumentation over the relay:** **Headless path** — no PTY needed: JSON-RPC items/events carry everything (messages, tool calls, approvals, diffs); the app-server runs git/fs locally on the remote host and the relay just forwards the JSON-RPC stream + bearer token. **TUI path over SSH** — forward the PTY bytes + SIGWINCH, and have remote hooks write to a remote file or HTTP callback the relay reads (key per-session by an injected env id); normalize the hook callbacks on the relay and route to the orchestrator with the connection stamped. Wait for the remote shell-ready marker before sending the `codex …`/`codex resume …` command (with a fallback timer). *(established method)*
- **Agent-specific remote limitations:** Remote hook scripts must use **POSIX `.sh`** paths even when the orchestrator runs on Windows. Host↔WSL `CODEX_HOME` compatibility must be gated so a Windows-path home isn't passed into a WSL Codex (and vice-versa). Encrypted reasoning + the alt-screen TUI mean *chat* fidelity is better via app-server / rollout-JSONL than via raw screen-scrape on a remote PTY. *(established method + constraint)*
