# Claude Code — orchestrator integration reference

Target CLI: **Claude Code** (`claude`), verified against version **2.1.183** (`claude --version` → `2.1.183 (Claude Code)`). Facts are grounded in the installed binary's `--help`/subcommand help, on-disk state under `~/.claude`, and official docs at `code.claude.com/docs` (formerly `docs.claude.com/en/docs/claude-code`, 301-redirected). Each answer enumerates all viable methods and tags them **established** / **fallback** / **not supported**. Discrepancies between sources are called out under "Conflicts / open questions" within a section.

Throughout, `~/.claude` is the default config home and is relocatable via the `CLAUDE_CONFIG_DIR` env var (the basis of per-account isolation, §4).

---

## 1. Discovery & identity

- **Binary detection (established):** command name is `claude`, stable, no documented rename/alias. "Installed?" = `claude` resolvable on PATH. On this machine it lives at `~/.local/bin/claude` (a launcher shim); the actual versioned build is under `~/.local/share/claude/versions/<version>/` (referenced by `daemon.lock.launchTarget`). A bare PATH probe of `claude` is the established install check; `claude doctor` additionally checks auto-updater health.
- **Version detection (established):** `claude --version` (or `-v`) prints `<semver> (Claude Code)`. Behavior **does** vary by version — the hook event set, and flags like `--effort`, `--remote-control`, `--teleport`, `--from-pr`, `--json-schema` have grown across releases; docs gate specific features by version (e.g. background sessions in the `--resume` picker "as of v2.1.144"). Parse the leading semver and feature-gate. No behavior should be assumed to branch on version inside the CLI; gate on the *orchestrator* side.
- **Existing-session discovery — transcripts (established):** JSONL, one file per session, at `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl`. Glob: `~/.claude/projects/*/*.jsonl`. `<encoded-cwd>` is the absolute cwd with every `/` replaced by `-` (leading slash → leading `-`), e.g. `/home/user/src/other/podium` → `-home-user-src-other-podium` (verified on disk). A sibling per-session directory `<sessionId>/` holds `tool-results/` overflow files and file backups. Take files ending `.jsonl`; recurse the project dirs.
- **Existing-session discovery — live process registry (established, agent-native):** `~/.claude/sessions/<pid>.json` — **one file per *running* process** (not per transcript). Verified field set on this machine: `pid, sessionId, cwd, startedAt, procStart, version, peerProtocol, kind ("interactive"|"bg"), entrypoint, status ("idle"|"busy"|"waiting"), statusUpdatedAt, updatedAt, bridgeSessionId` (docs/usage additionally describe `waitingFor, name, jobId`). This is the authoritative "what is alive right now + its cwd + sessionId + live status" map; `pid` + `procStart` verify liveness.
- **Other identity stores (established):** `~/.claude/history.jsonl` (prompt history; fields `display, pastedContents, timestamp, project (=cwd), sessionId`); `~/.claude/session-env/<sessionId>/` (per-session env, usually empty); `~/.claude/file-history/`, `tasks/`, `plans/`, `jobs/`, `paste-cache/`.
- **Stable session identity (established):** a UUID `sessionId`. At launch you can **set it yourself** with `--session-id <uuid>` (must be a valid UUID) — the cleanest contract for an orchestrator. Otherwise it is auto-generated and discoverable from `sessions/<pid>.json` (live) or the newest transcript filename in the project dir. A human-friendly `-n/--name` alias also works as a resume key. In `-p` JSON output the `session_id` is returned in the result envelope. Hook payloads carry `session_id` (§8) — the live capture path.
- **Session ↔ repo/worktree/cwd (established):** the transcript dir encodes cwd, but decoding is **ambiguous if a real path contains `-`**; prefer reading `cwd` directly from inside the JSONL records or from `sessions/<pid>.json`. Every transcript record carries `cwd` and `gitBranch`. Worktrees created via `claude -w/--worktree <name>` land under `<repo>/.claude/worktrees/<name>`.

**Conflicts / open questions:** session id can be captured two ways — from the structured `sessions/<pid>.json` (live) or from hook payload `session_id`. Both are valid; the process registry is preferred for discovery, hooks for live event correlation. Decoding the project-dir name back to a cwd is lossy — always cross-check against an in-record `cwd`.

## 2. Launch & process model

- **Spawn mechanism (established):** `claude` is a TUI (Ink/React terminal app). The two orchestrator-friendly paths are **direct exec inside a PTY** (interactive) or `--print`/`-p` (headless, §12). For interactive multi-agent use, host the PTY in a multiplexer (tmux/abduco) or your own daemon-held PTY master for durability — the CLI provides no detach for interactive mode (but see durable backing below).
- **Launch command / args (established):** `claude [options] [prompt]`. Useful launch args (all verified in `--help`): `--model`, `--effort`, `--permission-mode`, `--session-id <uuid>`, `-n/--name`, `--add-dir <dirs...>`, `--agent`/`--agents <json>`, `--append-system-prompt`/`--system-prompt`, `--settings <file-or-json>`, `--setting-sources`, `--mcp-config <configs...>`, `--strict-mcp-config`, `-w/--worktree [name]`. YOLO: `--dangerously-skip-permissions`.
- **Env it reads (established):** `ANTHROPIC_API_KEY`, `ANTHROPIC_AUTH_TOKEN`, `ANTHROPIC_MODEL`, `CLAUDE_CODE_OAUTH_TOKEN`, `AWS_BEARER_TOKEN_BEDROCK`, the `CLAUDE_CODE_*` family, `MAX_THINKING_TOKENS`, `CLAUDE_EFFORT`, `CLAUDE_CONFIG_DIR` (relocates `~/.claude`), `CLAUDE_CODE_REMOTE` (set "true" in web envs), `OTEL_*`/`CLAUDE_CODE_ENABLE_TELEMETRY` (telemetry, §5). Env injected *to hooks/tools*: `CLAUDE_PROJECT_DIR`, `CLAUDE_PLUGIN_ROOT`, `CLAUDE_ENV_FILE`, `CLAUDE_EFFORT`.
- **Env hygiene at launch (established):** to force the CLI's own subscription OAuth, **strip the auth env vars** from the child before spawning: `ANTHROPIC_API_KEY`, `ANTHROPIC_AUTH_TOKEN`, `CLAUDE_CODE_OAUTH_TOKEN`, `AWS_BEARER_TOKEN_BEDROCK` (and auth-like `ANTHROPIC_CUSTOM_HEADERS`). Otherwise an inherited key wins and bills the API. For managed multi-account, also set `CLAUDE_CONFIG_DIR` to the account's config home (§4). `--bare` strictly limits Anthropic auth to `ANTHROPIC_API_KEY`/`apiKeyHelper` (never OAuth/keychain).
- **Durable backing (established + fallback):** *Fallback:* no built-in PTY detach for interactive mode → use tmux/abduco, or hold the node-pty master in an out-of-process daemon with a server-side history log + headless-xterm snapshot for redraw (the generic durable path; §11). *Established agent-native:* Claude Code ships a **background-agent daemon** — `~/.claude/daemon.{lock,log,status.json}` + `claude agents` (agent view) dispatch/monitor parallel **background** sessions that survive independently; `claude agents --json [--all]` lists them for scripting, `claude rm <id>` removes one. `--remote-control [name]` starts an interactive session reachable via Remote Control; `--teleport` resumes a web session locally. These are agent-specific durability options beyond a generic multiplexer.
- **Initial-prompt injection (established):** three native modes — (a) **argv positional**: `claude "do X"` (interactive; prompt consumed at startup, so no bracketed-paste/CR timing needed); (b) **headless flag**: `claude -p "do X"`; (c) **stdin**: `echo "do X" | claude -p`, or `--input-format stream-json` for realtime streamed input in `-p`. argv/`-p` are the robust programmatic paths — a real argv, so no quoting gymnastics beyond normal shell quoting (single-quote escaping `'\''` on POSIX). Type-after-ready into the PTY is the *follow-up* path (§6), not the initial one.
- **First-run / trust handling (established):** interactive launch shows a workspace-trust dialog; it is **skipped automatically in non-interactive mode** (`-p`, or when stdout is not a TTY). For interactive orchestration, the documented escape is to run non-interactively for one-shots, or to pre-trust the dir / use `--permission-mode`. `--dangerously-skip-permissions` and (added to the Shift+Tab cycle without starting in it) `--allow-dangerously-skip-permissions` avoid per-tool prompts. `--safe-mode`/`--bare` give clean boots that skip customizations.
- **Permission / YOLO / sandbox (established):** `--permission-mode <mode>` ∈ `default, acceptEdits, plan, auto, dontAsk, bypassPermissions`. `--dangerously-skip-permissions` ≡ `bypassPermissions` (the YOLO flag). `--tools`/`--allowedTools`/`--disallowedTools` constrain the tool set. Default mode is read from `settings.json` `permissions.defaultMode` (observed default `auto`). For headless chores, `--permission-mode plan` is a safe choice that prevents tool side effects.
- **Model / effort / fast-mode at launch (established):** `--model <alias|full>` — aliases `opus`, `sonnet`, `haiku`, `fable`, or full ids like `claude-sonnet-4-6` / `claude-opus-4-8` (overrides `ANTHROPIC_MODEL`). Prefer **aliases** over hardcoded version ids for portability (full version ids can be rejected by Bedrock/Vertex). `--effort <low|medium|high|xhigh|max>` (also env `CLAUDE_EFFORT`, setting `effortLevel`). `--fallback-model <model>` for overload failover (print mode). **No separate named "fast mode"** — the closest levers are a lower `--effort`, a faster model (`haiku`), or bounding `MAX_THINKING_TOKENS`. `--betas` adds beta headers (API-key users only).

**Conflicts / open questions:** the precise on-disk workspace-trust marker file/key is unverified; the reliably-documented escape is non-interactive mode + `--dangerously-skip-permissions`. There is no agent-specific default *env* required for a plain interactive launch (instrumentation env in §8 is orchestrator-injected, not CLI-required).

## 3. Resume & reattach

- **Resume support (established):** `-r/--resume [id|name]` (resume by sessionId UUID or by `--name`; bare opens a picker). `-c/--continue` (most recent conversation in cwd). `--fork-session` (resume but mint a **new** sessionId, leaving the original transcript untouched — **preferred when seeding history into a fresh id**). `--from-pr [value]` resumes the session linked to a PR (by number or URL). `--teleport` resumes a Claude-Code-on-the-web session into a local terminal. The id comes from `sessions/<pid>.json`, the transcript filename, or your own `--session-id` from first launch. Resume rebuilds the shell command as `claude --resume <quoted-id>`.
- **Id sanitization (established, hazard guard):** before passing a captured id to `--resume`, validate it as a UUID and reject leading `-` and control chars to avoid argv injection.
- **Restored vs lost on resume (established):** restored = full conversation history (re-read from the JSONL transcript) + the resume cwd. **Lost unless re-passed:** model and permission-mode default to the *current* flags/settings, so re-pass `--model`/`--permission-mode` to pin them. Resume only finds sessions in the current project dir + its git worktrees (or those that `/add-dir`-ed it). `--fork-session` preserves history under a new id.
- **Reattach to a still-running PTY after orchestrator/daemon restart (fallback / generic):** Claude Code has **no PTY-reattach for interactive mode** — this is the generic multiplexer/daemon concern: reattach the tmux/abduco (or daemon-held) master, replay a serialized screen snapshot + history log, then send a redraw nudge / SIGWINCH so the Ink TUI repaints (it only repaints on resize/activity). For *background* agents, reconnect via the daemon / `claude agents`. `--remote-control` is the agent-native remote-reattach path.
- **Idempotency / race hazards (established):** (1) Resuming a sessionId that is **already live** in another process can double-attach / corrupt the transcript — **check `sessions/<pid>.json` for a live pid on that sessionId before resuming**, or use `--fork-session`. (2) The transcript is append-only JSONL written by the live process; tailing while it writes is safe but parse defensively (the last line may be partial). (3) Don't SIGKILL the attach client mistaking it for the agent (generic multiplexer hazard). (4) When both a cold-restore path and a warm-reattach path could consume the same stored session record, tag the record's origin so only one launches.

**Conflicts / open questions:** none material — both sources agree resume is `--resume`/`--continue` keyed on the UUID; the strongest pre-resume liveness check (`sessions/<pid>.json`) is the agent-native addition to prefer.

## 4. Auth & subscription

- **Credential locations (established):** file `~/.claude/.credentials.json` (mode 600). Verified shape: `{"claudeAiOauth": {"accessToken", "refreshToken", "expiresAt" (epoch ms), "scopes"[], "subscriptionType", "rateLimitTier"}}`. On macOS the OAuth tokens are typically in the **Keychain** instead: active service name **`"Claude Code-credentials"`**, user = `$USER`; Claude 2.1+ **scopes** the item per config-dir — `"Claude Code-credentials-<first 8 hex of sha256(CLAUDE_CONFIG_DIR)>"` — so reads must try both the scoped and legacy unscoped service names. API-key path uses `ANTHROPIC_API_KEY` env or an `apiKeyHelper`.
- **Auth status / management (established):** `claude auth status [--json]` returns machine-readable identity `{loggedIn, authMethod ("claude.ai"), apiProvider ("firstParty"), email, orgId, orgName, subscriptionType}`. `claude auth login | logout`. `claude setup-token` mints a long-lived token (requires a subscription) — useful for headless/CI.
- **Reuse model (established, both):** *Spawn-and-let-CLI-auth* — inherit the user's `~/.claude/.credentials.json` (simplest). *Orchestrator-managed multi-account* — isolate by per-account `CLAUDE_CONFIG_DIR` pointing at distinct config homes, each with its own `.credentials.json`; on macOS also write the account's credentials into the scoped + unscoped Keychain services at launch. The token is a standard OAuth pair, so it can be captured (`captureSystemDefaultSnapshot`-style), stored per-account, and materialized by pointing `CLAUDE_CONFIG_DIR` at the account dir.
- **Token refresh (established):** the CLI auto-refreshes the OAuth access token using `refreshToken` when `expiresAt` passes, writing back to `.credentials.json` — **the simplest correct path is to let the CLI own refresh** (or copy the updated file back). If the orchestrator must refresh out-of-band (e.g. to drive `/usage` or the usage API itself), the refresh endpoint is `https://platform.claude.com/v1/oauth/token` with `grant_type=refresh_token` and public client id `9d1c250a-e61b-44d9-88ed-5944d1962f5e`, using ~5-minute expiry skew, persisting the rotated single-use refresh token + new access token atomically back into the `claudeAiOauth` blob.
- **Env hygiene (established):** strip `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` / `CLAUDE_CODE_OAUTH_TOKEN` / `AWS_BEARER_TOKEN_BEDROCK` (and auth-like `ANTHROPIC_CUSTOM_HEADERS`) to force subscription OAuth — otherwise the inherited key wins and bills the API. `--bare` deliberately ignores OAuth/keychain and uses only the key.
- **Multi-account (established):** isolate via separate config dirs (`CLAUDE_CONFIG_DIR`) per account; select by launching with the matching dir + (macOS) the matching Keychain entry. Enforce ownership of managed dirs (realpath check, reject symlink escapes, require a marker file). There is **no first-class "account switch" flag** — selection is the config-dir swap.

**Conflicts / open questions:** the OAuth refresh endpoint/client-id and the macOS Keychain service name are confirmed at the values above; on Linux the credential lives in the `.credentials.json` file (no Keychain). `CLAUDE_CONFIG_DIR` is confirmed as the relocation var in 2.1.x.

## 5. Models, usage & accounting

- **Model listing (established + fallback):** there is **no dedicated `models`/`--list-models` subcommand**. *Established:* hardcode the documented aliases `opus|sonnet|haiku|fable` plus full ids; full ids appear in transcripts as `message.model` (verified `"claude-opus-4-8"`). *Fallback / freshest:* the interactive `/model` picker (human surface, not machine-readable), or hit the Anthropic Models API with the same creds for an authoritative live list. Prefer aliases for launch portability; prefer the API when a UI needs the current catalog.
- **Runtime model / effort switching mid-session (established, interactive only):** `/model` and `/effort` slash commands (and `/config` for defaults) switch live — but only by **typing the slash command into the PTY** (§6). **No flag flips a live session's model from outside.**
- **Usage windows / quota / rate-limits (established, multiple):**
  1. *OAuth usage API (preferred, programmatic):* `GET https://api.anthropic.com/api/oauth/usage` with the OAuth bearer token, beta header `oauth-2025-04-20`, UA `claude-code/2.1.0`. Returns `five_hour` and `seven_day` windows, each `{ utilization, resets_at }` (map to used-percent + window-minutes 300/10080 + reset time). Refresh the OAuth token first if expiring (§4).
  2. *Interactive `/usage` (TUI view):* plan-usage bars, 24h/7d breakdown (`d`/`w` toggle), per skill/subagent/plugin/MCP attribution — computed from **local** session history (other devices/claude.ai excluded). `/usage-credits` sets a monthly spend cap (Pro/Max).
  3. *PTY scrape fallback:* spawn interactive `claude`, send `/usage`, strip OSC/CSI, regex-scrape "Current session"/"Current week" percent + "resets…" lines (cap ~25s / ~100KB).
  4. Authoritative billing/quota is the Claude Console Usage page / API. The credential's `rateLimitTier` indicates the plan tier. No documented local quota-window file.
- **Token accounting (established, rich):** each `assistant` transcript record carries `message.usage` with verified fields `input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens, server_tool_use {web_search_requests, web_fetch_requests}, service_tier, cache_creation {ephemeral_1h_input_tokens, ephemeral_5m_input_tokens}, speed`, plus a per-`iterations[]` breakdown. Sum across records for per-turn/per-session totals (dedup later rows that have fuller usage by message/request id). Also exposed via OTEL metric `claude_code.token.usage` and the `-p --output-format json` result envelope.
- **Pricing / cost (established + fallback):** *In-CLI:* `/usage` prints `Total cost: $X` (local estimate); OTEL metric `claude_code.cost.usage` emits cost; the `-p --output-format json` envelope includes cost. There is **no local pricing-table file** in `~/.claude`. *Orchestrator-computed (fallback, full control):* maintain a per-model `$/M tokens` rate table (input/output/cache-read/cache-write, incl. >200k long-context tiers) keyed by normalized model id (strip `-thinking`/dotted/placeholder suffixes via aliases) and multiply against the transcript `usage` tokens.
- **OTEL telemetry (established, best push-based analytics):** enable with `CLAUDE_CODE_ENABLE_TELEMETRY=1` + `OTEL_METRICS_EXPORTER` (otlp|prometheus|console|none) + `OTEL_LOGS_EXPORTER` + `OTEL_EXPORTER_OTLP_ENDPOINT`/`_PROTOCOL`/`_HEADERS`. Verified metrics include `claude_code.token.usage, claude_code.cost.usage, claude_code.lines_of_code.count, claude_code.session.count, claude_code.commit.count, claude_code.pull_request.count, claude_code.active_time.total, claude_code.tool.execution, claude_code.api_request, claude_code.api_error`. Events/logs: `claude_code.user_prompt, claude_code.tool_result, claude_code.api_request, claude_code.api_error`.

**Conflicts / open questions:** both the OAuth `api.anthropic.com/api/oauth/usage` endpoint and the OTEL channel are valid programmatic quota/usage sources — the OAuth endpoint gives precise reset windows, OTEL gives push metrics; `/usage` PTY scrape is the last-resort fallback. The exact alias→version mapping (`opus`→which build) drifts across releases; query the Models API for a live catalog.

## 6. Driving & input — two-way control

- **Send a new user message mid-session (established):** type bytes into the PTY, then submit with **Enter (`\r`)**. For multi-line / pasted content use **bracketed paste** (`ESC[200~ … ESC[201~`, i.e. `\x1b[200~`/`\x1b[201~`) so embedded newlines aren't treated as submits. **Critical timing quirk:** send the paste body via bracketed paste, then send the submitting `\r` as a **separate PTY write after a short delay (~50ms)** — if paste-end and Enter arrive in the same write, Claude Code can leave the prompt as editable text instead of submitting. For headless, `--input-format stream-json` accepts realtime user messages on stdin.
- **Read current input-box contents / draft-sync-out (fallback only):** **no API / OSC / file exposes the composer buffer** → must **screen-scrape** the PTY prompt-box region (honoring dim-placeholder SGR vs real text to tell empty-vs-drafted). No clean structured source.
- **Pre-fill / set the input box / draft-sync-in (established for launch; fallback mid-session):** the supported pre-fill is the **initial** argv/`-p` prompt at launch (and `-n/--name` text). *Mid-session pre-fill* = type bytes into the PTY (optionally via bracketed paste) **without** a trailing CR so it stays un-submitted. There is no flag/env for live composer pre-fill of an already-running session.
- **Answer interactive prompts / permission menus (established detection; fallback selection):** *Detection (authoritative):* `sessions/<pid>.json` → `status:"waiting"` with `waitingFor` (`"permission prompt"`, `"dialog open"`), and/or the `PermissionRequest`/`Notification`/`Elicitation` hooks (§7/§8). *Selection (no structured channel):* menus are driven by **arrow keys + Enter** and **number keys** for options, **Esc** to cancel — reading the *set of options* requires scraping the menu region. **Preferred:** pre-answer via `--permission-mode` / a `PreToolUse` hook decision so menus never appear.
- **Interrupt / cancel / escape (established):** **Esc** stops the current turn immediately; **double-tap Esc** (or `/rewind`) restores to a previous checkpoint (backed by `file-history-snapshot` transcript records); **Ctrl+C** interrupts/exits. As a safety net, after sending ESC/Ctrl-C, if the CLI misses its own cancellation hook within a settle window (~500ms) synthesize a final `done`+`interrupted` state. Claude's own interrupt is also captured directly: a `Stop` hook with `is_interrupt:true`.
- **Slash commands (established):** `/model, /effort, /config, /usage, /usage-credits, /clear, /compact, /rename, /resume, /context, /mcp, /add-dir, /rewind` plus user/plugin/skill commands (`/<skill-name>`). All issued by typing into the PTY. The only slash command the orchestrator typically *issues itself* is `/usage` (rate-limit fallback). `--disable-slash-commands` turns them off.
- **Attachments / large paste (established):** `--file <file_id:relative_path>` downloads file resources at startup. Pasted images/large content are stored under `~/.claude/paste-cache/` and referenced as `attachment` records in the transcript (`type:"attachment"`). Large pasted text uses bracketed paste into the PTY (size-cap inline drafts on Windows and fall back to post-ready paste above ~24k chars).
- **Important keyboard shortcuts / modifiers (established; for mobile soft-keys):** **Enter** = submit (`\r`); **Shift+Enter** / `\`+Enter / **Option+Enter** = newline; **Esc** = interrupt; **double-Esc** = rewind; **Ctrl+C** = cancel/quit; **Shift+Tab** = cycle permission mode (default→acceptEdits→plan→auto→bypass); **Up/Down** = history / menu nav; **`/`** = slash menu; **`@`** = file/at-mention; **Tab** = autocomplete; **number keys / arrows + Enter** = menu choice. Alt/Option fidelity matters for newline-on-Option+Enter and non-US keyboards (macOptionIsMeta); surface a dedicated newline key + Esc on mobile.

**Conflicts / open questions:** both sources independently confirm the bracketed-paste + *separate* CR submit quirk — treat it as load-bearing. There is no structured composer read/write channel; draft sync in either direction is screen-scrape (out) or type-into-PTY (in).

## 7. Agent-state classification

- **State vocabulary (established, agent-native):** the CLI's own `status` enum is **`idle | busy | waiting`** (verified in `sessions/<pid>.json`), with `waitingFor` qualifying the wait. Map to the orchestrator's `working | waiting | done` (+ `interrupted` flag, + a generic `error`): `busy`→working, `waiting`→waiting-on-user, `idle`→done/ready, process-exit→done/error. (`blocked` is available in a broader vocabulary but the CLI never emits a distinct blocked signal.)
- **Signal sources & authority rank (established):**
  1. **`sessions/<pid>.json` `status`/`waitingFor`** — authoritative, real-time, structured, agent-native (`statusUpdatedAt` for freshness). **Strongest signal.**
  2. **Hooks** (§8) — push-based, low-latency event stream (authoritative for transitions).
  3. **Transcript JSONL tail** — new `assistant` record = activity; an unmatched `tool_use` (no `tool_result`) = in-flight; on `Stop`, read the last assistant message.
  4. **Process foreground / child presence** — confirm the `claude` child holds the PTY foreground (`ps stat` `+`); child tool processes (Bash etc.) indicate working.
  5. **Process exit** — done/crash.
  6. **OSC title** — Claude sets OSC 0/2 with prefixes (`✳`/braille-spinner = working, `". "`/`"* "` = idle) usable for label + a *hint*, but **do not treat the OSC title as the authoritative status source** — use it for unread/notification UI only.
  7. **OSC 9999 JSON status channel** — a generic `\x1b]9999;{json}\x07` channel exists but Claude is hook/registry-driven, not OSC-9999-driven (no-op for Claude).
- **Event→state map (established):**
  - `UserPromptSubmit` → **working** (also the new-turn boundary)
  - `PreToolUse` / `PostToolUse` / `PostToolBatch` → **working**; `PostToolUseFailure` → **working** (surface the error text but keep working)
  - `PermissionRequest` / `PermissionDenied` / `Elicitation` → **waiting** (needs user)
  - `Notification` → **waiting** (idle-prompt / input-needed)
  - `Stop` → **done** (set `interrupted:true` when `is_interrupt===true`)
  - `StopFailure` → **done/error** (the post-error completion path — see §14)
  - `SubagentStart` → working (nested); `SubagentStop` → parent may resume
  - `SessionStart`/`SessionEnd` → lifecycle
  - anything else → ignore
- **"Stopped but needs user" vs "done" (established):** distinguished **authoritatively** by `status:"waiting"` + `waitingFor` (permission/dialog) vs `status:"idle"` after a `Stop`. Hook-wise: `PermissionRequest`/`Elicitation`/`Notification`(input-needed) = needs-user; `Stop` with no pending permission = done. (`settings.json` even exposes `inputNeededNotifEnabled`/`agentPushNotifEnabled` toggles for exactly this distinction.)
- **Reconciliation (established):** prefer `sessions/<pid>.json` `status` as ground truth; use hooks for instant transitions; cache per-pane last prompt/tool/status and reset on the new-turn (`UserPromptSubmit`) event; apply stickiness so a momentary idle between tool calls doesn't flap to "done" — confirm done only on `Stop`/`idle`+quiet-window; debounce output activity ~100–300ms; decay status freshness after ~30min stale.
- **Sub-agent / nested state (established):** `SubagentStart`/`SubagentStop` hooks + transcript records carry `isSidechain:true` and hook fields `agent_id`/`agent_type` (and `tool_use_id`). Background agents appear in `sessions/*.json` with `kind:"bg"` and in `claude agents --json`.
- **Latency (established):** hooks fire synchronously on the event (sub-second); `sessions/<pid>.json` updates on status change (`statusUpdatedAt`); transcript tail is near-real-time as records append. The managed shell hook should cap its own POST (e.g. `--connect-timeout 0.5 --max-time 1.5`).

**Conflicts / open questions:** the two sources differ on the *primary* live-state source — one leads with hooks, the other with `sessions/<pid>.json`. **Resolution:** `sessions/<pid>.json` is the strongest single source (structured, agent-native, includes `waitingFor`); hooks are the lowest-latency transition signal. Use both — registry as ground truth, hooks for instant edges. The broader hook event set (`PermissionDenied`, `Elicitation`, `PostToolBatch`, `StopFailure`, `Notification`) is real in current versions; a 7-event subset (`UserPromptSubmit, Stop, StopFailure, PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest`) is sufficient for core state and is the conservative install set.

## 8. Hooks & instrumentation install

- **Mechanism (established):** first-class **hooks** — managed shell-command scripts configured in settings JSON. Handler types beyond `command`: `http` (POST the JSON directly), `mcp_tool`, `prompt`, `agent`. Plus **plugins** (`hooks/hooks.json`) and skill/agent frontmatter hooks. This is the preferred instrumentation surface over scraping; no trust/hash step is required for user-scope settings (Claude trusts its own settings.json).
- **Install location & trust (established):** `~/.claude/settings.json` (user / all-projects), `<repo>/.claude/settings.json` (project, shareable), `<repo>/.claude/settings.local.json` (gitignored), managed-policy settings (org), plugin `hooks/hooks.json`. Verified live: this machine's `~/.claude/settings.json` already wires `Notification, Stop, UserPromptSubmit, SessionStart, PermissionRequest, SessionEnd, PreToolUse, PostToolUse`, including a custom NDJSON status sink. Multiple hook objects coexist under one event (merge, don't replace). On Windows, slash-normalize the hook command path (Claude runs hooks through Git Bash).
- **Hook event taxonomy (established):** Session — `SessionStart` (source: startup|resume|clear|compact, + `model`), `Setup`, `SessionEnd`, `InstructionsLoaded`. Per-turn — `UserPromptSubmit`, `UserPromptExpansion`, `Stop`, `StopFailure`. Tools — `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PostToolBatch`, `PermissionRequest`, `PermissionDenied`. Agent — `SubagentStart`, `SubagentStop`, `TaskCreated`, `TaskCompleted`, `TeammateIdle`. File/config — `FileChanged`, `CwdChanged`, `ConfigChange`, `WorktreeCreate`, `WorktreeRemove`. Context/display — `PreCompact`/`PostCompact`, `MessageDisplay`, `Notification`, `Elicitation`, `ElicitationResult`. (Matchers like `*` apply to tool events.) **Conservative core install set** that covers state classification: `UserPromptSubmit, Stop, StopFailure, PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest`.
- **Payload fields worth extracting (established):** common stdin JSON: `session_id, transcript_path, cwd, permission_mode, hook_event_name, effort{level}, agent_id, agent_type`. Tool events add `tool_name, tool_input` (per-tool key map for preview — Read/Write→`file_path`, Bash→`command`), and on Post add `tool_response`/result. `Stop` carries `last_assistant_message`, `is_interrupt`, `transcript_path`. SessionStart adds `source, model`; Notification adds `message, type`; FileChanged adds `file_path`. Also `tool_use_id` for tool/subagent correlation.
- **Transport back (established):** command hook → your script reads stdin JSON and POSTs to the orchestrator with an injected port/token (e.g. `POST http://127.0.0.1:${HOOK_PORT}/hook/claude` with an auth header + `paneKey/tabId/worktreeId/version/payload`), or appends to a file/socket. The native `http` hook type POSTs the JSON directly to `url` with `headers`/`allowedEnvVars`. **Fail-open:** missing port/token → `exit 0` silently; exit 2 = block (stderr → error); other = non-blocking error. Use `async:true` for fire-and-forget so callbacks never block the turn. Source the live port/token from an endpoint file so a PTY that survived an orchestrator restart refreshes stale baked-in env.
- **Lifecycle (established):** install by writing settings JSON (idempotent — strip prior managed entries via a managed-command matcher, then merge); update/uninstall = edit the JSON (remove only managed entries); `getStatus` can report `installed | partial | not_installed | error` per missing event. Version-skew: the event set grows across versions (gate on `claude --version`); unknown events are simply never fired. `--safe-mode`/`--bare` disable hooks (boot caveat). Remote install (SFTP) writes the POSIX script + `$HOME/.claude/settings.json` on the remote box.

**Conflicts / open questions:** both sources agree hooks are the mechanism and `~/.claude/settings.json` the install point. They differ only on event-set breadth (full taxonomy vs the 7-event core) — both are correct; install the core set for portability and opt into the richer events (`Elicitation`, `PermissionDenied`, `Notification`, `PostToolBatch`) where the version supports them. One operational note: if `DEVIN_PROJECT_DIR` is set, an imported Devin hook config may also fire — a managed script may want to self-skip in that case.

## 9. Session titles

- **Source of truth (established, agent-native):** Claude Code **generates an AI title** — a transcript record `{"type":"ai-title","aiTitle":"…","sessionId":…}` (appears after the first prompt). This is the preferred title. A user-set `-n/--name` (or `/rename`) display name also lives in `sessions/<pid>.json` `name`, the prompt bar, the `/resume` picker, and the terminal title. For historical transcripts, prefer explicit title records in priority order `custom-title` → `ai-title` → `agent-name`, else the first user message text.
- **OSC title (established, live channel):** the CLI sets the terminal title (OSC 0/2) to the session name/title with status prefixes (`✳`/spinner/`". "`/`"* "`). An orchestrator can intercept the OSC sequence as a live title channel — but use the `ai-title` record / `name` as the canonical name source rather than the churny OSC string.
- **Read / update / eventing (established):** tail the transcript for the `ai-title` record; watch `sessions/<pid>.json` `name` for renames; `-n/--name` is set at launch, `/rename` mid-session.
- **Fallback (established):** if no `ai-title` yet, synthesize from the first user prompt (truncate `history.jsonl` `display` / first transcript `user` message at a word boundary), or use `--name`.

**Conflicts / open questions:** agreement that `ai-title` (or `custom-title`) is the canonical title and first-prompt truncation the fallback; the exact OSC title payload format is unverified, so treat the structured records as truth and the OSC as a hint.

## 10. Transcript & history

- **Storage (established):** JSONL, **one file per session**, `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl`. Append-only. Overflow tool results spill to `<sessionId>/tool-results/<toolUseId>.txt`. Filename ≈ session id.
- **Record types (established, verified `type` values):** `user`, `assistant`, `attachment`, `last-prompt`, `mode`, `permission-mode`, `bridge-session`, `ai-title`, `file-history-snapshot`, and `summary` (some sessions); plus title records `custom-title`/`agent-name`. Top-level keys seen across records: `uuid, parentUuid, sessionId, type, timestamp, cwd, gitBranch, version, userType, isSidechain, isMeta, message, requestId, toolUseResult, promptId, promptSource, origin, entrypoint, permissionMode, leafUuid, bridgeSessionId, lastSequenceNum, aiTitle, lastPrompt, snapshot, messageId, isSnapshotUpdate, sourceToolUseID, sourceToolAssistantUUID, attachment, attributionPlugin, attributionSkill`.
- **Message schema (established):** `user.message = {role, content}` (content is a string or a content-block array; extract text via the first `.text` of array blocks). `assistant.message = {id, type, role, model, content[], usage, stop_reason, stop_sequence, stop_details, diagnostics}` (verified `model: "claude-opus-4-8"`). Content block types: `text`, `thinking` (reasoning), `tool_use {id, name, input, caller, type}`, `tool_result {tool_use_id, content, type}`. `parentUuid` links the DAG (supports forks/rewind). Token counts read from `message.usage` (§5).
- **Live tail vs historical (established):** *Live* — tail the JSONL (inotify/poll on size), parse new whole lines, order by file order (or `timestamp`/`lastSequenceNum`), dedup by `uuid`; on the `Stop` hook, reverse-chunk-scan (e.g. 64KB chunks up to a few-MB cap) for the last `assistant` text. *Historical* — full streamed line-by-line parse; detect changes by file mtime on rescan; dedup later rows with fuller usage by message/request id. Robust against a partial last line.
- **Consumers (established):** chat view ← `user`/`assistant` text (+ optionally `thinking`); last-message/preview ← last `assistant` text or `last-prompt`; clickable items ← `tool_use` (`input.file_path`, diffs from Edit/Write old/new); state classification ← unmatched `tool_use` + `Stop` (via hooks) + `usage`; resume id ← `sessionId`/filename.
- **Transcript ↔ terminal mapping (established):** loose — the JSONL is the structured truth; the PTY is the rendered view. For a browser chat that mirrors the terminal, render from the transcript (clean) and use the PTY only for raw fidelity; reconcile by `uuid`/`timestamp`. No line-level reconciliation between the two.
- **Special content types / quirks (established):** **thinking** blocks are explicit `type:"thinking"` content (don't surface if hiding reasoning). **Images / pasted content** → `attachment` records + `~/.claude/paste-cache/`. **Tool outputs** can be inline in `tool_result.content` or spilled to `tool-results/<id>.txt` (follow the reference). **Diffs** come from Edit/Write `tool_use.input` (old/new) — synthesize the diff. **`isMeta`/`isSidechain`** mark internal/subagent entries (filter for the user chat). **`bridge-session`** links a local session to a cloud/web bridge id (`bridgeSessionId`/`cse_…`). Cache-token fields (`cache_read_input_tokens`, `cache_creation_input_tokens`) are the main accounting quirk.

**Conflicts / open questions:** both sources agree on JSONL/one-file-per-session and the core `user`/`assistant`/`usage` schema; the richer record-type and content-block list (`thinking`, `attachment`, `file-history-snapshot`, `bridge-session`) is verified on disk and supersedes any narrower description.

## 11. Terminal rendering & display fidelity

- **PTY → stream → xterm (established, generic):** Claude Code is an Ink/React TUI emitting ANSI/SGR + cursor moves; pipe PTY bytes straight to xterm.js. Intercept/strip OSC 9999 status sequences before display (generic; mostly a no-op for Claude). It uses its own redraw for the conversation region (menus/spinners repaint).
- **Resize / reflow (established, generic):** send the new cols/rows + `SIGWINCH` on resize; the Ink app repaints on resize, so a resize nudge doubles as a redraw trigger after reattach. Mobile↔desktop breakpoint remounts must re-send size + a redraw nudge.
- **Scrollback model (established, generic):** orchestrator-owned xterm scrollback for history is the safe model; the TUI repaints its live region. Forward mouse clicks/scroll only when the TUI enables mouse reporting (menus); otherwise let xterm own scrollback. Alt-screen/mouse-tracking private modes are tracked generically.
- **Snapshot / serialize for reattach (established, generic):** use headless xterm (`@xterm/headless`) + serialize-addon to persist the last screen; on reattach replay the serialized buffer, reconnect the live PTY, then SIGWINCH-nudge for a fresh repaint. Pair with an incremental history log for full scrollback restore.
- **Local cache for instant render then live reconnect (established):** persist the serialized xterm buffer per session for instant paint; **additionally** render the structured transcript JSONL for an instant clean chat view that needs no PTY at all (a major advantage — the JSONL is the durable history). Handoff: show cached buffer/transcript immediately, attach the live PTY, redraw-nudge, then reconcile new JSONL records by `uuid` to avoid dupes.
- **Composer / cursor (established):** the input box shows a **dim placeholder when empty** (detect dim SGR to tell empty-vs-drafted, needed for §6 draft sync). The bracketed-paste-ready handshake is **DECSET 2004** (`\x1b[?2004h`); wait a quiet window (~1.5s) after it before pasting a draft.
- **Color/theme/OSC (established):** themes via `/config`/custom themes; sets OSC terminal title (§9); OSC links collected generically. `--ax-screen-reader` renders flat text without borders/animation — useful as a low-fidelity/mobile fallback render mode.

**Conflicts / open questions:** rendering is agent-generic; whether Claude enters a full alt-screen for any modal is unverified (treat menus as repaintable in-place). The big lever both sources agree on: the JSONL transcript is a second, clean render source independent of the PTY.

## 12. Background / headless invocation

- **Non-interactive one-shot (established):** `claude -p "<prompt>"` (prompt via argv or stdin: `echo … | claude -p`). `--output-format text|json|stream-json`: `json` returns a single result envelope (includes `session_id`, result text, usage/cost); `stream-json` streams events (+ `--include-partial-messages`, `--include-hook-events`, `--verbose`). `--input-format stream-json` for streamed input. `--json-schema <schema>` enforces structured output. `--max-budget-usd <amount>` caps spend (print mode). `--no-session-persistence` / `CLAUDE_CODE_SKIP_PROMPT_HISTORY` avoid writing transcripts. `--fallback-model` for overload failover. Resume headless: `claude -c -p "…"` or `claude -r <id> -p "…"`. Detect a headless one-shot by the presence of `--print`/`-p` or `--output-format json|stream-json` (so it isn't treated as an interactive pane).
- **Recommended one-shot spec (established):** binary `claude`, prompt via **stdin** for large diffs, args `-p --output-format text|json --model <alias> --permission-mode plan [--effort <level>] [--json-schema <schema>]`. Claude is a strong default for the orchestrator's own LLM chores.
- **Uses (established):** commit messages, PR title/body, branch names, session summaries/titles, classification — `--json-schema` + `-p --output-format json` gives clean structured returns. Model discovery is *not* a headless call (no `--list-models`; use aliases / Models API, §5).
- **Auth path for headless (established):** inherits the same OAuth/`.credentials.json` (or `ANTHROPIC_API_KEY`) as interactive; `claude setup-token` provides a long-lived token for CI; `--bare` for a minimal key-only run. Same managed-account materialization (`CLAUDE_CONFIG_DIR`) applies.
- **Cost / latency / timeout / locality (established):** `--max-budget-usd` caps cost; background functionality itself is cheap (<$0.04/session per docs). Set your own process timeout (no documented hard request cap). Runs local or over SSH/daemon (§15).

**Conflicts / open questions:** the default per-request timeout is unverified — set an orchestrator-side timeout. Both sources agree `-p` + `--output-format json` is the robust headless contract.

## 14. Failure, exit & recovery

- **Crash / exit detection (established):** process exit code — `0` clean, non-zero = error. In `-p` mode the `json` envelope carries success/error + result. **Distinguish crash from clean completion via the structured signals, not the exit code alone:** a normal turn ends with the `Stop` hook (→done) / `status:"idle"`; `StopFailure` fires on a turn-level API/model error (the CLI may *skip* the normal `Stop` after such an error and emit `StopFailure` instead — without handling it the turn appears to spin forever). `PostToolUseFailure` surfaces tool errors while keeping state working. OTEL `claude_code.api_error`/events surface API failures.
- **Reattach-after-restart healing (established):** a `sessions/<pid>.json` whose `pid` is no longer alive (verify with `procStart`) = stale → mark exited/recoverable and offer `--resume`. The append-only transcript means **no data loss on crash**; resume rebuilds history. For interactive PTYs, the multiplexer/daemon master surviving = still-alive even if the orchestrator restarted (don't mark exited just because the server restarted; use a foreground-process `+` gate so a stopped child can't masquerade as live). Reap orphan/zombie tool subprocesses by pid.
- **Error surfacing (established):** route `StopFailure`, `Notification` (with error/type), OTEL `api_error`/`api_refusal`/`internal_error` events, non-zero exit, and `assistant.message.stop_reason`/`diagnostics` from the transcript into a capped `lastAssistantMessage` shown in the dashboard. There is no distinct CLI `error` agent-state — map error completion to `done` with an error flag (and `interrupted` for user cancellation).

**Conflicts / open questions:** both sources agree `StopFailure` is the critical "API error skipped the normal Stop" signal — install it or turns hang. Whether a given version emits a richer error event set varies; gate on version.

## 15. Remote / transport

- **Transports (established):** (a) **local** PTY/daemon — default; (b) **SSH to a remote worktree** — run `claude` on the remote host over an SSH-hosted PTY (generic, no agent-specific blocker); (c) **daemon-subprocess** — Claude Code's own background-agent daemon (`~/.claude/daemon.*`, `claude agents`) hosts sessions out-of-process and survives client restarts; (d) **Remote Control** — `--remote-control [name]` (+ `remoteControlAtStartup` setting) exposes a session for remote driving; `--teleport` resumes a Claude-Code-on-the-web session into a local terminal (`CLAUDE_CODE_REMOTE=true` marks web envs). (c) and (d) are agent-native remote paths beyond raw SSH.
- **Forwarding over the relay (established):** for remote sessions, forward PTY bytes + SIGWINCH over the relay; run git/fs ops on the host that owns the worktree (the transcript / `sessions/*.json` / credentials all live under that host's `~/.claude`). Instrumentation callbacks: install hooks (esp. the `http` hook type, or a managed POSIX `claude-hook.sh` + `~/.claude/settings.json`) that POST to a relay-reachable endpoint (`127.0.0.1:${HOOK_PORT}` on the remote, with a relay-side listener that stamps the real connection id on receipt), or write to a file the relay tails — **fail-open** so a relay outage never blocks the agent. Ship the structured channels (transcript JSONL, `sessions/*.json`, hooks, OTEL), not just the PTY — they are the durable, clean state. Remote hook install is POSIX-first (remote-Windows is a gap).

**Conflicts / open questions:** both sources agree on local / SSH / daemon transports; the agent-native `--remote-control`/`--teleport`/background-daemon paths are version-gated and complement (don't replace) a generic SSH+multiplexer setup.
