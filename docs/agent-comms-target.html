Cross-agent communication: what we land on

2026-07-17 · evidence: 491-message POD-279 fan-out audit, live ledger forensics, POD-463/468/471/473/495 fixes, POD-285/216/475/480/487/493/527/530/652 · author: POD-237 session

00The verdict

A month of live fire and a 491-message coordinator audit produced one clear split: the substrate (attributed, ledgered, durable messages) is sound — every bug below was diagnosed from its ledger. The delivery + notification layer is where all the pain is, and the POD-279 fan-out quantified it: 49% of all traffic was ceremonial acks, 243 acks-of-acks, every notification a duplicate settle-nag, and 70 load-bearing messages that reached no one. We keep the core, rebuild delivery around pull at safe boundaries, push only by explicit interrupt, and put a single arbiter in front of every poke.

Keep — it works

Message substrate: server-stamped sender, envelope (unwrapped = operator), ledger, threads/acks, clamp matrix, wake-resume, spawn/await with budget + provenance.

Fix — it hurts

Typing into live PTYs (menus, mid-turn), delivered ≠ consumed, unclearable mail nag, ack ceremony storm, level-triggered re-firing subscriptions, three uncoordinated notifiers, child needs_user invisible to its parent.

Add — it's missing

Notification arbiter, read/clear semantics, dead-lettering, milestone events (integrated), structured provenance, scheduled wake, stop primitive, upward-visible child blocking.

01The five layers of cross-agent comms

These are five distinct relationships, not one mechanism — you named them and they have genuinely different needs. Reading top→bottom is tightest coupling → loosest: L1 shares a context; L5 shares only a codebase. Each layer says what Podium provides today and what's missing.

▲ tightest — shares one session & context
1
native subagents · built-in, not Podium

In-context helpers

The harness's own Task tool. Ephemeral, share the parent's session / cwd / worktree, invisible to Podium (no session row, no issue binding, no ledger). Right for context-local legwork inside one issue. Footgun today: their podium commands run on the parent's relay env, so a subagent's issue attach re-homes the parent — and because they're in-process (same env), nothing local can tell them apart. Blocker for M5 (§11); the deterministic path is a harness marker or parent-mediated attach.

2
remote subagents · Podium-managed

Real workers, any harness / model / machine

Full Podium sessions: durable, attachable, on their own issue+branch or sharing the parent's. The coordinator↔executor↔reviewer relationship. Podium owns spawn, provenance, budget, ledger, the read ladder. Frontend is CLI today, MCP-mirror proposed (§03).

3
blocking issues · hard dependency

“I can't finish until yours lands”

A concrete dep edge. Agent A didn't necessarily drive B (may have issue started it), but its own work depends on B's outcome — a discovered blocker, or two intertwined initiatives. A wants to be notified & woken when B reaches the agreed milestone.

4
related work · soft coordination

“Adjacent to yours — tell me before I merge”

Not dependent, but working the same area toward a shared goal. Needs to know at specific points only — chiefly its own merge point and when B lands — to rebase and avoid merging at the wrong time. Ordering matters without being a start-block. Pull-gated at A's boundary, not push on every B event.

5
general awareness · ambient

“What's been happening in the codebase?”

No relationship at all — just wants a read-on-demand overview, typically at the start of work and again before merging, to catch up on what landed since it started. Never a push; always a pull.

03Frontend: MCP or CLI (both — over one engine)

This is a frontend/ergonomics choice, not an architecture choice — and both already coexist today: the superagent reaches Podium over MCP (mcp__podium__*, wired in mcp-route.ts), while spawned workers use the CLI over PODIUM_ISSUE_RELAY. Same relay procs underneath. Your idea — MCP tools that mirror each harness's own subagent/Task tool shape — is extending the MCP frontend down to workers so cross-harness spawning feels identical to a native subagent.

CLI shipped, universal

  • Works everywhere — any harness can shell out; no per-harness integration.
  • Operator-usable and scriptable/composable.
  • Already the worker path; discoverable via --help.
  • Cost: stringly returns; the agent must learn Podium's vocabulary; it isn't the affordance the model was trained on.

MCP-mirror feasible on live rails

  • Zero learning curve — mirror the native Task tool's shape/params, so “spawn a Codex reviewer” feels like the built-in subagent. Highest ergonomic win.
  • Structured params & returns; rich status objects, not parsed strings.
  • Cost: per-harness MCP support varies (Claude strong; Codex takes raw JSON; grok/opencode/cursor uneven). A long spawn+await doesn't fit one synchronous MCP call — you still need poll/await tools.

Decided (your q4): CLI for now. Keep the relay procs as the single source of truth and stay on the universal CLI floor — no MCP-mirror build this round. The MCP-mirror stays a documented later option (the rails exist — mcp-route.ts, per-agent --mcp-config — and MCP is already live for the superagent), to revisit once the messaging foundation lands. Either way it's one engine, never two. Reconcile with POD-285's “CLI/relay only” note — that was scoped to one workflow's notify path, not a global MCP retirement.

04bSynchronous send & the death of acks

The ack storm (§09-A) and the silent black hole (§09-C) have one shared fix: make send synchronous and split “receipt” from “response”. This is the single biggest quality-of-life change and it removes ~half the traffic.

Send returns after it lands new

  • The CLI blocks until the message is durably queued to a valid target. No more “queued but never delivered” — the sender knows it landed or gets an error. Kills §09-C and §09-5a.
  • interrupt: return once injected (ESC + text confirmed).
  • next-turn / fyi: return once durably enqueued to a target that exists and can receive — we don't block for the read; the boundary/harness drains it. That's enough: the sender has certainty of landing, not of reading.
  • Issue with no session / gone: the send tells you (“held for next session” / “dead-lettered”), never silently drops.

Receipt is mechanical; response is opt-in new

  • Receipt acks vanish. Synchronous send already proves receipt; the ledger records delivery. Senders pull delivery status; nothing is pushed. — kills 243 ack-of-acks.
  • A response is requested, not reflexive. Only mail send --expect-response (or a question) asks for one — “I want this answered / proven”.
  • Responses are pull-delivered — put in the sender's mailbox, surfaced on their next stop, never a next-turn that burns a turn. An ack now means “the thing you asked is done”, never “received”.
  • The settle-nag only fires for an unfulfilled requested response — not for every unacked message.

Net effect on POD-279: the 49% ack traffic, the 243 ack-of-acks, the 15× self-echo, and the false “without acking” nags all disappear — receipt is mechanical and self-delivery is suppressed. What remains is real: targeted messages, and responses that were explicitly asked for.

Spawn returns the resolved truth (your point). Debounce the state — don't trust the instant (your point). And since we deliver on idle, a false idle = a premature mid-turn injection — the very echo-miss that fed the requeue loop. So debounce transitions, especially → idle: the state must hold for a short window before it's authoritative, and nativeSubagentCount > 0 means “not truly idle”. Stable status isn't just nicer reporting — it's a precondition for safe delivery.

05bThe delivery surface — stop typing into the one visible input line

Two live harms, one root cause. Delivery types into the single visible input surface the human shares with the agent. So (1) a message appended itself to a half-typed line and submitted it — the human's own input, corrupted; and (2) on coordinators (high message volume) the transcript fills with [podium message …] blocks that bury the agent's real last output, forcing the human to hunt for it. Both are the PTY-paste primitive again — the same fragile surface §00 flagged, now hitting the human, not just the agent.

The fix is keyed on the composer's state — empty, dirty, or dirty-and-abandoned. We already store exactly that: session_drafts(session_id, text, updated_at) tells us which.

1 · Composer empty → deliver behind a one-liner

  • Put the full message into the agent's context as background instructions (the model acts on it), and show only a thin one-liner in the transcript — “↩ message from POD-833 delivered” — so you see why the agent took a turn, without a wall of quoted text.
  • This is your “inject in the background”: agent gets the payload, human gets a breadcrumb.

2 · Composer dirty → hold, attach on next send

  • You're typing: never inject. Attach the message to the session and show a UI marker — “1 message awaiting delivery” — that auto-attaches on your next send (your text goes; the message rides along as context).
  • No timers, no corruption: the human's keystroke is what releases it.

3 · Dirty but abandoned → save, deliver, restore

  • Left dirty and idle past a timeout: save your draft, deliver our message, then restore your draft.
  • Needs the draft to be a first-class synced object → builds on POD-816 (draft sync chat/native) and only ships when 816 does.

This replaces the “heavy sidebar” idea for this problem — you were right, it's too much. One-liner-in-transcript + attach-on-next-send needs no new surface. The ledger sidebar (POD-467) stays a separate, optional thing for browsing message history, not for making a single delivery legible. Open harness question for case 1: can the agent receive full content that the UI renders as a one-liner? If not natively, store the full turn and let the UI collapse it — same effect, no harness change.

07bThe coordination model — publish, subscribe, wake

Three of your questions are one question: should this be pub/sub? is the arbiter the steward? how does a coordinator loop without a timer? The answer ties them together.

Publish state, don't mail everyone

  • Two modes on one substrate: targeted messages (agent→agent, human→agent — delivered directly) and published events (an agent changes state / hits a milestone — the substrate fans it to subscribers).
  • POD-279's storm was the coordinator mailing its own issue for everything. Publish-state replaces that: it emits “POD-298 integrated”, and only subscribers hear it.
  • No subscribers ⇒ no delivery — a published event with nobody listening is free. That's how you avoid storms: the audience is defined by subscriptions (implicit edges + explicit watches), not by the sender broadcasting.

The arbiter IS the steward

  • Don't build a second component. The steward already ticks the event log and dispatches — give it the arbiter's rules (edge-trigger, fact-dedup, retire-on-consume, suppress-in-context).
  • Tick-based for system/implicit notifications (settle, milestones, digests): the tick naturally batches & dedups, and a fact that resolves within a tick never fires.
  • Live for explicit targeted messages: they bypass the arbiter's batching and deliver immediately — the sender already chose to poke a specific someone.
  • Automated notice always yields to a human/agent one. See the box below.
Your sharper question — the agent deliberately messages the parent “I integrated”, THEN closes, and the close fires a second notice. How do we make it near-impossible to get wrong?
Primary fix: the state change IS the message — there is no separate “notify” step to double up. The agent doesn't mail “I integrated” and then close; it closes with its context attached: podium issue close --note "integrated as abc123, here's what changed". The close publishes one notification to the subscribers, carrying that note. Because there's no separate send, there's nothing to duplicate — the natural, easiest action is also the correct one.
Safety net (for when an agent still sends a separate message): the automated notice is a fallback, not a duplicate. Before the steward fires fact F to target T, it checks the ledger — has T already been told about this issue since the change, by any message? If so, F is already-communicated and suppressed. The producer of the fact does the notifying; the steward only fills a gap.
The coordinator loop, without a timer. The parent parks; it doesn't poll. A child event (done / blocked / errored / exited-without-report) is published on the spawner edge, which carries wake rights — so a parked parent is woken to process it (the POD-279 fix: parked parents don't self-wake). The event is durable and re-surfaces until the parent's receiving turn completes (that turn-completion is the “processed” signal); the arbiter dedups so re-surfacing is never a storm. Result: an event-driven coordinator that sleeps between children and wakes exactly when one needs it — the spawner↔child relationship is the one we make watertight first (M4).

When A blocks on B, A doesn't pick “accepted vs integrated” from a menu — A writes a cleartext condition of what it's waiting for (“B's auth migration is on main”, “B's endpoint returns the new field in prod”). At A's stop-hook that condition is injected back as a question: “You're blocked on <condition> — satisfied? If yes, clear the block.” If A can't tell or it isn't met, A is prompted to update state / why it's still stopping and the usual issue fields refresh. The structured integrated fact can auto-clear a condition that literally says “B integrated” (offer that as a template); everything else is A judging at its boundary. Principle: auto-detect only what git makes certain; record decisions/signals when a real one exists; let waits be words, re-surfaced at the right moment — nothing fabricated, nothing forced.

10bOne agent, many issues — over time

You restricted it to one issue per agent because issues vanished from the sidebar when their agent moved on — but a person may genuinely want one agent to work several issues. The fix isn't to duplicate a live agent under every issue (you're right, that's confusing); it's to separate “where the agent is now” from “every issue it has touched”.

The model

  • Keep a persistent agent ↔ issue link table (many-to-many over time): {session, issue, first_seen, last_seen, active}.
  • Exactly one link is active — the issue the agent's live terminal is bound to right now.
  • podium issue attach flips which link is active and keeps the old one — so the previous issue never loses its record of that agent. That's what stops issues from vanishing.

The sidebar

  • Issues are the spine (they're durable); sessions hang under them.
  • Each issue lists its linked agents as drawers. The active one is open & lit; ones that worked it before are collapsed & greyed, with a “→ now on POD-Y” pointer you can click to follow.
  • So a given agent shows live under one issue and as a faded breadcrumb under the others — present everywhere it's been, unambiguous about where it is.

✓ You approved this model. Don't conflate it with coordinators. A coordinator overseeing POD-834/835/836 is not “attached to” them — it's attached to its own issue (POD-833), and the children are separate issues shown nested by provenance (spawnedBy), which is M6. So: sequential reuse of one agent across issues → this link-history model; concurrent oversight → provenance nesting. Two different pictures, neither needs a live agent duplicated across drawers.
