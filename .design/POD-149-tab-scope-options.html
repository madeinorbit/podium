Tab scope clarity — why POD-131's file shows up under POD-144, and four ways out

The strip looks issue-scoped (it wears the selected issue's tint) but is actually issue tabs ∪ worktree tabs: any file tab whose worktree path matches the issue's checkout is shown, and since most issues share ~/podium, files opened under one issue surface under every other. Artifact tabs opened via the artifact list carry an issueId and scope correctly; legacy path-only opens (openFileInWorktree) carry none.

POD-144 is selected (violet). The POD-131 mobile-redesign artifact rides along because it lives in the same checkout — nothing in the strip says it belongs to another issue.

Option A — Strict issue scoping recommended

Every file open made while an issue is selected is stamped with that issue's id; the strip shows only tabs owned by the selected issue. Worktree-path matching remains only for the worktree-selected (no-issue) workspace. Re-finding a closed-over file happens through the “+” menu → Recent files, which lists recently opened files across the checkout.

For
  • Strip finally means what its tint claims: “this issue's workspace”.
  • Matches the existing per-issue tab-order persistence (issue:<id> keys).
  • Small, mostly data-model change — stamp issueId at open time, tighten one filter.
Against
  • Tabs “vanish” on issue switch — needs the Recent-files affordance so nothing feels lost.
  • A genuinely shared reference file must be opened once per issue.
Cleanest mental model; pair with A′ below if cross-issue visibility is worth keeping.

Option B — Keep them, mark their provenance additive

Behaviour unchanged; foreign tabs become visually honest. A tab owned by another issue swaps the violet square for its own issue's colour and wears a tiny owner chip. Clicking the chip navigates to that issue (the reveal behaviour artifacts already have).

Option C — Segment the strip: issue zone · shared zone middle path

One strip, two zones. Left of the divider: tabs owned by the selected issue, tinted as today. Right: checkout-shared file tabs, rendered neutral slate with a mono zone label — visibly “of the worktree, not of this issue”.

Option D — Files leave the strip entirely structural

The strip becomes sessions-only; files and artifacts open in the right dock (which already hosts viewers) or a dedicated overlay. Scope confusion is dissolved because file viewing stops competing with agent tabs for the same row. Biggest change, touches split panes, warm-set mounting, and the artifact-reveal flow — a redesign of its own, not a fix.

Worth keeping in mind for the shell redesign, not as the answer to this bug-sized confusion.

Recommendation

A as the fix, B's chip as the escape hatch (A′). Stamp ownership at open time and scope the strip strictly (A). For the one real cross-scope case — an artifact you deliberately want alongside another issue's session — offer “Open here” from the artifact list, which creates a tab owned by the current issue but wearing the origin chip from B. Deliberate borrowing stays possible and visible; accidental leakage ends.
