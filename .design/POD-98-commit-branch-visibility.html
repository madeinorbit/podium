Design proposal POD-98 2026-07-20

Where a task shows its commits and its branch

Every agent session should answer two questions at a glance, without opening a shell: has it committed yet? and which branch is it sitting on? Podium's model makes this simpler than it looks — issues own branches, sessions never do — so this is per-issue state, surfaced wherever the issue's sessions render.

Recommendation One shared "git stamp" in two densities: a full chip pinned to the right edge of the pane header (always visible for whatever you're looking at), and a micro-stamp appended to the sidebar row's existing status line (ambient, across every task at once). Clicking either opens the RightDock Git tab — which already exists as a placeholder. The tray's "finished / review" cards get the stamp as a footer line, because "done" without "committed" is exactly the moment the question matters most.

01 One grammar, six states

Whatever surface it lands on, the stamp reads the same way: a branch name in mono, an ahead-count, and one dot that encodes commit state. Learn it once, read it everywhere.

⎇pod/98-commit-vizno commits Fresh branch, clean tree. Nothing committed, nothing pending — dashed hollow dot, everything muted.
⎇pod/98-commit-viz+4 files Uncommitted work. The tree is dirty but nothing is committed yet — half-filled amber dot. This is the "agent says done but nothing is saved" tell.
⎇pod/98-commit-viz↑3 Committed, clean. Three commits ahead of the parent branch, working tree clean — solid green dot. The state you want to see before merging.
⎇pod/98-commit-viz↑3+2 Committed and dirty. Three ahead plus two modified files still uncommitted — both signals side by side, no ambiguity.
✓ merged→main Merged. The branch landed on its parent — the chip relaxes to a quiet confirmation and stops demanding attention.
⎇main+7 files Branch mismatch. Red is reserved for one anomaly: the issue has a private branch, but its session is dirtying a shared branch instead. Deliberate shared-branch work (main, v3, …) is a first-class mode with its own grammar — section 05.

05 Shared checkouts — attribution everywhere

The split that matters is not main-vs-branch — it's private worktree vs shared checkout. A long-lived project branch like v3 is main in all but name: multiple tasks sit on it directly, and checkout-level facts lie there exactly as they do on main — a neighbor's commit moves the head, a neighbor's half-done edits dirty the tree. So attribution is the universal substrate: every chip's counts describe the task's own changes, which Podium can pin precisely because it runs the harness. Podium already knows which mode a task is in — an issue-owned worktree has exactly one task; any other checkout is shared.

⎇v3+4 yours This task's files, uncommitted. The count is the intersection of files this task's sessions touched with git status — a neighbor's dirty files never turn your chip amber. Their count lives in the hover ("+3 more from other sessions").
⎇v3✓ 2 commits· 8m ago This task committed. Not inferred from head movement — these are commit shas captured at the moment this task's session made them. The answer to "did the agent commit?" is exact, however many tasks share v3 or main.
⎇main✓ 2 commits⇡ unpushed Committed but not pushed. ⇡ lights when any of the task's own shas are still in @{u}..HEAD — committed locally isn't shared yet.
⎇mainno changes Nothing yet. No touched files dirty, no captured commits. Same dashed-hollow dot as a fresh branch.
Shared-checkout grammar · per-TASK attribution: counts describe this task's changes, never the checkout's aggregate — same on main, v3, or any long-lived branch

How attribution works — three layers, cheapest first. (1) Commit capture: the harness sees every tool call; a post-tool hook compares HEAD before/after each of the session's own shell commands, and any new sha is stamped into the task's gitState.commits. Interleaved agents can't cross-contaminate — each delta is measured around one session's own call (and git's index lock already serializes simultaneous commits in one checkout). (2) Touched files: the harness also sees every Edit/Write, so the task's touched-file set is known; dirty-for-you is that set ∩ porcelain output. (3) Trailer as belt-and-braces: agent commits carry a Podium-Issue: POD-98 trailer, so attribution survives server restarts and covers commits made from a task's dock shell by hand — git log --grep re-derives the list from history alone.

The old baseline-sha comparison survives only as the fallback for checkouts Podium can't observe (a human terminal outside any session) — and when the chip is running on fallback data it says so in the hover rather than pretending to per-task precision.

Why not make every chip attribution-only, then? Attribution runs everywhere — it's the substrate. But on a private issue branch there are two different questions and the chip keeps both: ✓ 2 commits (task axis: did the agent commit?) and ↑3 (merge axis: what would land if you merged the branch now — including a commit the human pushed onto it by hand). On a private worktree the two usually coincide and the chip collapses to the ↑ form; they're shown together only when they disagree, which is itself information. On a shared checkout the merge axis doesn't exist (the branch isn't going to be merged into itself), so only the task axis renders. Two honest costs of strong-everywhere: rebases and amends rewrite shas, so captured commits are matched by patch-id and re-derived from the Podium-Issue: trailer rather than pinned to raw shas; and commits made outside any session remain visible only to the branch-level counters — which is exactly what the fallback disclosure is for.

06 Data plumbing — small, mostly there

The branch half already rides the wire. The commit half is one new object computed server-side per active worktree.

  • No new git surface needed: the daemon's allowlisted RepoOp set already includes status, log, and isMergedInto (discovery.ts:184) — today they're pull-on-demand; this feature just runs them proactively and stamps the result on the issue.
  • When to refresh: no interval polling. Recompute when a session's agent turn ends (the only moment commits appear), on worktree create/merge, and on Git-tab open. Each probe is three subprocess calls, tens of milliseconds, only for issues with live sessions.
  • How it ships: gitState becomes an optional field on IssueWire — additive and zod-optional, so legacy payloads parse; it rides the existing WS push / changesSince replica untouched.
  • Why per-issue (with a per-checkout fallback): issues own branches, sessions never do — attaching state to the issue keeps one source of truth and lets the sidebar aggregate for free. On-main sessions without an issue worktree key their gitState by checkout path instead (the same cwd → worktree resolution repoBranchForCwd does today), so every session gets a chip either way.
  • Reuse the branch-lookup primitives: repoBranchForCwd() (client-core derive.ts:368) already resolves a session's cwd to {repo, branch}, and RepoBranchTag renders a repo·branch stamp today — but only on pinned panels. The chip generalizes what that tag started.
  • Performance budget: probes are event-driven (turn end, worktree lifecycle, tab open) — never interval-polled — and only for issues with live sessions; each is a few ms warm. HEAD-delta capture is rev-parse (sub-ms), fire-and-forget after the tool result, never in the tool call's latency path. The one trap: plain git status opportunistically takes the index lock and can collide with an agent's concurrent git commit in a shared checkout — all probes run with --no-optional-locks. Trailer re-derivation (log --grep) is reconcile-only and --since-bounded.
  • The detail view is free real estate: the RightDock Git tab is already stubbed ("working-tree status, diff view, commit log"). Every stamp is a link to it — the chip answers whether, the tab will answer what.
