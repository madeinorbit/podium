01 · Baseline

What the code already does, so this doc doesn't re-propose it

Audited against SidebarUnified.tsx / derive.ts (2026-07-21): rows never reorder from activity (#64), no pulse anywhere in row motion (motion.css:1-5), spin-offs stay top-level with a ⤷ origin whisper, lone-agent issues click straight through with no child row, session child rows appear only when they earn it (sessionsNeedChildRows), and finished sessions already fold behind a disclosure. All of that carries forward unchanged. What follows is only what's new.

02 · Hierarchy — the merged model

Five laws for the tree

Proposal B's diagnosis stands: the old sidebar drew three different relationships — issue contains issue, issue is executed by session, session spawned subagent — with one visual grammar, so every agent made the plan look bigger. The fix is not to delete the tree (v2's mistake) but to give each relationship its own body:

L1

The chevron keeps one promise

Expanding an issue reveals subtasks — issue rows only, on a guide line tinted with the parent's issue color. There is no second kind of child in the tree.

L2

Sessions are a band, not branches

When an issue's sessions earn visibility (multi-agent, subagents, remote), they render in a rail-navy roster band under the row — one tone tier below the panel, hairline-framed, smaller, mono-voiced, terracotta-glyphed, labeled AGENTS · N. Execution looks like execution; it can never be misread as plan structure.

L3

Attention outranks structure

The parent's yellow pill counts needs-you across its whole branch, collapsed or not, and its sub-line whispers where (deep: POD-224). Depth may hide work; it never hides a request for you. Nothing yellow ⇒ nothing needs you.

L4

Two levels, then numbers

The sidebar renders parent + children. Anything deeper rolls into a quiet mono line — └ 3 deeper · 1/5 done — that opens the full tree in the issue peek. No third indent, no camera modes.

L5

Done leaves — on the shipped clock

Done ≠ archived. Done is a stage (the work closed); archived is removal from every list. A done child keeps its row only while its result awaits you — unread, up to 7 days — plus a short after-read grace; then it leaves the sidebar (derive.ts:1290, already shipped). No fold pile: history is the parent's k/m and the peek, not rows.

L6

Loose sessions are guests, not issues

A worktree row exists only for sessions no live issue owns (plain terminals, unbound shells). It renders in the roster grammar — rail-navy band at its project group's tail, labeled repo · branch in machine voice — never as a pseudo-issue row named “main” (derive.ts:1385, SidebarUnified.tsx:1339). Sessions orphaned by a deleted or archived issue carry a provenance whisper (from POD-32 · deleted) and decay on the session clock — they must never silently pool into an anonymous branch row (POD-133 found ~17 such orphans live).

Manual order (R1) applies inside the tree too: children drag within their parent; a dragged row cannot change parents in the sidebar (reparenting is an issue-page action). Pinning a child moves it to PINNED with full identity; its origin shows on hover.

03 · The design, live

One sidebar, four situations

    04 · Ordering model

    Manual order needs one new field; everything else already exists

    The server has no order field today — IssueWire carries priority, pinned, seq, createdAt, deferUntil and nothing positional (protocol/messages/issues.ts).

    RuleDefinition
    New field: sortKeyFractional/lexo-rank string on the issue: reorder writes one row, no renumbering. Server: issues module + issues.update patch. One key space per sibling scope — top level of a project group, a parent's children, and the PINNED section each sort independently.
    Default key = top of scopeA new issue mints a key above its scope's current minimum — “new appears at top” is the sort's natural behavior. New subtasks land at the top of their parent's children the same way.
    Band modelPINNED (own order, above all groups) → project groups (manual order) → snoozed sink to the group tail exactly as today (derive.ts:1569).
    Pinning = move, not copyA pinned row leaves its group (or its parent's children) for PINNED; unpinning returns it to its sortKey position.
    Drag affordance10px grip (⠿) on row hover; drag lifts the row with a tight transient shadow, siblings FLIP 180ms. Drops stay within the sibling scope; into/out of PINNED toggles pinned.
    What never sortsUrgency, activity, updatedAt — #64 stays law. Attention renders in the row (yellow pill, blue dot), never as motion of the row.
    05 · Arrival motion

    “Arrive” — a new one-shot in the existing grammar

    PhaseSpec
    1 · Make roomThe new row's wrapper expands height: 0 → auto over 260ms cubic-bezier(.22,1,.36,1); rows below slide down as a consequence — one animated element, no layout thrash.
    2 · MaterializeContent fades in and settles 4px, 300ms, delayed 60ms — the gap visibly opens first.
    3 · WashAn issue-tint wash (26%) decays over ~1s. Yellow is not used: arrival is information, not a request.
    Reduced motionInstant appearance; spinner frames freeze. Same gate as motion.css:184.
    06 · State grammar

    Selection is a surface; unread is a beacon

    Selectedsurface · tint 28% + ring + notch + strong ink
    Unreadbeacon · plain blue dot + medium weight
    Workingblue spinner + timer · the only motion
    Needs youyellow pill + frozen ago-stamp · zero motion

    Color budget, final: yellow = needs you (pill, ago-stamp, pin star) · blue = information (unread dot, working spinner/timer, done ✓) · red = alerts and destructive only · issue colors carry identity through the tint channel. The unread dot hides while its row is selected; selection's weight goes to 600 while unread stays 500 — weight alone never distinguishes them, the surface does.
