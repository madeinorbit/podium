All communication between agents, the superagent, the steward, and the chat UI is one durable message substrate — a single messages table with a per-message delivery ledger. No separate mail/nudge/notification mechanisms. Full design: docs/superpowers/specs/2026-07-13-agent-messaging-design.md (issue #237). Decisions below were made explicitly by Michael in the 2026-07-13 design review.

Delivery semantics

  • Two orthogonal axes: urgency (fyi / next-turn / interrupt) × lifecycle (wait / wake). Sender declares both; a relationship-based policy clamps (downgrade, never reject).
  • Senders never check target state; the server resolves session state at delivery time (TOCTOU-safe fire-and-forget).
  • Peers may wake. Hard interrupt (ESC) is superagent/parent-only; peers cap at next-turn.
  • Wake auto-spawns a fresh agent on the issue when nothing is resumable. This deliberately reverses the earlier "no auto-dispatch product-wide" stance; containment = wake cooldown per sender→issue pair, per-issue spawn budget, hop-depth clamp on message-triggered chains.

Attribution & authority

  • Sender identity is server-stamped; the envelope is server-rendered at delivery. Unwrapped turn = operator is an invariant: only the human's own words land without an envelope.
  • Superagent ≠ operator: distinct principal, parent-grade rights, always enveloped — a transcript always shows whether the human or the superagent said it.
  • Enveloped content carries the sender's authority, not zero authority: superagent/parent messages are instructions (within the receiver's issue scope); peer messages are requests/information that never redefine the receiver's task.
  • Human paths: direct terminal typing stays raw PTY (not a message); chat-UI sends ride the substrate as operator — unwrapped, unclamped, ledgered.

Acks & notifications

  • Receivers are asked to ack with context (what they did). The deterministic steward fallback on session settle fires only if no agent ack preceded it.
  • Steward/subscription notifications are ordinary messages of a distinct kind: coalesced, short-TTL, never expect acks, never wake (an explicit parent settle-subscription is the one exception).

Reading other sessions & cross-harness subagents

  • Read ladder, cheapest first: structured status → bounded cursor transcript read → server-side recap (watermark delta) → seance (ask: wake/resume the session, it answers from its own context, only the answer crosses over).
  • Cross-harness subagent children are full Podium sessions — visible, human-attachable, grouped under the parent and auto-tidied. No hidden broker daemons. All child driving uses the harness's interactive surface (PTY/hooks/native resume), never headless -p, preserving subscription auth.

Topology

  • No hub (deferred): the server owns the messages table and drives delivery; daemons are dumb executors told exactly what to do when the target session lives on them.
