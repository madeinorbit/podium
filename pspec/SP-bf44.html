Decided by Michael in the POD-833 cross-agent communication rebuild (design: docs/agent-comms-target.html §04b / §09-A; POD-279 field forensics). This supersedes the earlier "receivers are asked to ack with context; the steward fallback fires unless an agent ack preceded it" rule in the parent — that model produced 49% ack traffic, 243 acks-of-acks, and 36 false "finished without acking" notices.

Receipt is mechanical; a response is opt-in.

  • No receipt acks. Synchronous send (POD-834) already proves receipt — the ledger records delivery and the sender pulls status (podium mail status <id>). No ack message is ever generated or pushed for mere receipt.
  • A response is requested, not reflexive. Only mail send --expect-response (or a question, which always wants an answer) sets expects_response. An ack/notification can never set it — an ack is never itself ackable.
  • Responses are pull-delivered. A reply defaults to fyi urgency: it lands in the requester's mailbox and surfaces at its next natural stop / on pull — never a next-turn that starts a fresh turn.
  • The settle-nag fires only for an unfulfilled requested response. The stop-hook reminder and the steward settle-fallback read only expects_response = 1 rows; an ordinary message (even next-turn) owes no reply and never nags.
  • Semantic-reply-as-ack. Any reply back to the requester within the thread — not only a kind:'ack' — satisfies the request (stamps acked_by). A thorough substantive reply clears the nag. A reply still confirms the original delivered (preserves the POD-834 echo-miss fix).
