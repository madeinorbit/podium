A Codex pane has a stable Podium session ID from creation. The native Codex thread ID may be absent until Codex starts a thread, including after the first submitted message.

Every live-session surface — sidebar row, selection and navigation target, terminal attachment, and input routing — is keyed only by that stable Podium session ID. A native conversation ID, transcript path, issue association, process ID, or present or absent resume value is metadata; none may replace, merge, hide, or redirect a live Podium session.

When the native thread ID appears, Podium must attribute it to the exact pane that launched it, even when several new panes share a working directory or remain unsettled for different lengths of time. A sibling thread must never be inferred merely because it is the nearest or newest rollout. Simultaneously live rows remain independently visible and clickable if native identity data ever conflicts; conflicts fail visibly and must not redirect one Podium session ID to another.

Native Codex integration uses documented hooks and must not encode Podium identity in model-visible prompts or instructions or write Codex private hook-trust state. The stable Podium ID is inherited in the launched process environment; exact process ownership of an open native rollout may provide a fallback binding. A stable instance-scoped local socket handles callback-address changes. Hook identity payloads are retained locally until the server acknowledges them, so daemon or server downtime cannot discard a binding. Hook transport and receipts must use the instance runtime namespace defined by multi-instance runtime isolation and remain separate from the agent command relay.
