Podium sits next to private code, so the bar for what leaves a machine is higher than for typical SaaS. Precedent: the default and the vendor drive community anger, not payload richness (Go 1.23 opt-out revolt; Homebrew vs Google Analytics). Payload minimalism is for the minority who read the schema; the prompt and the moment decide the opt-in rate.

Decisions

  • Opt-in, default off. Nothing is sent until the user says yes.
  • Two tiers, independently consented: usage and crash. No perf tier — field histograms across heterogeneous self-hosted machines are noisy and hard to action, and local perf instrumentation already exists. Perf can later slot into usage as bucketed counters without a new tier.
  • No collection before consent. Deliberately not Go's local mode: Go's local is its default (data must exist for a later delayed prompt) and backs gotelemetry view. Podium asks at setup, when counters would be zero anyway — so local collection buys nothing and adds an objection surface. Off means off.
  • Consent is tri-state per tier: absent | on | off. Absent (never asked) behaves as off; the distinction exists so setup knows to ask.
  • Asked only during podium setup, as the last step, after the machine already works — the only optional question must not gate a working install, and Ctrl-C there leaves a working install with telemetry off. Re-running setup asks again. Never prompted on bare podium (Homebrew's repeating-notice complaints).
  • Only hosts emit and only hosts are asked. The server is the sole emitter; joined daemons/clients are covered by the hub's decision.
  • The example report is shown by default in the prompt — the audience is developers, and the JSON documents itself better than prose about it. The prompt names the opt-out location (Settings → Privacy).
  • First-party relay, disclosed vendor. Clients POST to a Podium-operated endpoint that drops the source IP and forwards to PostHog Cloud. Relay source lives in the public repo; deployment is in podium cloud. PostHog is named explicitly in the docs — undisclosed processors being discovered is the scandal pattern. The relay's real value is vendor abstraction: no third-party domain in firewall logs, and the vendor can be swapped server-side.
  • The relay URL is updatable via the signed update manifest. Whoever controls the update channel can already ship arbitrary code, so a URL field adds no trust surface — but it stays subordinate to the consent toggles, and an unsigned response can never change it.
  • Telemetry state lives in config.json, not the settings blob, so podium telemetry off works whether or not the server runs. Consent is read fresh at flush time, so it takes effect without a restart.

Deferred

Public aggregate dashboard — worth doing as trust proof and social proof, but gate it on install count (small numbers read as failure, and small slices de-anonymize; suppress buckets below ~20). Self-hosted PostHog — the relay makes it a server-side-only change if scale or sentiment demands it.
